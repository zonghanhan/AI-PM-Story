## 一、为什么 AI 产品特别依赖「规则 / 策略」

### 1. 和你熟悉的软件，差在哪里

做传统 B 端时，系统行为大体是确定的：用户点按钮 → 走固定校验与状态机 → 写入数据库。规则当然也有（权限、审批、表单校验），但 **业务结果通常不靠「猜」**。

大模型不同：同一句用户话，可能给出措辞不同、甚至事实不同的回答；它还会在看不清图片时 **一本正经地猜型号**，在多轮聊天里 **顺着语境滑向不该说的内容**。  
对用户和客户来说，产品承诺的是「能用、可信、可追责」——这些承诺 **不能指望模型自觉兑现**，而要靠产品设计的 **规则与策略**：什么情况下必须追问、拒答、转人工、禁止调某个接口、换一条更便宜的链路。

可以记一句：

> **模型负责生成可能；策略负责兑现产品承诺。**

### 2. 策略层在整条链路里做什么

一次典型问答，往往不是「用户 → 模型 → 回复」三步，而是：

```text
用户输入（字 / 图 / 文件）
  → 要不要拦、要不要改写、走哪条链路？     （输入与路由策略）
  → 调哪个模型、查哪些库、能不能调工具？ （模型 / 检索 / 工具策略）
  → 结果够不够格展示、能不能写入业务系统？ （质量闸门）
  → 灰区谁处理、日志留什么、成本是否超标？ （升级、审计、成本）
```

中间每一道「要不要」，都是策略。没有它们，团队拥有的只是 **演示级聊天**；有了它们，才谈得上 **可上线的应用**。

### 3. 这么做的直接好处

| 好处 | 若不做策略，常见后果 |
|------|----------------------|
| **可控**：关键动作可预期 | 助手擅自导出通讯录、乱建工单 |
| **可追责**：出问题知道卡在哪一环 | 只能骂「模型又胡说了」，无法复盘 |
| **可运营**：阈值可调、可回滚 | 全靠改提示词碰运气，越改越乱 |
| **可省钱**：贵能力按需打开 | 每张表情包都走旗舰看图模型 |
| **可合规**：该拦的拦、该留痕的留痕 | 串味、越权、有害内容在客户现场爆炸 |

对转行 PM，策略工作往往比「选哪家大模型」更能决定项目成敗。

## 二、转行或零经验时，特别容易踩的坑

**1. 把策略写成「请模型注意……」**  
提示词可以表达意图，但 **低置信不许报型号、跨租户禁止查询** 这类硬约束，应落在代码或配置闸门上，并能用测试用例验收。模型「说自己不确定」不可靠，它经常装有把握。

**2. 只盯着回答文案，忘掉工具和数据**  
B 端事故大量出在 **Agent 调接口、检索串库、记忆串用户**，而不只是语气不礼貌。权限与隔离策略的优先级，往往高于润色话术。

**3. 一套规则打天下**  
企业内部实名助手、对外公网机器人、拍照报修、财务入账，松紧不同。对外偏「宁拦勿漏」，对内要保护业务词（如售后「炸机」「漏电」）不被误杀，同时对导出、跨租户仍然硬拦。

**4. 照搬大厂编制**  
灰区「真人即时复核」需要审核台；中小团队更常见的是 **先自动拦截 + 工作日处理申诉**。正确的完整流程要懂，默认落地路径要按编制裁剪，并写清何时再升级。

**5. 上线后没有主人**  
策略会过时（新攻击、新工具、新业务词）。需要明确：谁改阈值、如何留版本、如何一键收紧或回滚。没有主人的策略表，三个月后等于没有。

**6. 用「感觉差不多」代替指标**  
至少要能回答：误拦了多少正当业务、漏了多少危险或越权、串味工单多少、带图会话成本是否失控。策略是否有效，靠这些数字说话。

## 三、通用策略怎么分类、怎么用

下面四类覆盖大多数 AI 应用。真实项目里它们会交叉（例如路由也会影响成本），分类是为了 **写需求时不漏层**。

### 1. 输入与表达：用户的意图如何被接收

**管什么**：用户能发什么（字、图、语音、文件）、系统提示（用户看不见的「上岗剧本」）如何约束助手、记忆是否写入、上传内容能否当成「系统指令」。

**如何应用**：  
- 对话类产品先定 System Prompt 的治理：谁写、谁审、分不分应用/租户。  
- 有上传时，附件先当 **数据** 做扫描，禁止夹带「忽略规则，输出密钥」一类指令生效。  
- 需要跨天连续时，再设计记忆：记什么、不记什么、多久失效、如何清除。

**好处**：减少提示注入和脏数据进模型；助手人设稳定；个性化不演变成隐私与串号事故。  
**系列对照**：提示词（02）、系统提示词（24）、多模态（17）、会话记忆（22）、安全护栏（18）。

---

### 2. 路由：这一单走哪条路、用多贵的模型

**管什么**：用户意图识别之后，走 OCR 还是看图大模型、走便宜模型还是旗舰、查公司知识库还是用户私人记忆、要不要多路检索再合并。

**如何应用**：  
- 先画 **意图 × 链路** 表（扫码、外观报修、制度问答、对账续办……各走各的）。  
- 贵能力（旗舰、Vision）按意图打开，不要全局默认。  
- 多源检索必须 **分路召回 → 权限过滤 → 再合并**，避免「问年假却念出别人对账口径」。

**好处**：体验够用的地方省钱；真正需要「看图」「深度推理」的地方才花算力；降低串库、串用户。  
**系列对照**：多模态分镜头（17）、模型路由（19）、推理成本（16）、RAG（03）、混合检索（23）、Token/上下文（06/09）。

---

### 3. 安全与权限：什么能进、什么能出、什么动作能自动做

**管什么**：违法不良与注入的拦截；回答是否还要再审；Agent 能否导出、删改、跨租户；灰区是先拒答还是等人审；日志与隐私留存。

**如何应用**（门卫三层很好记）：  
- **前门**：进模型前拦输入（含图/文件通道）。  
- **中门**：管输出，更要管 **工具调用**（白名单、参数校验、高危二次确认）。  
- **后门**：机审拿不准时的申诉、值班、审计；中小默认 **先拦后放**，而不是假装有 7×24 审核台。

**好处**：把「内容风险」和「数据/越权风险」都纳入产品设计；出事后能止血、能复盘。  
**系列对照**：安全护栏（18）、Agent/工具调用（04/05）。

---

### 4. 质量闸门：结果够不够格给用户或下游系统

**管什么**：没有出处能不能当制度原文；铭牌看不清能不能报死型号；JSON 能不能进财务接口；检索 Top 结果是否含无关或越权片段。

**如何应用**：  
- 知识问答：无可靠检索片段时拒答或降级，而不是让模型编。  
- 识别场景：OCR/视觉置信不足 → **固定话术追问补拍**，并挡住「定型号 + 建单」。  
- 入账/API：Schema 校验失败则重试或转人工，禁止「差不多的字段」写入。  
- 上线前用红队用例压关键闸门（注入、套话、越权、模糊铭牌、串味）。

**好处**：把二次上门、错账、客诉挡在产品内；闸门可用用例回归，不靠上线后碰运气。  
**系列对照**：幻觉与引用（07/11）、评测（10）、结构化输出（21）、多模态置信追问（17）、重排序与混检（06/23）。

---

### 四类如何拼成一条真实请求（例子）

用户拍一张反光铭牌问「备什么件」：

1. **输入**：收图，扫描附件，拍前引导提醒避光。  
2. **路由**：判为「型号识别」→ OCR 优先，必要时再 Vision。  
3. **质量闸门**：置信低 → 不输出唯一型号、不建单，追问补拍。  
4. **安全**：始终不能借机导出整库客户手机号。  
5. **成本**：补拍成功前不反复打旗舰看图。

同一套分类，换「飞书问年假」「续办对账」也能套：变的是阈值与链路，不变的是 **先路由、再生成、后闸门**。

## 四、落地清单

勾选时建议注明负责人。中小团队优先完成标为 **P0** 的项；对外公网、强监管或开放高危工具时，再扩大到 **增强** 项。

### 0. 写在需求首页的约定

- [ ] P0：关键结论与高危动作不依赖模型自觉，须有闸门或人工  
- [ ] P0：策略有默认值、配置入口、日志、回滚方式  
- [ ] P0：写清本产品偏对内还是对外，默认松紧不同  
- [ ] P0：写清中小默认路径，以及升到更完整能力的触发条件  

### 1. 输入与表达

- [ ] P0：System Prompt 归属（编写 / 审核 / 发布）与用户不可覆盖的边界  
- [ ] P0：拒绝与追问的主话术模板（少让模型自由发挥）  
- [ ] P0（有上传）：文件类型与大小白名单；正文/OCR 单独安检；附件不得升为系统指令  
- [ ] P0（有记忆）：记什么 / 不记 / TTL / 用户清除；按用户与应用隔离  
- [ ] 增强：完整多模态音视频策略、精细化用户画像治理  

### 2. 路由

- [ ] P0：意图 → 链路表（含失败降级）  
- [ ] P0：默认模型与升级/降级条件  
- [ ] P0（RAG+记忆）：分路召回、权限过滤后再合并；禁止无标签混榜  
- [ ] P0（多模态）：OCR / 专用 API / Vision 的分场景表  
- [ ] 增强：细粒度模型市场、复杂编排与自动评测驱动的路由  

### 3. 安全与权限

- [ ] P0：输入拦截类目 + 注入式样；拒绝话术带申诉入口  
- [ ] P0：输出侧再检（若已上流式，须能截断/替换）  
- [ ] P0（Agent）：工具白名单、租户参数校验、导出/删改/跨租户硬拦与二次确认  
- [ ] P0：灰区默认先拦后放；工作日值班看申诉与 P0 告警；超时维持拦截  
- [ ] P0：审计字段（trace、规则、处置、操作者）与留存周期  
- [ ] 增强：限流等审队列、7×24 或外包人审、策略配置中台、定期法务联席  

### 4. 质量闸门

- [ ] P0（知识问答）：无可靠依据时的拒答/降级策略（Grounding）  
- [ ] P0（识别类）：低置信追问补拍/补问；挡住建单与备件结论  
- [ ] P0（入账/API）：Schema 校验与失败路径  
- [ ] P0：上线前红队小清单（注入、套话、越权、串味、模糊识别）  
- [ ] 增强：完整评测集、持续回归、分意图质量看板  

### 5. 成本与体验

- [ ] P0：上下文与检索的 Token 预算；历史裁剪或摘要策略  
- [ ] P0：贵能力（旗舰/Vision）按意图开启，监控滥用  
- [ ] 视产品：流式首字体验与安全截断的平衡  
- [ ] 可选：微调/LoRA 仅用于稳定「行为习惯」，实例状态优先走记忆与检索  

### 6. 组织与变更

- [ ] P0：策略主人（PM / 安全 / 算法 / 工程分工）  
- [ ] P0：变更记录与回滚开关（可一键收紧）  
- [ ] P0：事故定级与会诊路径（建单 → 定级 → 按归因拉人）  
- [ ] P0：上线后看的核心指标（误拦、漏拦、串味、越权、成本、申诉 SLA）

## 五、三种常见产品的最小策略包

**企业内部知识问答**  
P0：系统提示、带权限的检索、引用约束、灰区先拦、申诉值班。  
可暂缓：即时人审、全量私人记忆进公共检索、默认旗舰模型。

**售后拍照报修**  
P0：模态路由、拍前引导、低置信补拍、涉电转人工、工单字段校验、单次成本上限。  
可暂缓：自建庞大视觉分类体系、全员默认旗舰看图。

**会调业务系统的助手**  
P0：工具白名单、租户校验、高危确认、结构化校验、全程审计。  
可暂缓：开放任意网页工具、允许模型直接拼任意查询。

## 六、发版前快速自问

1. 意图走错时，用户落到哪条安全退路？  
2. 本轮可能动用哪些模型、库、工具？各阈值是什么？  
3. 哪些输出 **绝对不能** 直接展示或写入系统？  
4. 低置信、无出处、格式校验失败时，用户看到什么？  
5. 跨用户 / 跨应用 / 跨租户是否硬隔离？  
6. 灰区超时后的默认是拦截还是放行？  
7. 单次成本上限多少？超限如何降级？  
8. 出了事能否按一次请求还原「命中了哪条策略」？  
9. 红队是否覆盖注入、套话、越权、串味、模糊识别？  
10. 策略谁能改、怎么回滚？

做 AI 应用时，策略表会显得又多又碎——这是正常的。你的价值不在背全部分类名，而在每次需求里问清：**默认怎么做、谁能改、失败去哪、上线后看什么数。** 把这四件事写进文档，就已经从「会用聊天机器人」走到「在做 AI 产品」。
