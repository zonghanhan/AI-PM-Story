## 寓言故事

### 一、一本账，把复盘会开成了批斗会

护栏灰度恢复后的第一个月，林薇以为最坏的时候过去了。

直到财务苏姐把一张表投到大屏上。表头很简单：**「小言 7 月推理账单」**——数字是 **三十八万**，比上月 **又涨了九万**。用户量只多了 **两成**。

老板没骂人，只问了一句：**「为什么涨得比用户快？」**

林薇准备好的答案——**多模态审核、输出复检、Agent 多跳**——刚念到一半，售后主管阿梅打断她：**「我这边八成会话就是问 **『发票怎么开』『保修期多久』**——三句话能答完的事，也要走全套旗舰？」**

工程大刘调出链路日志，投影上跳出一条典型会话：

```
用户：保修期多久？
  → 前门安全审核（小模型）
  → 意图识别（小模型）
  → 知识库检索 Top5
  → 主回答（旗舰模型）
  → 输出安全复检（中模型）
  → Grounding 引用校验（小模型）
```

「你看，」大刘指着中间那行，「 **主回答** 这一步， **一百条里有七十条** 是 FAQ 级问题，却 **清一色调用旗舰**——像 **感冒也挂专家号**。」

算法小陈苦笑：「不是不想省，是 **没人敢拍板**：路由错了，用户投诉；不路由，苏姐投诉。」

散会后，运维老周把林薇拉到走廊：**「我守机房二十年，见过最烧钱的不是空调，是 **所有人进门都往核心区跑**。你们缺的不是更贵的模型，是 **大堂里的分诊台**。」**

他指了指一楼闸机旁那个常年坐着喝茶的老师傅：**「老何。他只问三句话，就知道你该去几楼。」**

---

### 二、老何的三句话

第二天，林薇、小陈、大刘跟着老周，在园区大堂观察了半小时 **「老何分诊」**。

一个员工跑来：**「打印机卡纸了！」** 老何头都不抬：**「三楼 IT 值班台，走左边电梯。」**

又一个：**「我要申请跨部门数据权限，涉及客户手机号。」** 老何抬眼：**「二楼信息安全室，带工牌和主管邮件，没有邮件别排队。」**

第三个：**「我们产线 PLC 和 MES 对不上账，停产两小时了。」** 老何按下内线：**「我帮你叫老关和工艺值班， **直接上四楼应急通道**。」**

林薇忽然明白老周的意思。她在白板写下 **「老何三句话」**：

1. **这是什么类型的事？**（FAQ / 标准流程 / 复杂判断 / 应急）  
2. **需要多大权限和能力？**（查手册就够 / 要调系统 / 要多人会商）  
3. **答错了代价多大？**（用户重问一遍 / 工单延误 / 停产或合规事故）

小陈在白板另一侧画出 **「小言版老何」**：

```
用户消息进入
  → 路由层（Router）：分类 + 规则 + 轻量模型
  → 分支 A：FAQ / 查手册 → 小模型 + RAG Top3
  → 分支 B：流程办理 → 中模型 + 工具白名单
  → 分支 C：复杂推理 / 谈判 / 合规 → 旗舰模型
  → 分支 D：高风险 / 低置信 → 拒答或转人工
  → 各分支输出后，仍走统一「中门」护栏
```

「 **路由（Routing）** 不是让 AI 变笨，」小陈说，「是 **先分诊，再派单**—— **派给谁**，取决于 **问题类型、所需能力、容错空间**，不是 **默认全员顶配**。」

林薇问出 PM 该问的那句：**「分诊本身要钱吗？要多久？」**

大刘答：**「老何喝茶也拿工资。路由层通常用 **极小模型或规则引擎**，毫秒级；但 **分错了** 的代价，比路由费贵一百倍。」**

---

### 三、四层楼，四种价

接下来一周，他们在测试环境搭了 **「四层楼」** 试点——林薇故意用楼层命名，方便跟老板和财务解释。

**一楼：规则 + 关键词（几乎零成本）**  
「营业时间」「客服电话」「下载链接」—— **正则和配置表** 直接回， **不进大模型**。阿梅欢呼：**「这类占我们会话量十五%，以前也在烧旗舰。」**

**二楼：小模型 + 窄域 RAG（低成本）**  
保修期、发票规则、常见故障码—— **专用小模型**，只喂 **售后手册切片**。小陈强调：**「二楼不是『劣质版旗舰』，是 **『只学过一本手册的专科生』**。」

**三楼：中模型 + 工具（中成本）**  
查订单、建工单、改预约——需要 **Tool Calling**，但不需要 **长篇推理**。参数校验仍走护栏。

**四楼：旗舰（高成本，限量）**  
续费谈判、合同条款争议、多轮复杂排障、跨系统归因—— **才升舱**。升舱条件写进配置，不靠工程师拍脑袋。

试点跑了一千条金标会话。数据出来，林薇心跳快了一拍：

| 楼层 | 会话占比 | 单次均成本 | 事实准确率（金标） |
|------|----------|------------|-------------------|
| 一楼 | 14% | ≈ 0 | — |
| 二楼 | 58% | 0.06 元 | 96.2% |
| 三楼 | 21% | 0.18 元 | 94.8% |
| 四楼 | 7% | 0.52 元 | 98.1% |

**加权平均成本从 0.41 元降到 0.11 元**——接近 **降七成**；整体准确率 **只掉 0.4 个百分点**，因为 **该上四楼的并没被硬压到二楼**。

苏姐在群里回了一个字：**「行。」**

---

### 四、分错诊，比不分诊更贵

试点扩大灰度的第三天，阿梅又冲进林薇工位。

**「用户问『这台交换机反复重启，产线已经停了』——小言回了一段 **保修期三个月** 的标准话术！」**

林薇调日志：路由层把 **「重启」** 命中成 FAQ 里的 **「如何重启设备」**，分到 **二楼**；二楼小模型 **没识别出「停产」的紧急度**，套了最省事的模板。

这是一次典型的 **Under-routing（该升舱没升舱）**。

复盘会上，小陈列出 **另一类事故**：

**Over-routing（不该升舱却升舱）**：用户只说「你好」，情感分类抖动， **连续三轮被送上四楼**——账单好看不了，体验也怪。

**Router 延迟**：为了准，路由层叠了 **三个分类模型串行**，首字延迟 **多了 800 毫秒**——用户以为卡死。

**升级链路过长**：二楼答「我不确定」→ 升三楼 → 再升四楼， **同一问题烧三遍**；没有 **「带上下文升舱」**，四楼 **从零开始**，用户要 **重复描述三遍故障**。

林薇在白板写下 **「分诊四坑」**：

1. **该轻不重、该重不轻**——路由规则 **只按关键词，不看语境**；  
2. **升舱没有「病历本」**——楼层之间 **不传摘要**，用户体验和成本双输；  
3. **路由本身过度豪华**——分诊台雇了 **五个专家会诊**；  
4. **没有回头路**——四楼答完才发现是 FAQ， **不能降级复盘**。

她和团队定了 **产品级补丁**：

- **硬规则优先于软分类**：出现 **「停产、漏电、数据泄露」** 等词， **强制四楼或转人工**，不等模型犹豫；  
- **置信度低 → 默认升一档**，宁可多花一点，不要 **Under-routing**；  
- **升舱带摘要**：二楼升三楼时， **只传结构化摘要**（意图、已查字段、失败原因），不传全文复读；  
- **路由层预算上限**：单会话路由开销 **不得超过主回答成本的 15%**；  
- **每周看「分诊争议样本」**：客服标注 **「答非所问」** 的 case， **回流改路由表**——老何的册子也要周更。

---

### 五、老何不干活，但决定谁干活

月底，林薇给老板讲 **「分诊台」**，不讲参数规模，讲 **经营逻辑**。

她展示最终架构：

```
用户输入
  → 前门（安全）—— 全员必经
  → 老何（Router：规则 + 轻量分类 + 置信度）
       ├─ 一楼：配置直出
       ├─ 二楼：小模型 + 窄域 RAG
       ├─ 三楼：中模型 + 工具
       └─ 四楼：旗舰 / 人工
  → 中门（输出护栏）—— 全员必经
  → 用户
```

**关键产品指标** 她也换成 **分诊语言**：

- **路由准确率**（金标 + 客服回流标注）；  
- **Under-routing 率**（该升舱却未升舱，导致答非所问）；  
- **Over-routing 率**（不该升舱却升舱，浪费比例）；  
- **加权单次成本**（分场景，不看笼统平均）；  
- **升舱一次解决率**（升舱后是否还需再次升舱或转人工）。

老周坐在后排，慢悠悠补了一句：**「老何的价值，不是自己会修 PLC，是 **知道修 PLC 不能安排在查手册那队**。你们以后加新模型、新工具， **先问该挂几楼**，别 **往大堂堆人**。」**

林薇点头。她终于把 **Model Routing（模型路由）** 从「Inference Cost 里的一行 bullet」里拎出来，看成 **独立的产品层**：

**路由是 **调度系统**，不是 **降级偷懒**—— **轻活轻办，重活重办，拿不准就升档**，并且 **全程可观测、可纠偏**。

陈姐在会议纪要里加了一行：**「分诊规则纳入变更评审；涉及合规类，四楼白名单只增不减。」**

苏姐跟了一句：**「下月账单，我按 **楼层占比** 看。」**

## 概念正解

**Model Routing / 模型路由**，是在 AI 产品收到用户请求后、调用主生成模型之前，用 **规则、轻量分类器或小模型** 判断 **「这类问题该交给哪一档模型、哪条链路、是否转人工」** 的调度机制。它解决的核心问题是：**避免所有请求都走最贵、最慢的旗舰模型**，在 **质量、时延、成本** 之间做 **可配置的分流；同时通过升舱、降舱与上下文传递，处理「分错了怎么办」**。

回扣故事：**老何的「三句话」** 对应 **意图分类、能力需求与风险等级**；**四层楼** 对应 **规则直出 / 小模型 / 中模型+工具 / 旗舰或人工** 的分层；**「感冒挂专家号」** 是 **Over-routing 浪费**；**「停产却回保修期」** 是 **Under-routing 质量事故**；**升舱不带病历本** 对应 **路由链路上下文断裂**；**老何不干活但决定谁干活** 点明 **路由层是调度中枢，不是替代业务模型**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写清 **路由维度**（意图、风险词、用户等级、模态）、**各档模型与触发条件**、**升舱/转人工规则**、**楼层间上下文传递字段** |
| **方案评审** | 问算法/工程：**「路由层用什么？延迟多少？Under-routing 怎么兜底？金标集分场景准确率多少？」** 而不是只问 **「能不能换小模型省钱」** |
| **指标设计** | 上线 **路由准确率、Under/Over-routing 率、加权单次成本、升舱一次解决率**；按 **FAQ / 工具 / 复杂推理** 分桶看质量是否退化 |
| **跨团队沟通** | 对财务：**「七成会话在二楼，四楼只占个位数」**；对客服：**「答非所问可能是分错楼层，请标意图方便回流」**；对算法：**「合规词硬升舱，别全靠软分类」** |

## 类比锚定

**核心类比：医院门诊 **分诊台** + **专家号预约系统**——传统 PM 或互联网 PM 都熟 **「先分诊再派单」**：普通咨询走全科/自助机，复杂病情升专科，急诊走绿色通道；**不是让专家扫厕所，也不是让自助机做手术**。模型路由是同一逻辑在 AI 链路上的实现。

| 你熟悉的 PM 经验 | Model Routing 要素 |
|------------------|-------------------|
| 客服 **IVR 语音菜单**：按 1 查物流、按 2 改地址、按 0 人工 | **规则路由**：关键词/意图直达对应处理档 |
| **工单分级**：P0 停产、P3 咨询，不同 SLA 和值班人 | **风险等级路由**：高风险词 **硬升舱** 或转人工 |
| **专家号稀缺 + 预约条件**：普通病不占用专家时段 | **旗舰模型配额**：仅复杂场景升舱，控制 **四楼占比** |
| 分诊护士 **「拿不准就升一级」** | **低置信度默认升档**，避免 Under-routing |
| 转诊单带 **病历摘要**，避免患者重复陈述 | **升舱上下文传递**：结构化摘要进高阶模型，不全文复读 |
| 自助 FAQ / 智能客服 **先拦简单问题** | **一楼/二楼**：配置直出、小模型+RAG 处理高频低成本场景 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **路由是调度层，不是单个模型**——在 **安全前门之后、主生成之前** 决策 **走哪条链路**；可与 **多模型、工具、人工** 组合，而非简单「大变小」。
2. **目标三角：质量、成本、时延**——路由错了要么 **浪费（Over-routing）**，要么 **事故（Under-routing）**；要用 **分场景指标** 量化，不能只看总账单。
3. **硬规则 + 软分类要配合**——合规、停产、隐私等 **硬升舱**；FAQ 等 **可走轻档**； **低置信升一档** 是常见产品策略。
4. **升舱必须带上下文**——否则 **多档串联** 比 **单档旗舰** 更贵更慢； **结构化摘要** 是 PRD 必写项。
5. **路由策略要可迭代**——用 **金标集 + 客服回流争议样本** 周更「分诊册子」；上线路由不是一劳永逸。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像 IVR，按 1 按 2 就行」 | 自然语言 **意图模糊、多轮渐变**；用户不会乖乖按菜单说 | 需要 **轻量分类模型 + 多轮语境**；IVR 只覆盖 **规则路由** 子集 |
| 「简单问题给小模型，一眼能分」 | **同一句话在不同上下文含义不同**（「重启」可能是 FAQ，也可能是停产） | **会话级特征 + 风险词硬规则**；不能 **只分类最后一句** |
| 「专家号不够就排队」 | 模型 **扩容比医院挂号快**，但 **成本线性涨**；没有绝对「号源」只有 **预算** | 产品用 **四楼占比上限、配额、峰时限流** 经营，不是无限升舱 |
| 「分诊错了再转诊」 | AI 里 **多次全链路重跑** 成本极高；用户 **无感多次升舱** 仍体验差 | 设计 **一次升舱到位 + 带摘要**；设 **路由开销上限** 与 **升舱次数熔断** |
| 「路由只影响用哪个医生」 | 不同档模型 **工具权限、RAG 库、护栏策略** 可能不同 | 路由是 **整条链路切换**（模型+检索+工具+限额），不是只换模型名 |

### 动手小练习（约 15 分钟）

1. 为你负责的 AI 功能列 **10 条真实用户问法**，标注应去 **「一楼～四楼 / 人工」** 哪一档，并写 **一条硬规则**（若与软分类冲突谁优先）。
2. 画 **「升舱带摘要」** 字段表：从低档升高档时，至少传递 **意图、已查数据、失败原因** 三项。
3. 定义 **1 个 Under-routing 事故样本**（如故事里的「停产却回保修期」），写 **产品兜底**：硬词表、低置信升档、还是直接转人工。

### Model Routing 产品 Checklist（可保存到笔记）

```markdown
## 需求阶段
- [ ] 路由维度：意图 / 风险 / 用户层级 / 模态
- [ ] 各档链路：模型 + RAG 范围 + 工具权限 + 单次成本上限
- [ ] 硬升舱词表与合规类（只增不减）
- [ ] 升舱上下文传递字段（结构化摘要）
- [ ] 低置信默认策略（升一档 / 转人工）

## 方案评审
- [ ] 路由层延迟与成本 ≤ 主回答 15%（或团队自定阈值）
- [ ] 金标集分场景准确率（二楼 FAQ / 三楼工具 / 四楼复杂）
- [ ] Under-routing / Over-routing 定义与采样方案
- [ ] 升舱次数熔断（防同一问题烧三遍）

## 上线与运营
- [ ] 加权单次成本 + 各楼层占比看板
- [ ] 客服「答非所问」回流改路由表（周更）
- [ ] 四楼占比异常告警（突增可能是路由坏了或攻击）
- [ ] 与 Inference Cost、Safety 策略联调记录
```
