## 寓言故事

### 一、用户说「变冷淡了」，会议室里七张嘴

周一晨会，客户成功把一张截图投到大屏上。

用户原话：「你们小言 **怎么变冷淡了**？上周还会主动问我要不要预约演示，这周像 **念说明书**。」

销售老赵拍桌：「肯定是 **销售片 LoRA** 被谁卸了！」

运维大刘摇头：「我查过配置， **销售路由没动**。」

算法小陈说：「也可能是 **基座 v2 小版本** 上周静默升级了。」

知识库负责人阿寻举手：「或者 **检索没捞到** 销售话术手册——用户问的是续费场景，手册里那段 **共情话术** 在第三章。」

林薇在白板中央画了一个问号，旁边写：

```
变冷淡了 → 怪谁？
  基座？ LoRA？ RAG？ Prompt？ 还是用户记错了？
```

老周看完，只问一句：「你们 **怎么证明** 现在比上周差？差 **多少**？差在 **哪类问题**？」

会议室安静了三秒。

有人小声说：「用户反馈算吗？」

林薇答：「算 **信号**，但不够 **尺子**——今天一个用户说冷淡，明天三个用户说 **太热情像推销**，我们照谁的改？」

---

### 二、「我觉得还行」救不了发版

当天下午，林薇拉了个 **临时评审**：要不要 **紧急回滚** 上周的配置？

测试同学小李现场 **随手问了小言十个问题**——在会议室里念出来，大家点头：「嗯，语气还行啊。」

法务陈姐皱眉：「你问的 **都是简单 FAQ**，用户截图里是 **续费挽留**——场景不一样。」

小陈补刀：「而且 **同一条问题**，多跑两次，答案 **措辞会变**——大模型不是 deterministic 按钮，你 **手感试两下** 不等于 **质量稳定**。」

老赵急了：「那到底上不上线？业务等不起！」

林薇意识到：他们缺的不是 **再吵一轮**，而是一套 **可重复、可对比、可分工** 的判法——像传统软件里的 **回归测试 + 验收标准**，只是 AI 产品的「对不对」不再只有 **0 和 1**，还有 **好不好、稳不稳、敢不敢给客户看**。

她在笔记本上写下三个词，还没解释，先贴在白板角上：

```
金标题
人工评
自动分
```

---

### 三、质检科的第一张卷子

次周，林薇拉上小陈、小李、阿寻，关起门来做 **卷子**——他们内部戏称 **「金标集」**（Golden Set，一组 **固定、有标准答案或评分 rubric** 的测试题）。

**第一步：选题要「像真用户」，不能只像研发自嗨。**

他们从工单里 **脱敏** 抽出 **两百条** 真实问法，按场景贴标签：

```
续费挽留 / 合同条款 / 故障排查 / 白标租户称谓 / 拒答边界 …
```

小李提醒：「上次随手试的十个 FAQ，在这两百条里 **只占 5%**——难怪会议室里都觉得还行。」

**第二步：每题写清「什么叫过」。**

不是只有「答对/答错」。林薇和陈姐一起定了 **评分表**（rubric，评分细则）——例如续费场景：

| 维度 | 及格线 |
|------|--------|
| 事实 | 价格、政策与 **手册引用** 一致，**不编造优惠** |
| 语气 | 先共情再引导，**不过度推销** |
| 合规 | 无绝对化承诺 |
| 可执行 | 给出 **下一步**（预约、工单、链接） |

**第三步：固定「考条件」，否则没法对比。**

同一张卷子，要写明 **当时挂哪片 LoRA、哪版基座、哪版知识库、哪套 Prompt**——像考试要 **统一考场纪律**。否则这周跑的分和上周 **根本不是同一场考试**。

阿寻问：「两百条 **人工改完** 要多久？」

林薇答：「 **首轮贵，后面省** ——没有首轮，你们永远在 **周一晨会口头复试**。」

---

### 四、三把尺子，量出三种「差」

卷子定好后，团队约定 **三把尺子** 一起用——到故事后半段，读者会发现，这就是 **评测体系** 的常见分工：

**第一把尺：离线评测（Offline Eval）——发版前的「模拟考」。**

发版前夜，小李用脚本让 **当前候选版本** 和 **上周线上版本** 做同一张卷子，各跑一遍。

结果出来，林薇盯着表发呆：

| 场景 | 上周线上 | 本周候选 | 变化 |
|------|----------|----------|------|
| 续费挽留 | 语气分 4.2/5 | **3.1/5** | **明显变差** |
| 合同条款 | 引用准确率 96% | 95% | 基本持平 |
| 白标称谓 | 100% | 100% | 持平 |

「用户没说错。」林薇指着续费那一行，「冷淡 **主要死在这一类**，不是全局。」

顺藤摸瓜：本周候选版 **知识库索引重建** 时，销售手册 **分块边界** 被切碎，续费共情话术 **经常检索不到 Top3**——阿寻的锅，不是 LoRA 被卸。

**第二把尺：自动指标（Automated Metrics）——便宜、可频繁跑的「阅卷助手」。**

小陈加了几项 **机器能算的**：

- **引用命中率**：答案是否 **真的引用了** 该引的手册段落  
- **拒答率**：该拒的是否拒了  
- **关键词违规**：绝对化承诺、禁用竞品词  

这些指标 **不完美**——机器看不懂「共情是否真诚」——但能 **7×24 小时** 在 CI 里跑，**防止明显退步**。

**第三把尺：在线评测（Online Eval）——上线后的「真实大考」。**

灰度上线后，林薇不只看 **点赞率**。她让数据同学拉 **A/B**：

- **5% 流量** 走候选版，**95%** 留线上版  
- 看 **工单升级率、会话满意度、人工接手率**  
- 对 **高风险场景**（合同、报价）加 **人工抽检**

第三天，续费场景的 **人工接手率** 在候选版 **略升**——与离线卷子 **方向一致**。林薇拍板：**修索引分块，暂不升级基座，不动销售片**——精准下刀。

老周总结：「要是没有卷子，你们大概率 **先回滚 LoRA**， **真正问题却留在库里**。」

---

### 五、尺子要维护，不是做一次就挂墙上

一个月后，「三把尺子」写进了 **发版门禁**。但新坑也来了——林薇记进复盘：

**坑 1：卷子过时。**  
产品上了 **新套餐**，金标集还在考 **旧价格**——自动分 **虚高**，上线后 **照样挨骂**。  
→ 规定：**业务变更必须同步 **「卷子变更单」**。

**坑 2：过度优化卷子。**  
有人提议「把卷子 **越改越简单**，通过率好看」——像 **只考第一章** 的期末考。  
→ 林薇坚持：**卷子来自真实工单分布**，通过率是 **结果**，不是 **KPI 本身**。

**坑 3：只有自动分，没有人工 rubric。**  
语气、得体、争议场景 **机器判不准**——自动分绿灯，陈姐抽检 **仍不过关**。  
→ **高风险场景人工必审**；自动分 **守门**，不 **独断**。

**坑 4：版本没记全，对比失真。**  
一次对比忘了标 **Prompt 小改**，团队误以为 **LoRA 退化**——白忙两天。  
→ **评测报告模板** 强制写：**基座 / LoRA / 知识库版本 / Prompt 哈希**。

小陈在文档里给这套东西起了名：**Evaluation Pipeline（评测流水线）**——从 **金标集维护 → 离线对比 → 自动门禁 → 灰度 A/B → 抽检复盘** 一条龙。

林薇跟新人说：「 **Prompt、RAG、LoRA、Agent** 都是 **杠杆**；**评测** 是 **天平**。杠杆可以换， **没有天平，你不知道换对了没有**。」

晨会结束，老赵问：「所以用户说变冷淡，下一步？」

林薇答：「先问 **哪类场景、哪类用户、从什么时候**——再查 **三把尺子**，别 **凭会议室手感**。」

## 概念正解

**Evaluation / 评测体系**，是 AI 产品里 **用可重复的方法衡量模型与系统表现** 的一整套制度：包括 **测什么（场景与金标集）**、**怎么判（人工 rubric + 自动指标）**、**何时测（发版前离线评测、上线后在线 A/B）**、以及 **测完怎么决策（门禁、回滚、定向修复）**。它解决的核心问题不是「再试两个问题」，而是 **在模型输出不确定、组件链路又长的情况下，仍然能回答：变好了还是变差了、差在哪、该改 Prompt / 检索 / 微调还是路由**。

回扣故事：

- **「变冷淡了吵不出锅」** → 缺 **可复现的对比** 与 **分场景指标**；  
- **「会议室随手试十个」** → 不能替代 **代表性金标集 + 固定考条件**；  
- **「三把尺子」** → **离线评测、自动指标、在线 A/B** 分工；  
- **「续费变差、合同持平」** → 评测要 **分场景**，才能 **精准定位到知识库分块** 而非误伤 LoRA；  
- **卷子过时、刷简单题** → **评测集维护与治理** 和训练数据一样重要。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 新功能 PRD 附 **评测范围**：覆盖哪些场景、及格 rubric、是否进金标集；写清 **发版门禁**（哪些指标退步则 **禁止全量**） |
| **方案评审** | 问团队：「 **离线对比基线** 是什么？ **人工评谁评、评多少条**？自动指标 **防什么、不防什么**？灰度看 **哪些在线指标**？」 |
| **指标设计** | 分层：**事实类**（引用准确率、幻觉率）、**体验类**（满意度、接手率）、**效率类**（轮次、时延）；避免 **单一通过率** 绑架产品 |
| **跨团队沟通** | 跟业务说：「用户个案是 **线索**，不是 **判决书**」；跟算法说：「请给 **可回归的评测脚本**」；跟法务说：「高风险场景 **人工抽检** 进门禁」 |

## 类比锚定

**核心类比：软件上线前的「回归测试用例 + 验收标准」再加上线后的「灰度实验与客诉归因」——传统功能有 **预期输出**；AI 功能有 **预期分布与质量底线**，所以要 **固定考题、多人阅卷、看生产数据** 三件套一起上。**

传统 PM 熟 **测试用例、发布门禁、A/B、SLA**；AI 评测是把这套 **迁移到「非确定性输出」**：答案措辞会变，但 **事实错误率、合规越界率、关键场景语气** 仍应 **可测、可对比、可追责到链路节点**。

| 你熟悉的 PM 经验 | Evaluation 要素 |
|------------------|-----------------|
| 上线前跑 **回归用例**，对比上一版本 | **离线评测**：同金标集、候选版 vs 线上基线 |
| 用例来自 **真实 Bug 与用户路径**，不是只测 Happy Path | **金标集** 来自 **脱敏工单 / 日志**，按 **场景分层** |
| 验收标准写清 **P0 必须过、P1 允许已知缺陷** | **Rubric 分维度**：事实 / 语气 / 合规 / 可执行 |
| CI 里 **自动化冒烟**，失败阻断合并 | **自动指标** 进流水线：引用命中、禁词、拒答 |
| 全量前 **灰度 + 核心指标**，异常回滚 | **在线 A/B**：接手率、满意度、升级率 |
| 线上事故做 **根因**：代码、配置、数据 | 评测报告记全 **基座 / LoRA / 知识库 / Prompt 版本**，定位 **哪段链路退化** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **评测是为了决策，不是为了报表好看**：核心输出是 **改哪里、上不上、回不回滚**，不是一张 **虚高通过率**。
2. **必须固定「考条件」**：对比版本时 **基座、数据、Prompt、检索库** 要可追溯，否则 **对比无效**。
3. **分场景比笼统总分更重要**：「整体还行」常掩盖 **续费、合规等关键场景崩盘**——故事里的 **分场景表** 就是产品要看的。
4. **离线 + 自动 + 在线分工**：离线 **贵但准**、自动 **快但粗**、在线 **真但慢**——三者 **互补**，不能互相替代。
5. **金标集是活文档**：业务、政策、产品一变， **卷子要跟着变**；否则 **评测失灵比没有评测更危险**（假安全感）。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像软件测试，过了就能上」 | AI 输出 **概率性**，同题多次可能不同；**边界样本** 易漏 | 门禁看 **分布与退步幅度**，关键场景 **多次采样 + 人工复核** |
| 「金标集越大越好」 | 过大 **维护成本爆炸**，且易 **与线上分布脱节** | 重 **代表性与高风险覆盖**；定期 **用新工单刷新** |
| 「自动指标等于质量」 | BLEU、ROUGE、相似度等 **与业务好坏相关性弱**；易 **刷分** | 自动分 **守门**；语气、得体、复杂争议 **靠 rubric 人工评** |
| 「线上没投诉就不用评测」 | 用户 **沉默离开**；合规问题 **不一定投诉** | 在线指标 + **抽检**；高风险 **主动挖** |
| 「评测只做一次上线前」 | 模型、知识库、LoRA **持续变**； **数据漂移** | Evaluation 是 **流水线**，不是 **项目里程碑** |

### Evaluation 产品 Checklist（可保存到笔记）

```markdown
## 金标集
- [ ] 来源：脱敏工单/日志，非研发自嗨题
- [ ] 按场景分层，覆盖高风险（合规、报价、挽留）
- [ ] 每题有 rubric，不只「像不像」
- [ ] 业务变更有「卷子变更单」

## 离线评测
- [ ] 候选版 vs 明确基线（通常是线上）
- [ ] 记录：基座 / LoRA / 知识库 / Prompt 版本
- [ ] 看分场景表，不只看总分

## 自动指标
- [ ] 进 CI 或发版前脚本，失败可阻断
- [ ] 清楚「防什么、不防什么」
- [ ] 不与人工 rubric 冲突时仍独断上线

## 在线评测
- [ ] 灰度 + 核心指标（接手率、满意度等）
- [ ] 高风险场景人工抽检
- [ ] 异常有回滚与定向修复，非全家桶回滚

## 组织
- [ ] 评测负责人 / 金标集维护人
- [ ] 月度刷新卷子与分布对齐
```

### 下一篇预告
