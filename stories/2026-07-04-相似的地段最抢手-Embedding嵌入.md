## 寓言故事

### 一、库里有的，为什么捞不着？

幻觉整改上线后的第二周，林薇以为能喘口气。

结果数据同学甩来一张表：RAG 命中率从 68% 升到 71%，**用户仍说「答非所问」的占比却从 12% 涨到 19%**。

她随手点开一条差评。用户问：

> 「耳机 **拆封了** 还能 **退钱** 吗？」

小言引用的知识库片段标题是：《**耳机防水等级** 与 **保养说明**》——全文在讲 IPX 评级，跟退款毫无关系。

林薇调出检索日志，眉头越皱越紧：

- 用户 query 原样送进检索；
- 系统返回 Top3，第一篇因为正文里出现 **「耳机」** 二字，得分最高；
- 真正写「拆封不退款、仅换货」的《耳机品类售后指引》，排在 **第七位**，根本没进上下文。

「我们不是上了 RAG 吗？」林薇找老周，「阿寻（知识库管理员那套流程）也在跑，怎么又像 **关键词搜索**？」

老周让她看检索配置：当前用的是 **BM25**——传统文本检索，**字面重合越多，分越高**。

「用户问『退钱』，库里写的是『退款』；问『拆封』，文档里写的是『已开封』。」老周在板上写，「**字不一样，分就低**。但人脑知道，这是在问同一件事。」

---

### 二、同义词表救不了场

林薇的第一反应很 PM：建 **同义词表**。

她拉客服骨干，一下午整理出 47 组：

| 用户常说 | 文档常用 |
|----------|----------|
| 退钱 | 退款 |
| 拆封 | 已开封、拆开包装 |
| 坏了 | 故障、无法开机 |
| … | … |

上线三天，拆封退款类问题好转一些。但新的差评来了：

用户问：「**降噪效果不好** 算质量问题吗，能换吗？」

检索 Top1：《**降噪技术白皮书**》——大段讲 ANC 原理；Top2：《**音质主观评价标准**》——讲听感测试方法。

真正该命中的《**耳机性能故障判定与换货标准**》——里面有一句「**降噪失效、单侧无声等属性能故障**」——又沉到后面去了。

因为用户没说「故障」二字，同义词表里 **没有**「效果不好 → 性能故障」这一条。

老周：「你这是在 **人工维护一张无限大的映射表**。用户说法千变万化，**穷举不完**。」

林薇有点烦：「那 RAG 到底差在哪？幻觉我们认了，是模型爱补全；**检索错了，总不能也怪模型吧？**」

老周：「检索错了，模型再老实也只能 **对着错材料说胡话**——或者 **看着防水等级编退款条款**。问题在 **找段** 那一步。」

他顿了顿：「你得见一个人。库房那边的 **阿寻**，上周刚帮另一个业务线做了一套 **『按意思找』** 的索引。」

---

### 三、阿寻的「地段图」

周三下午，林薇在老周工位旁见到阿寻——不是真人，是同事们给 **向量检索模块** 起的外号，纪念半年前《库房管理员阿寻》那套 RAG 改造。

阿寻的维护同学姓陈，打开可视化后台，屏幕上不是文件夹树，而是一片 **密密麻麻的点**，像城市夜景里的灯。

「每一个 **知识库切片**（chunk），」小陈指着其中一个光点，「入库时先过一道 **Embedding 模型**——你可以理解成：把这段话的意思，**翻译成地图上的一个坐标**。」

林薇凑近看。几个点挨得很近，标注分别是：

- 「已开封商品不支持退货」
- 「拆封后仅提供换货服务」
- 「用户自行撕开塑封膜视为已使用」

另一团点离得很远，是「蓝牙配对步骤」「固件升级方法」。

「用户问 **拆封能退吗**，」小陈在搜索框输入这句话，地图上 **亮起一个红星**——那是 **query 自己的坐标**。系统瞬间圈出 **离红星最近的几个点**，直线距离最短的就是 TopK。

林薇眨眨眼：「所以……不是数 **有几个字一样**，是数 **意思有多近**？」

「可以这么理解。」小陈放大一角，「『退钱』和『退款』两个字不同，但 Embedding 把它们落在 **相邻地段**；『拆封』和『已开封』也是。甚至『降噪效果不好』和『降噪失效』——字面对不上，但 **语义上挨着**。」

林薇想起上周那单差评。她让小陈用旧 BM25 和新向量检索各跑一遍同一条 query。

| 检索方式 | Top1 结果 | 是否相关 |
|----------|-----------|----------|
| BM25 关键词 | 《耳机防水等级与保养说明》 | 否 |
| Embedding 近邻 | 《耳机性能故障判定与换货标准》 | 部分相关 |
| Embedding 近邻 | 《耳机品类售后指引·拆封换货条款》 | 是 |

「**相似的地段，最抢手**。」小陈笑，「谁离用户问题最近，谁先进上下文。」

---

### 四、地段图不是万能地图

林薇以为可以开香槟了。小陈却主动泼冷水：「别神话。Embedding 解决的是 **『像不像』**，不解决 **『对不对』**。」

他讲了三个他们踩过的坑：

**坑 1：旧文档和新文档挤在同一个地段**

《售后总则 2023》写「7 日可退」，《2024 修订版》写「数码类 7 日、拆封不退」——两段话 **意思相近**，坐标几乎重叠。用户问退款，**两篇都可能被捞出来**，模型若没被告知 **以新为准**，仍可能引用旧版。

**坑 2：专业黑话把坐标带偏**

内部把「**RMA**」当「返厂换货」的缩写。Embedding 若没见过足够多 **RMA + 换货** 的共现，用户问「返厂维修怎么申请」，可能捞到《**RMA 财务报销流程**》——因为都有 RMA 三个字母，**字面近，业务远**。

**坑 3：多语言与口语**

港澳台用户问「**退貨**」，文档用简体「退货」——多数模型能拉近，但 **方言、错别字、中英混杂**（「refund 怎么搞」）会让坐标 **偏到奇怪邻居**，检索日志里会出现 **完全无关但向量距离尚可** 的片段。

林薇问：「那和我们之前 **幻觉** 什么关系？」

小陈：「检索捞错段，模型 **看着错误上下文自信作答**——幻觉整改里的 **引用溯源** 还在，但 **引用的就是错页**。产品上要 **同时盯 Embedding 质量和幻觉护栏**。」

老周补了一句：「而且 Embedding **有成本**：每篇文档入库要算一次向量，query 每次也要算，还有 **向量库存储**。chunk 切太大、切太小，坐标都会变——这是 **产品参数**，不是纯研发自嗨。」

---

### 五、先画地图，再谈聪明

又过了一周，林薇主导了一版 **「语义检索 + 关键词兜底」** 的混合方案：

1. **主检索用 Embedding**，解决「用户说法和文档写法不一致」；
2. **关键词规则兜底**，对订单号、SKU、条款编号等 **精确字段** 仍走 BM25；
3. **切片策略写进 PRD**：售后政策按「**一条规则一块**」切，不把 47 页合同整段 embedding——避免 **一个坐标代表太多意思**；
4. **元数据过滤先于近邻**：先按「品类=耳机」「文档状态=生效」「版本≥2024」筛一圈，再在子集里找最近邻——小陈叫 **「先圈商圈，再看距离」**；
5. **监控换指标**：不只盯「检索命中率」，加 **MRR@3、首条相关率、检索-人工标注一致率**；每周抽检 50 条，看 **近邻是否真相关**。

灰度上线后，「答非所问」占比从 19% 降到 9%。林薇在复盘会上写了一句她后来常挂在嘴边的：

> **RAG 不是「把库接上」就完事；Embedding 是把每段知识 **钉在意义地图上的坐标**。地图画得准不准，决定模型 **第一眼看见的是答案还是噪声**。**

散会后，她独自看阿寻后台那张点图。用户问「拆封能退吗」，红星落在「已开封」「换货」「不支持退货」那一簇旁边——**没有一个字完全匹配，但地段对了**。

她忽然想起半年前老周说的「闭卷考试」：Embedding 不是让模型更聪明，而是 **在开卷之前，帮它把正确的页码递到桌上**。

## 概念正解

**Embedding（嵌入 / 向量化）** 是把文本（词、句、段落）通过模型 **映射成一组数值坐标（向量）** 的技术；在向量空间里，**语义相近** 的文本坐标 **彼此靠近**，语义远的则 **相距较远**。RAG 检索时，把用户问题也变成坐标，用 **「找最近的邻居」** 代替单纯的关键词匹配，从而召回 **意思相关** 而不仅是 **字面重合** 的知识库片段。

回扣故事：

- **阿寻的「地段图」** → 知识库切片入库时生成 **文档向量**，构成可检索的向量空间；
- **红星与最近邻** → 用户 query 被 **query embedding** 后，做 **相似度检索（TopK）**；
- **「退钱」与「退款」坐一桌** → 语义相近的不同表述，向量距离近；
- **同义词表救不了场** → 纯规则无法覆盖开放说法，Embedding 用 **统计语义** 泛化；
- **旧版新规挤在同一地段** → 「相似 ≠ 正确」，需 **元数据、版本、重排序** 等产品手段纠偏。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写明 **切片（chunk）规则、embedding 模型版本、是否混合关键词检索、元数据过滤字段**；定义「相关」的 **业务标准**（如首条必须含可执行政策） |
| **方案评审** | 问研发：「向量用什么模型？**中英/黑话** 测过吗？库更新后 **重算向量** 的流程谁触发？TopK 太大对 **Token 成本** 的影响？」 |
| **指标设计** | 除回答满意度外，加 **首条检索相关率、MRR@K、检索空结果率**；建立 **人工标注集** 做回归，避免「向量距离近但业务无关」 |
| **跨团队沟通** | 跟业务说：「**搜得到靠 Embedding，搜得准还要版本和权限**」；跟算法说：「换 embedding 模型 = **整张地图重画**，要排 **重索引** 窗口」 |

## 类比锚定

**核心类比：房产中介带看——客户说「想住得安静、通勤半小时」，中介在地图上圈板块，不是按楼盘名字逐字对**

传统 PM 都做过 **用户画像与推荐**：用户不会用你们后台标签原话描述需求，但好销售能听出「安静」≈「远离主干道」「学区不要太吵」≈「低容积率」。Embedding 就是给每段文本贴 **「意义地址」**，让系统像资深中介一样 **按需求地段找房源**，而不是只会 **按楼盘名字搜字典**。

| 你熟悉的 PM 经验 | Embedding / 检索要素 |
|------------------|----------------------|
| 用户访谈原话五花八门，你整理成需求标签 | 原始文本 → **向量表示**；口语与文档写法不同，但 **标签/坐标可对齐** |
| 推荐「相似用户还买了」 | **近邻检索**：与 query 向量距离最近的 chunk |
| 房源先按「区域、预算、户型」筛，再在子集里排序 | **元数据过滤 + 向量检索**；先圈商圈，再看语义距离 |
| 同一条地铁线，2020 年版地图和 2024 年扩线后坐标系变了 | **换 embedding 模型需全量重索引**；旧坐标与新坐标 **不可混比** |
| 楼盘名含「花园」但与「花园小区」不是一回事 | **字面相似 ≠ 语义相近**；也可能 **语义相近但政策版本错误** |
| 带看清单只给 3 套，太多客户会晕 | **TopK 与 Token 预算**；召回过多 **挤爆上下文**，反而干扰模型 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Embedding 把文本变成可计算的「意义坐标」**，使 **语义相似性** 可用距离度量，是 RAG **语义检索** 的常见基础。
2. **它解决的是召回「意思相关」**，不保证 **事实正确、版本最新、权限合法**；「离得近」只是 **候选**，不是 **答案**。
3. **切片、模型版本、语言与领域** 都会改变坐标；**chunk 太大** 地图点含糊，**太小** 丢上下文——都是 **产品可配置项**。
4. **有算力与存储成本**：入库 embedding、查询 embedding、向量库运维；规模上来要谈 **重索引策略与预算**。
5. **常与关键词检索、重排序（Rerank）、元数据过滤组合使用**；生产环境很少「只押 Embedding 一招」。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「地段近就是好房」 | 向量近邻可能是 **过期政策、错误部门文档、竞品提及** | 必须加 **版本、生效时间、权限、品类** 等硬过滤；近邻只是初筛 |
| 「中介听得懂人话」 | Embedding 对 **极少出现的专有缩写、新造活动名** 可能 **坐标漂移** | 黑话表、别名库、**混合检索** 仍是刚需；别假设「语义万能」 |
| 「地图画一次就行」 | 文档增删改、**换 embedding 模型** 都要 **重新计算向量** | 产品要定义 **更新触发器**（发布即索引、夜间全量、版本回滚怎么办） |
| 「相似推荐越高越好」 | TopK 过大 **塞满上下文**，引入噪声，**Token 成本飙升**，模型反而 **选错重点** | TopK、Rerank、**摘要后再进 Prompt** 要在 PRD 里写清权衡 |
| 「安静 = 远离主干道」一一对应 | 高维空间里 **多义、反讽、双重条件**（「拆封不能退但可以换」）可能落在 **多个簇之间** | 需要 **多向量、重排序模型、或结构化字段**；Embedding 不是语义理解的全部 |

### 动手小练习（约 15 分钟）

1. 打开任意带「语义搜索」的演示（如 OpenAI Embedding Playground、国内大模型知识库试玩），**同一意图写 5 种不同问法**（如「退钱」「退款」「能不能把钱要回来」），看 Top 结果是否 **稳定指向同一段政策**。
2. 列出你们业务 **10 个内部黑话/缩写**，测 embedding 检索是否 **常错邻**；标出需要 **关键词兜底** 的词——可直接写进 PRD「混合检索规则」。
3. 画一张 **RAG 数据流简图**：从「文档发布」到「切片 → embedding → 入库 → 用户 query → 近邻 → 进 Prompt」，标出 **你可配置的 3 个旋钮**（chunk 大小、TopK、元数据字段）。

```markdown
## 切片与入库
```
