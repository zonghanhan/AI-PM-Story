## 寓言故事

### 一、八秒白屏，像等一封没来的信

模型路由灰度两周后，林薇盯着新看板：**四楼占比** 从 **18% 降到 7%**，财务苏姐终于没再皱眉。

但运营小夏甩来一份 **用户录屏合集**，标题刺眼：**「小言让我干等」**。

第一条：用户问 **「发票怎么开」**——路由正确走了二楼 FAQ。日志显示 **总耗时 1.2 秒**，在工程眼里已经很快。录屏里却是：**用户发完问题 → 输入框上方转圈 8 秒 → 答案「啪」一下整段弹出**。

第二条：四楼复杂工单，用户问 **「产线 PLC 和 MES 对账步骤」**——路由升舱没错，总耗时 **14 秒**。录屏里用户 **第 3 秒就开始戳「停止生成」**，第 6 秒切到竞品客服，**小言的第 7 秒才出第一个字**。

小夏指着屏幕：**「用户感知的不是『总耗时』，是『我多久能看见第一个字』。你们后台 1.2 秒算完，前端为什么还白屏 8 秒？」**

大刘调出链路：**检索 0.3 秒 + 模型生成 0.9 秒 + 前端等「整段 JSON 收齐」再渲染 6.8 秒**——其中 **6.8 秒是产品没要求的「假装思考」动画**，加上 **非流式接口必须等最后一个 token**。

林薇愣住：**「我们优化了『谁回答』，却没优化『答案怎么出现在用户眼前』。」**

---

### 二、直播间不能等导播剪完片

第二天站会，大刘在白板上画了两条路：

```
非流式：模型写完整封信 → 一次性塞进信封 → 用户拆信
流式：  模型写一句推一句 → 用户边听边读
```

算法小陈补充：**「API 层面叫 stream=true。服务端每生成一小段 token，就推一条事件；前端订阅这条『水管』，收到就画字。」**

售后阿梅举手：**「像直播带货？主播嘴里的词还没说完，字幕已经往上滚——用户至少知道 **有人在回我**。」**

林薇想起老周说过的话：**「等整封信寄到才拆，和电报员边拍边发，用户心态完全不一样。」**

她当场拍板：**「下周灰度：二楼 FAQ + 四楼长回答，开启 **边生成边展示**。指标盯两个：**首字出现时间** 和 **用户中途离开率**。」**

---

### 三、抄到一半的维修手册

灰度第三天，投诉来了。

用户张先生问：**「自助更换滤芯的步骤」**。小言流式输出，前三秒已经滚出：

> **第一步：关闭进水阀。第二步：逆时针旋转滤芯盖——**

张先生 **截图发给车间师傅**，自己 **点了停止**——他觉得已经够了。

五秒后，屏幕其实还会继续生成 **「第三步：注意型号 X-200 与 X-300 螺纹相反」**——但用户 **已经走了**，也没看见 **文末 Grounding 引用的官方手册页码**。

车间按截图操作， **拧坏了螺纹**。工单升级为 **P1 停产风险**。

复盘会上，阿梅调出另一条录屏：用户 **边流式输出边复制**，复制到一半 **Markdown 表格只显示了一半表头**；粘贴到 Excel， **数据列全乱**。

林薇在白板写下 **流式带来的新坑**：

1. **用户以为「已经说完」**——其实后半段还有关键限定  
2. **可复制内容不完整**——表格、代码块、引用脚注可能还没闭合  
3. **停止 ≠ 没生成**——用户点停，后端可能还在烧 token（第 16 篇运费照付）

小陈苦笑：**「我们解决了『等太久』，制造了『走太早』。」**

---

### 四、拦不住的直播，门卫追不上

更棘手的是安全。

门卫老关的 **输出复检** 原本设计在 **「整段答案生成完毕」** 之后跑一遍。流式上线后，敏感词出现在 **第 47 个 token**——前端已经展示 **两秒**，复检才 **掐断并替换**。

合规部问：**「这两秒谁负责？用户截图了呢？」**

老关拿出 **直播延时播出** 的比喻：**「电视台可以晚 7 秒播；你们是 **零延时直播**，却想 **事后剪辑**——来不及。」**

工程方案变成三套并列：

| 方案 | 做法 | 代价 |
|------|------|------|
| A. 边流边扫 | 每收到 N 个 token 跑一次轻量敏感词 | 延迟略增，可能误杀半句话 |
| B. 短缓冲 | 前端晚展示 300–500ms，攒一小段再放出 | 首字变慢，但可撤回未展示的 |
| C. 高风险非流式 | 涉及报价、医疗、法律 **整段生成+复检后再展示** | 体验回退，但合规可控 |

林薇拍板：**「不能一刀切全开流式。路由（老何）分完楼层后，还要加一维：**这一趟允不允许直播式播出**。」**

---

### 五、停止键、光标与「到底说完没有」

体验组又提了一串 **「直播细节」**：

- **光标闪烁**：字在蹦时要有 **正在输入** 的反馈，否则用户以为卡死  
- **停止生成**：按钮要在 **第一 token 到达前** 就可点；点了要 **前端停画 + 后端 cancel 请求**，不能「界面停了后台还在写」  
- **「说完了」信号**：流结束要有 **明确收束**（勾选、折叠按钮、引用块一次性补齐），别让 FAQ 像没关上的水龙头  
- **长文锚点**：四楼 SOP 类答案，边流边出 **目录骨架**，用户不必等 14 秒才知道有没有他要的章节  

大刘实现 **cancel** 后，苏姐问：**「省下来的 token 账单能算清吗？」** 工程答：**「能统计 abort 率，但已生成的 token 通常仍计费——产品要在 PRD 里写清：**停止是体验功能，不是省钱按钮**。」**

两周后数据回来：

- **首字时间（TTFT）** 从 **8.1 秒 → 1.4 秒**（去掉假动画 + 流式）  
- **长回答会话中途离开率 ↓ 23%**  
- **不完整复制投诉 ↑**（新坑）  
- **流式敏感内容曝光事件 3 起**（缓冲 + 高风险非流式后归零）

林薇在周报写：**「用户不再等一整封信，但产品要管 **电报怎么拍、直播怎么延时、抄送能不能抄一半**。」**

运营小夏看完说：**「所以这不是工程开关，是 **体验合同**：什么时候直播、什么时候录播、什么时候允许用户插嘴叫停。」**

小陈点头：**「对。模型路由决定 **找谁写**；流式决定 **怎么把字交到用户手上**——两个旋钮，不能混成一个。」**

## 概念正解

**Streaming / 流式输出**，是模型生成回答时 **不等待整段完成**，而是 **每产出少量 token 就向客户端推送一次**，让用户 **尽早看到首字、边生成边阅读** 的传输与展示方式。它主要解决 **「感知等待」**：总耗时也许只少一点点，但 **首 token 时间（TTFT）** 大幅缩短，用户更愿意留下来看完。

回扣故事：**「等整封信」vs「电报边拍边发」** 对应 **非流式一次性返回 vs 流式增量推送**；**8 秒白屏** 是 **前端等非流式收齐 + 假 loading**；**抄到一半的滤芯步骤** 是 **用户提前离开/复制导致内容不完整**；**门卫两秒才掐断** 是 **流式场景下输出护栏必须边生成边检或缓冲**；**停止键与 cancel** 是 **前后端协同中断**；**高风险改录播** 是 **按场景关闭流式、整段复检后再展示**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写明 **哪些场景开流式**（FAQ / 长 SOP / 合规类）、**TTFT 目标**、**停止/cancel 行为**、**流结束 UI**、**表格/代码是否缓冲到闭合再展示** |
| **方案评审** | 问工程：**「stream 协议用什么（SSE/WebSocket）？cancel 是否真停推理？护栏是逐 chunk 扫还是短缓冲？」** 问设计：**「生成中可复制吗？引用脚注何时出现？」** |
| **指标设计** | 上线 **TTFT、生成中断率、流式会话完读率、abort 后仍计费 token 占比、流式敏感曝光次数**；区分 **二楼短答 vs 四楼长答** |
| **跨团队沟通** | 对运营：**「首字 1 秒内可见，不等于答案已完整」**；对合规：**「高风险类目走非流式或缓冲」**；对财务：**「停止按钮省不了多少 token，别当成本手段」** |

## 类比锚定

**核心类比：新闻 **直播** vs **录播重播**——传统 PM 做内容、客服、交易产品时都遇到过：**体育赛事实时解说**让用户愿意等；**医疗/财经重大发布**往往 **延时几秒** 防口误；**短信/推送**可以一条一条到，但 **合同条款** 必须 **整份 PDF 生成完再签**。流式输出是 AI 回答在「直播感」与「录播可控」之间的产品选择。

| 你熟悉的 PM 经验 | Streaming 要素 |
|------------------|----------------|
| 外卖 App **「骑手已接单 → 已到店 → 配送中」** 分步状态 | **增量推送**：不必等整单结束才给用户反馈 |
| 视频 **直播弹幕** vs **点播拖进度条** | **边生成边展示** vs **整段就绪再播放** |
| 直播 **延时 7 秒播出**，导播台可掐画面 | **短缓冲 + 流式护栏**：未展示内容可拦截 |
| 用户 **挂断电话**，交换机要真断线 | **停止生成 = 前端停画 + 后端 cancel** |
| 长文 **先出目录和摘要**，正文后台加载 | **骨架/首段优先**，降低长回答焦虑 |
| 合同页 **「生成中，请勿关闭」** 整页完成后才可签署 | **高风险答案非流式**：复检后再一次性展示 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **流式优化的是「感知时延」，不是总成本**——通常 **token 照付**；价值在 **TTFT、完读率、中途离开率**，别把它当省钱开关。
2. **开流式 = 签一份「播出方式」合同**——要定义 **何时直播、何时录播、何时缓冲**，与 **路由、护栏、Grounding** 联动，不是工程默认 true。
3. **用户会提前走、提前复制**——半成品答案带来 **操作风险**；需要 **收束信号、闭合后再可复制、关键限定前置** 等产品规则。
4. **停止要前后端一致**——只停 UI 不停推理，既 **浪费成本** 又可能在用户看不见处继续生成。
5. **展示层有细节债**——Markdown 半段渲染、引用晚到、表格截断；流式 PRD 必须写 **渲染与交互策略**，不能只写「开启 stream」。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像直播，越快越好」 | 直播 **不能撤回已播出的记忆**；AI 流式 **可以缓冲未展示 chunk** | 产品可在 **未展示区** 做护栏；已展示区要按 **合规留存** 设计 |
| 「首字快了，体验就一定好」 | **闪烁、跳动排版、反复改口**（模型自我修正时）反而晕 | 需要 **节流渲染、稳定布局、思考过程是否展示** 等体验决策 |
| 「像短信一条条到」 | 短信 **每条语义完整**；token 块可能 **半句话、半个 JSON** | 前端常要 **按句子/段落缓冲**，不能来一个 token 画一个 |
| 「录播更安全就全关流式」 | 长回答 **14 秒白屏** 流失更严重；合规场景才 **非流式** | **分场景开关**：FAQ 流式、报价单非流式，由 **路由 + 风险标签** 决定 |
| 「用户点停就不算了」 | 多数 API **已生成 token 仍计费**；cancel 只是 **不再继续** | PRD 写清：**停止是体验能力**；成本看 **abort 率与平均生成长度** |
