## 寓言故事

### 一、一道必须答对的题

峰会结束后的周一，林薇刚坐下，客服主管就冲进工位：

「林薇，商家在问：**接入你们智能客服后，夜间自动应答算不算『人工客服时长』？合同里有没有写？** 小言昨天在试用版里回：『一般算，具体看合同条款。』商家追问：『你们合同第几条？』——小言卡住了，开始重复『建议联系商务同事确认』。」

林薇打开后台，调出那段对话，心里一沉。

小言的语气依然专业、流畅，像上周写演示话术时一样自信。但这不是创意文案——这是**公司内部合同条款**。答错一个字，商务就要介入擦屁股。

她找到老周：「Prompt 我已经写好了，角色、约束、禁止瞎编，全上了。怎么还是不行？」

老周推了推眼镜：「你让小言**闭卷考试**。合同、价目表、售后 SOP——这些不在他『读过的全世界』里，在你们 Confluence 里。闭卷时他只会接『听起来最像客服话术』的下一句。」

林薇皱眉：「那我把合同贴进 Prompt？」

老周：「你试试。」

---

### 二、整本合同塞进一封信

林薇从法务共享盘下载《标准商务合同 v3.2》，全文 47 页，复制粘贴进 Prompt：

> **角色**：你是本公司智能客服，只能依据下方合同原文回答，不得推测。  
> **合同全文**：  
> [47 页正文……]  
> **用户问题**：夜间自动应答是否计入人工客服时长？请引用条款编号。

小言这次确实引用了条款——**《标准商务合同 v2.1》第七条**。

林薇核对文档：当前版本是 **v3.2**，第七条讲的是「数据安全」，跟客服时长毫无关系。小言引用的编号存在，内容却是**把不同章节里出现过的短语拼在一起**，读起来像真的，查原文全不对。

更糟的是，林薇又试了第二个问题：「Pro 版和专业版的价格差多少？」

小言沉默很久，最后回：「根据您提供的上下文，未能找到完整价目信息。」——47 页合同里其实夹着一页《产品价目表附录》，但淹没在长篇法律术语里，**整本塞进去，它反而找不到那一页**。

老周凑过来看日志：「两个问题。第一，**上下文窗口有限**——你塞 47 页，前面或后面会被截断，模型『看到』的只是一部分。第二，**没有检索**——就像把整箱档案倒扣在桌上，让实习生自己翻，他只会抓最显眼的那几段话，不管是不是你要的那条。」

林薇深吸一口气：「所以不是 Prompt 不够长，是**给法不对**？」

「Bingo。」

### 三、库房管理员阿寻

老周在白板前画了一个新角色：

「我们给系统加一个人——**阿寻**。他不回答问题，只做一件事：**用户问什么，先去资料库把最相关的几页找出来**，再交给小言。」

资料库怎么建，老周用 PM 能听懂的话讲：

1. **入库**：把合同、价目表、售后 SOP、FAQ 等文档上传，系统按段落切成一块一块的「**切片（Chunk）**」——就像把厚书撕成一张张索引卡片，每张卡片顶上有标题和来源。
2. **建索引**：每张卡片做一份「**语义摘要**」（技术上叫向量化），方便按**意思**搜索，而不只是按关键词匹配——用户问「夜间算不算人工」，也能搜到写着「自动应答服务时长不计入坐席 KPI」的那张卡。
3. **检索**：用户提问时，阿寻在毫秒级内捞出最相关的 **Top-K 张卡片**（比如 3–5 张）。
4. **增强**：把这 3–5 张卡片和用户问题**一起**塞进 Prompt，再让小言生成回答。

老周画完流程：

```
用户提问
   ↓
阿寻检索资料库 → 捞出 Top-K 相关切片
   ↓
Prompt = 系统指令 + 检索到的切片 + 用户问题
   ↓
小言生成回答（并尽量引用切片来源）
```

林薇问：「所以小言还是闭卷？」

老周摇头：「**开卷**——但开卷范围不是全世界，是阿寻刚帮他翻到的那几页。这叫 **先检索，再生成**。」

### 四、第一次像样地回答

同一问题再次测试：「夜间自动应答算不算人工客服时长？请引用合同条款。」

日志里，林薇先看到阿寻的检索结果——3 张卡片：

| 排名 | 来源 | 片段摘要 |
|------|------|----------|
| 1 | 《标准商务合同 v3.2》§4.3 | 自动应答模块产生的服务记录不计入「人工坐席有效时长」…… |
| 2 | 《产品 FAQ》Q17 | 夜间模式、节假日模式下的应答属于系统自动服务…… |
| 3 | 《商务话术指引》§2.1 | 对外解释时避免使用「替代人工」表述，应强调「分流重复咨询」…… |

然后小言收到的是**这 3 张卡片 + 用户问题**，而不是 47 页全文。

小言回复：

> 「根据《标准商务合同 v3.2》**第 4.3 条**，通过自动应答模块产生的服务记录**不计入**人工坐席有效时长。夜间自动应答属于系统自动服务范畴（参见产品 FAQ Q17）。如需向商家书面确认，可在商务邮件中引用该条款编号。」

林薇逐字对照原文——**一致**。

她又追问：「Pro 版比专业版贵多少？」

阿寻捞出《价目表附录》卡片；小言回答：「Pro 版年费 28,800 元，专业版 18,800 元，差价 10,000 元（来源：价目表附录，2026-Q2 生效）。」

客服主管在群里回了一个字：「稳。」

### 五、一次新的翻车

周三下午，商家问了一个看起来简单的问题：

「你们支持**抖音小店**接入吗？」

小言自信回答：「支持。详见《渠道接入清单》第 12 项抖音小店 API 对接说明。」

林薇去查《渠道接入清单》——**第 12 项是快手小店**。抖音小店在 **第 8 项**，状态标注为「**规划中，预计 Q3**」。

她找老周复盘。阿寻的检索日志显示：Top-1 卡片确实是第 12 项（快手），Top-3 里才有第 8 项（抖音，规划中）。但小言在生成时**只看了排名第一的片段**，把「小店接入」理解成了「任意小店都已支持」。

老周 chalk 板上写了两个词：**检索不准** 和 **生成瞎发挥**。

「RAG 不是装上就完事。阿寻捞错卡片，小言会跟着错；阿寻捞对了，小言也可能**忽略『规划中』这种否定信息**，仍然说『支持』。产品上要三件事：检索质量、Prompt 里强制『仅依据引用作答』、答不上来说不知道。」

林薇补充：「还有——价目表更新了，资料库没同步，阿寻捞到的就是旧卡片。」

老周点头：「**资料库不是一锤子买卖**，跟 Confluence 一样要版本管理和更新机制。你作为 PM，要定义：哪些文档入库、多久更新、检索不到时用户看到什么。」

### 六（谜底渐显）

周五复盘会，林薇在白板上画了三层：

```
┌──────────────────────────────────────────┐
│  用户问题：「合同里夜间应答算不算人工？」     │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│  阿寻（Retriever 检索器）                  │
│  在向量资料库中按语义搜索 → Top-K 切片      │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│  Augment 增强 Prompt                      │
│  = 系统指令 + 检索切片 + 用户问题           │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│  小言（Generator 生成器 / LLM）            │
│  基于给定切片组织语言 → 带引用的回答         │
└──────────────────────────────────────────┘
```

她在「阿寻」旁边写下全称，念给会议室听：

> 「这套做法叫 **RAG**——**R**etrieval **A**ugmented **G**eneration，**检索增强生成**。  
> **检索（Retrieval）**：阿寻从资料库找相关片段，不是让小言凭记忆瞎编。  
> **增强（Augmented）**：把找到的片段**塞进 Prompt**，给小言「开卷范围」。  
> **生成（Generation）**：小言仍然负责把片段组织成用户听得懂的话——**查资料的是阿寻，说话的是小言。**  
> 上一篇 Prompt 解决『怎么问』；这一篇 RAG 解决『**依据什么答**』——当答案必须来自**你的**文档，而不是模型的通识。」

老板问：「跟 Fine-tuning 微调有什么区别？」

林薇答：「微调像送小言去培训班，把知识写进脑子，周期长、改文档要再训；RAG 像给他配一个**随时可更新的资料库和图书管理员**，合同改一版，重新入库就行，**更适合 FAQ、政策、产品手册这类经常变的知识。**」

## 概念解释：什么是 RAG（检索增强生成）？

**RAG（Retrieval-Augmented Generation）** 是一种让大语言模型**先查资料、再回答**的架构：用户提问时，系统先从外部知识库中**检索**与问题最相关的文档片段，把这些片段**增强（注入）**到 Prompt 里，再交给 LLM **生成**最终回答。

典型流程（四步）：

| 步骤 | 做什么 | 故事角色 |
|------|--------|----------|
| **1. 建库** | 文档切片（Chunking）→ 向量化（Embedding）→ 存入向量数据库 | 把合同撕成索引卡片，放进库房 |
| **2. 检索** | 用户问题也向量化，在库中找语义最相近的 Top-K 片段 | 阿寻按问题找卡片 |
| **3. 增强** | 将检索结果 + 用户问题组装成 Prompt | 开卷范围交给小言 |
| **4. 生成** | LLM 基于 Prompt 输出， ideally 带引用来源 | 小言组织语言作答 |

**为什么需要 RAG？**（接前两篇）

- **LLM** 擅长「接龙」，但训练数据**不包含**你公司的内部合同、最新价目、未公开 roadmap。
- **Prompt** 可以约束「怎么答」，但**不能凭空制造**库里才有的 fact；把整库塞进 Prompt 又会超长、噪声大。
- **RAG** 在「不改模型权重」的前提下，让回答** grounded（有据）** 于可更新的企业知识，是 AI 客服、企业知识问答、Copilot 类产品的**标配方案**。

作为 AI 产品经理，你要关心的不是向量公式，而是：

- **知识从哪来、多久更新**（数据源与版本）
- **检索准不准**（用户问法多样，能否捞对片段）
- **生成老不老实**（是否严格依据检索结果，能否标注来源）
- **检索不到怎么办**（降级话术、转人工、澄清追问）

## 类比锚定：从「常规 PM 工作」映射到 RAG

| 你熟悉的产品经理工作 | 对应 RAG 概念 |
|----------------------|---------------|
| 写 PRD 前先搜 Confluence / 飞书文档 / 历史 PRD | **检索（Retrieval）** |
| 把相关文档链接贴进需求邮件给研发 | **增强 Prompt（Augment）** |
| 实习生根据你给的资料写答复，而非凭印象编 | **生成（Generation）** |
| 把 200 页规范拆成章节，方便按需查阅 | **文档切片（Chunking）** |
| 用标题 + 标签 + 全文搜索找「那一条规定」 | **向量检索 / 语义搜索** |
| Confluence 文档过期未更新，研发按旧规范开发 | **知识库未同步 → 检索到过时内容** |
| 「请仅依据附件作答，找不到就写『需确认』」 | **RAG Prompt 约束 + 拒答策略** |
| 法务只批准「引用官方话术库」的对外口径 | **Grounding：回答必须可追溯到来源** |

**核心类比一句话：**

> **RAG = 给 LLM 配一个「会搜索的内部 Confluence + 一个只许照着搜到的片段说话的答复流程」。** 你平时答老板问「这个需求合规吗」时，一定是**先翻制度再开口**——RAG 就是把这套动作产品化、自动化。

## 故事深化：概念在场景里如何「运行」

| 场景中的动作 | RAG 如何运转 | 边界 / 易踩的坑 |
|--------------|--------------|-----------------|
| 只改 Prompt，不查内部文档 | 纯 LLM 闭卷 → 话术流畅但条款可能编造 | **坑：把「禁止瞎编」写进 Prompt 挡不住 hallucination** |
| 47 页合同整篇塞进 Prompt | 超 context / 信息淹没 → 引错条款或答找不到 | **坑：以为「给全量就准确」——缺检索，噪声比信号大** |
| 阿寻检索 Top-3 切片 + 强制引用 | 开卷范围小、可核对 → 条款与价目答对 | **解法：Chunk 大小、Top-K 数量、引用格式要调优** |
| 问「抖音小店」却捞到「快手」片段 | 检索语义相近但实体错误 → 生成张冠李戴 | **坑：检索质量是 RAG 天花板；要测 bad case 问法** |
| 检索到「规划中」仍回答「支持」 | 生成阶段忽略否定信息 | **坑：Prompt 需写「仅据引用；状态为规划中则明确未上线」** |
| 价目表更新但资料库未 re-index | 检索到旧切片 → 报价错误 | **坑：知识库是运营对象，不是一次性配置** |
| 用户问「你们 CEO 手机号」 | 资料库无此信息 | **需产品定义：拒答 / 转人工 / 不检索敏感域** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **RAG = 检索 + 增强 Prompt + 生成**，三者缺一不可；只建向量库不约束生成，仍会胡说。
2. **知识存在于外部可更新资料库**，适合政策、手册、FAQ；不要求 retrain 模型即可换文档。
3. **准确性上限由检索决定**：捞错片段，后面全错；PM 需投入 **评测集（哪些问法必须捞对）**。
4. **RAG 不保证 100% 正确**：生成仍可能曲解引用；关键场景要 **引用溯源 + 人工兜底 + 定期审计**。
5. **与 Prompt 的关系**：RAG 是 Prompt 的「**动态附件**」——每次问题不同，附件（检索切片）不同。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「RAG = 先搜 Confluence 再答」 | 人会跨多篇文档综合推理 | 检索通常只给 **Top-K 局部片段**，复杂推理链可能跨切片断裂，需 **Agent 多步检索** 或更大 context |
| 「资料库 = Confluence」 | Confluence 有目录结构、人工浏览 | 向量库靠 **语义相似度**，没有人类「目录感」；近义问法、实体混淆（抖音/快手）是典型失败模式 |
| 「开卷就不会错」 | 开卷考试也会答错题 | 生成阶段仍会 **hallucination**，可能捏造不存在的条款编号或合并多个片段 |
| 「更新文档 = 更新 Confluence 页面」 | 改网页即生效 | RAG 需 **重新切片、重新 embedding、重建索引**，有 pipeline 延迟与版本一致性问题 |
| 「RAG 替代 Fine-tuning」 | 二选一 | 很多产品 **RAG + Prompt + 轻量微调** 组合使用；风格、格式、领域术语有时仍需微调 |
| 「切片像撕索引卡」 | 卡片独立 | 切片 **切断上下文**，表格跨页、图文混排、脚注易丢；Chunk 策略（大小、重叠、按标题切）是产品/工程重点 |

## 了解途径与学习资料

### 入门路径（产品经理向，无需数学）

1. **动手体验（1–2 小时，最高优先级）**  
   - 使用带「知识库 / 文档问答」的产品（如 ChatGPT 自定义 GPT 上传 PDF、Coze / 通义 / Kimi 上传文件、Notion AI Q&A）  
   - 同一问题测三次：**不上传文档** vs **上传文档** vs **上传错误/过时文档**  
   - 观察：引用是否准确、检索不到时怎么说

2. **画一张你的业务 RAG 流程图（半天）**  
   - 列出：知识来源 → 谁更新 → 用户问什么 → 答错代价 → 检索不到怎么办  
   - 这就是 AI 知识问答产品的 **PRD 骨架**

3. **系统补课（3–5 天，选读）**  
   - 吴恩达短课：[*Building Systems with the ChatGPT API*](https://www.deeplearning.ai/short-courses/building-systems-with-chatgpt/) 中关于 LangChain、文档加载与问答链的章节（可跳过代码看流程图）  
   - 重点理解：Document Loader → Splitter → Vector Store → Retriever → QA Chain

4. **建立评测意识**  
   - 准备 20 个「必须答对」的真实用户问题 + 标准答案出处  
   - 任何 RAG 改版前后都跑一遍——PM 的「检索召回率 / 答案 grounded 率」比背公式重要

### RAG 产品 Checklist（可保存到笔记）

```markdown
## 知识侧
- [ ] 知识来源与 owner（谁负责更新）
- [ ] 更新频率与 re-index 触发方式
- [ ] 切片策略（按标题 / 固定长度 / 表格单独处理）
- [ ] 敏感文档是否入库、权限隔离

## 检索侧
- [ ] Top-K 取几条？过长则噪声，过短则漏信息
- [ ] 评测集：20+ 真实问法，测召回与错召
- [ ] 实体混淆、口语问法、多跳问题 bad case

## 生成侧
- [ ] Prompt：仅依据引用作答；无引用则拒答
- [ ] 是否展示来源（文档名、章节、链接）
- [ ] 幻觉抽检与转人工规则

## 体验侧
- [ ] 检索中的等待反馈
- [ ] 「找不到」时的降级话术
- [ ] 用户反馈「这条答错了」如何回流优化
```

### 下一篇预告

RAG 让小言能「查资料再说话」——但当任务变成「先查合同、再查库存、再帮用户填退货单、最后发邮件通知仓库」这种**多步骤、要调外部系统**的流程时，单靠一次检索 + 一次生成就不够了。下一篇将拆解 **Agent（智能体）**：当 AI 不只会说，还会**规划、用工具、分步执行**。
