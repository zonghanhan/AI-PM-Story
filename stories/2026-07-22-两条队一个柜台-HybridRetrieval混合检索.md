### 本集业务场景（先读这段，再进故事）

| 项 | 内容 |
|----|------|
| **产品** | **内小言（应用 C）**：员工在 **飞书** 里问公司制度、福利、内部 FAQ |
| **本集终端用户** | **李女士**：某租户下的普通员工，今天只想查 **年假制度原文** |

| **系统里已有的两类「材料库」** | **① 公司制度库（公共 RAG）**：年假制度 PDF、操作手册等，租户内员工可查；**② 用户记忆索引（Memory）**：便签、工单指针、偏好摘要——本应 **按用户、按应用隔离** |
| **本集要解决的产品事故** | 李女士在内小言问年假时，回答里出现了张先生业务里的 **对账口径**——不是模型瞎编，而是 **检索把多路材料混成一个排行榜，且跨用户热词没做权限** |
| **PM 对齐一句** | 像企业内部搜索：同时可能命中 **Confluence 制度** 和 **某人的草稿便签**；必须先弄清 **搜了谁的、哪些库**，再谈分数高低 |

**时间线（故事里的「发生了什么」）**

1. **上周**：张先生在售后/对账场景里，和助手确认了「B 线」口径；系统为他写了记忆便签，运营还把部分高频摘要塞进了 **跨用户「近期热词池」**（为降重复劳动）。
2. **本周·事故日**：李女士打开 **内小言**，问年假；小言引用「检索材料」却念出 B 线对账段落；李女士投诉。
3. **当日复盘**：林薇拉日志，按请求还原「这一轮到底搜了哪几路、Top 结果从哪来」。
4. **第一周改造**：上混合检索（分路 → 权限过滤 → 再合并）；共享热词池下线。
5. **第二周**：张先生续办对账——权限没串了，但 **合并策略太松**，年假制度又挤进对账上下文。
6. **第四周**：按意图收紧参与队列后，两类问法各走各的，指标好转。

## 寓言故事

### 一、事故现场：李女士问年假，助手念出对账口径

先把现场钉死。

**地点 / 入口**：飞书里的 **内小言**（应用 C）——员工查制度用的那个窗口，不是修小言的报修聊天，也不是账小言的入账确认单。

**谁在问**：李女士，普通员工。她没有正在进行的对账工单，也没提产线、停线、MES。

**她发出的原话**：

> 「我们公司年假按自然年还是入职周年？有没有制度原文？」

**小言实际干了什么（用户视角）**：气泡开始 **流式** 出现，开头仍是「根据检索到的材料……」，紧接着正文却是：

> 「B 线 MES 与 PLC 以 `batch_id` 对齐，停线三条需备注导出批次……」

（白话：这是在讲 **哪条产线对账时，制造执行系统与控制器日志用哪个批次号对齐，停线记录要备注哪次导出**——和年假制度无关。）

李女士截图进群：**「我问年假，它跟我讲产线对账？」**

林薇（产品）立刻做了三件事，把「发生了什么」从投诉还原成可复盘的请求：

1. 找到 **这一通请求的 trace_id**；
2. 看 **进模型上下文之前，检索返回了哪些片段**（带来源标签）；
3. 对照 **李女士账号** 与 **片段所属用户 / 应用**。

结论先说在前面：**模型没有「幻想」出一段对账话术**；那段话 **确实来自检索结果里的某几条**，而且排进了本次回答所用的 Top 材料。事故发生在 **「找材料」这一层**，不是生成层临场发挥。

门卫老关（第 18 篇护栏）把工单标成 **P0 串味**：对用户表现为答非所问；对合规表现为 **看见了不该看见的业务上下文**。

---

### 二、日志还原：这一轮请求，系统一步步做了什么

站会上，林薇把同一轮请求拆成时间轴（产品能听懂的版本）：

| 步骤 | 系统实际做了什么 | 业务含义 |
|------|------------------|----------|
| 1 | 收到李女士在 **内小言** 的一句话 | 入口是 **制度助手**，意图应是 **查 HR 政策** |
| 2 | **同时** 对两路索引发起向量检索（Embedding，第 8 篇） | **路 A**：公司制度库；**路 B**：`user_memory` 记忆向量表 |
| 3 | 路 A 返回约 5 条制度相关片段；路 B / 热词侧也返回约 5 条 | 多路都「有命中」，不等于「都该给李女士」 |
| 4 | 把各路结果 **丢进同一个相似度排行榜**，「分高的赢」 | **公司制度** 与 **对账热词** 在抢同一张榜 |
| 5 | Top3 里出现了 **「B 线口径」** 摘要 | 模型按「检索到的材料」组织回答 → 用户听到对账 |

**「B 线口径」是怎么进榜的？** 这里要把上周业务说清楚：

- 张先生上周在 **设备售后 / 对账续办** 场景里，和助手对齐过：**B 线** 以 `batch_id` 对齐 MES（制造执行系统）与 PLC（产线控制器）日志；停线三条要带导出批次备注。
- 系统为他写了 **用户记忆便签**（第 22 篇 Memory）。按设计，这类便签应带 **`user_id` + 应用域**，**李女士账号读不到张先生本人的便签**——这一层隔离，测试环境里其实已经做了。
- 但运营为了「少重复解释高频口径」，另做了一张 **跨用户的「近期高频摘要池」**：把若干工单里反复出现的结论抽成热词索引，**没有打「仅本人 / 仅应用 B」标签**，却挂在了 **内小言也能搜到的记忆检索入口** 上。
- 李女士问「年假」时，向量检索觉得某些热词片段「也有点像」，热词池条目 **挤进公共问题的 Top3**。

工程小刘原先的设想是：**「用户分不清查制度还是查记忆，统一检索、分数高的赢就行。」**  
算法小陈指着白板说：统一入口可以，**统一排行榜不行**——李女士问的是 HR 政策，却排到了 **隔壁业务线的工单热词**；这不是幻觉，是 **排错队**。

他又补上第三路，免得产品以为「只有制度 vs 记忆」两分法：

```
队 A：公司库房（RAG）—— 制度 / 手册，租户内可读、版本受控、适合带出处引用
队 B：用户抽屉（Memory）—— 便签 / 工单指针 / 偏好，仅本人（及授权），可过期
队 C：关键词 + 结构化—— 文号、「第 7 章」、入职日、司龄；补语义检索的漏与查表需求
```

林薇把产品要拍板的问题写成三句，贴在复盘文档最上头：

1. **用户这句话进来，系统默认查哪些队？**  
2. **各队结果怎么合并、谁先谁后？**  
3. **谁有资格看见哪一条？**（看不见的，不能靠「分数低一点」蒙混，要 **进不了候选集**）

### 三、第一周改造：分路召回，先安检，再摆盘

团队没有「关掉记忆」——张先生那类 **续办工单** 仍然需要队 B——而是上了 **Hybrid Retrieval（混合检索）**。改造后，**同类事故请求** 的理想路径被写成：

**当用户在内小言问「年假怎么算」时：**

1. **意图粗分**（规则 + 轻量分类，可复用模型路由第 19 篇的分诊思路）：判为 **制度 / 福利 / 通用 FAQ**。  
2. **参与队列**：开 **队 A + 队 C（关键词）**；**默认不开队 B**，除非用户明确说「结合我的入职情况」。  
3. **分路召回**：A 取 Top-30，关键词取 Top-10——**各算各的**，不在数据库层先糊成一张大表。  
4. **权限硬过滤（安检）**：  
   - 任何 `source=user_memory` 的片段，必须 `user_id == 当前登录用户`；  
   - **跨用户热词池下线**，改回「仅会话内 / 仅本人摘要」；  
   - 片段若带其他用户 ID 或应用 B 工单域标签，在内小言制度意图下 **直接丢弃**，不进 Rerank。  
5. **合并与重排**：候选集带上来源标签（如 `policy`）；制度类问题给 `policy` **加权**；再用 Rerank（第 10 篇）精排后，才塞进上下文给模型。

小陈强调：**混合不是把两堆材料搅一锅**；是 **分厨配菜，最后一道菜再摆盘**。

**权限在这里具体指什么？**  
指：**当前这次请求的登录身份 + 应用入口 + 数据标签**，决定某条片段能不能进入「可被模型看见的候选集」。  
**隔离**是要达成的结果（张先生的对账抽屉不出现在李女士的年假问答里）；**权限过滤**是流水线上的动作（不匹配就删掉）。过滤发生在 **打分/Rerank 之前**，不是指望模型「自己别念」。

### 四、第二周翻车：人没串了，合并又松了

灰度第二周，**另一条业务线**也来验收。

**现场**：张先生续办对账（售后/对账场景，不是李女士那个年假窗口）。  
**他发出的原话**：

> 「还是按 B 线，把停线三条带上。」

**这一轮系统做对了什么**：

- 意图判为 **续办个人工单** → **队 A + 队 B 并行**（合理）；  
- 队 B 命中 **他自己的** 便签指针（`user_id` 匹配）；  
- 队 A 命中 MES 导出说明类手册片段。

**又做错了什么**：

- 合并阶段，Rerank 仍把 **「年假制度 2024 版」** 塞进了最终 Top 材料；  
- 原因可追溯：张先生上周聊天里 **吐槽过加班和「年假」**，便签向量与制度库里的「年假」条文在空间上靠近，合并时 **没有按「续办对账」意图抑制无关制度 chunk**。

**用户侧后果**：

- 上下文窗口（第 9 篇「圆桌」）被无关制度占掉一截；  
- 真正需要引用的「停线三条」Grounding（第 11 篇）被挤到截断线外；  
- 结构化输出字段看起来齐，**出处却对不上本轮工单**。

评测（第 10 篇）补了 **混合检索** 用例：

- **福利 / 制度场景**：Top5 **不得**含他人记忆、跨应用工单词、无关对账热词；  
- **续办工单场景**：Top5 **必须**含工单指针；**不得**因聊天里提过「年假」就让年假制度插队。

林薇在 PRD 写死 **合并策略表**（意图 → 参与队列 → 禁止项）：

| 用户意图（产品定义） | 参与队列 | 合并规则（示例） | 禁止 |
|----------------------|----------|------------------|------|
| 公司制度 / 福利（李女士问法） | A + 关键词 C | 优先 `source=policy`；Rerank 输入带来源标签 | B；跨用户热词；应用 B 工单域 |
| 续办个人工单（张先生问法） | A + B | B 的指针优先；A 补操作手册 | 跨用户 B；与工单无关的制度大段 |
| 纯闲聊 | 仅当前 session | 不查 B 档案柜 | 为「显得聪明」硬捞库房 |

她在复盘里写：**混合检索的坑，常常是两头——该并行的没并行（漏记忆）；不该并行的瞎并行（串味、挤窗口）。**

### 五、修好之后：两类问法，各走各的柜台

灰度第四周，用两条真实路径验收「发生了什么」是否符合设计。

**路径 1｜李女士再问年假（内小言）**

- 意图：制度 → 只开 A + C；  
- 回答先引用制度库第 7 章片段，Grounding 链到 PDF 页码；  
- 上下文里 **不再出现** B 线对账、张先生便签、跨用户热词。

**路径 2｜张先生续办对账**

- 意图：续办 → A + B；  
- 界面上同时出现 **本人便签指针** 与 **库房操作手册**；  
- **年假条文不再插队**；停线三条引用能对上导出批次。

指标会上（业务结果，不是概念口号）：

- **「答非所问」工单 ↓ 41%**（其中归因 **混检串味** 的：每月 17 单 → 2 单）；  
- **单次检索延迟 ↑ 约 80ms**（多路召回 + Rerank）——用路由与热制度 chunk 缓存换回来；  
- **Token 均值 ↓ 6%**（少把无关片段塞进上下文）。

运营小夏问：**「所以用户还是一个输入框？」**

林薇：**「对用户是一个框；对系统是 **分流器 + 多柜台 + 安检（权限过滤）+ 摆盘（合并）**。记忆和 RAG 要一起上场时，学名叫 **Hybrid Retrieval——多源、分路、再合一**。」**

张先生听完复盘，总结事故体感：**「上次就像在一个窗口办业务，却听到隔壁窗口在念我家账本。」**——对应的正是：公共制度问答，捞进了 **不该属于这条业务、不该属于这个人** 的材料。

## 概念正解

**Hybrid Retrieval / 混合检索**，指在一次回答用户之前，**从两种及以上不同来源或检索方式**（例如 **向量语义检索、关键词/BM25、结构化数据库、用户记忆索引、实时 API**）**分别召回候选材料**，再经过 **权限过滤、去重、融合打分或重排序**，选出最终塞进上下文（或交给模型）的片段集合。它解决的是：**真实产品里「该查什么」往往不是单一路径**——纯 RAG 会 **漏个人上下文**，纯记忆会 **答不了公司制度**，纯关键词又 **听不懂口语**；若 **简单把多路结果按一个相似度榜硬拼**，就会出现 **串权限、挤窗口、答非所问**。

回扣故事（对齐「发生了什么」）：

| 情节 | 对应机制 |
|------|----------|
| 李女士在内小言问年假，却念出 B 线对账 | 多源结果 **未分意图、未做权限** 就进同一排行榜；跨用户热词污染 |
| 「统一检索，分数高的赢」 | 忽视 **来源边界与应用域** 的错误产品默认 |
| 分路 Top-K → 安检 → 带标签合并 | 混合检索的标准形态 |
| 张先生续办时年假制度插队 | **合并阶段**未按意图抑制无关路（隔离对了，摆盘仍错） |
| 延迟↑、Token↓、串味工单↓ | 混合检索的 **体验 / 成本 / 合规** 要一起看 |

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写清 **产品入口（如内小言）**、**两类用户问法**、每路 **数据源 / K / 权限 / 触发条件**；附 **意图→参与队列** 表 |
| **方案评审** | 问算法：**「合并在 Rerank 前还是后？跨路去重键是什么？」** 问工程：**「记忆与制度是否分表？日志能否按 trace 列出本轮各路命中 ID？」** |
| **指标设计** | **串味率**、无关片段占比、分意图 Grounding 失败、P95 检索延迟、Token 浪费 |
| **跨团队沟通** | 对法务：**「用户记忆不得与公共库无标签同榜竞争」**；对客服：**「答错先查检索摆盘，再查模型」** |

## 类比锚定

**核心类比：机场安检与登机口——** **国内航班旅客、国际旅客、贵宾、货运** 都从 **同一个大门** 进航站楼（用户只打一个提问框），但 **安检通道、证件校验、登机口** 不同；**行李（检索片段）** 在 **分拣中心（合并/Rerank）** 按 **航班号（意图）** 装车，**不能把货运托盘塞进经济舱座位（上下文窗口）**。传统 PM 做 **多端/多角色入口** 时熟悉：**同一搜索 UI 背后往往是 **多索引、多过滤、再聚合结果**。

| 你熟悉的 PM 经验 | Hybrid Retrieval 要素 |
|------------------|-------------------------|
| 统一搜索框，背后 **商品库 + 店铺 + 评价** 多索引 | **多路召回**（RAG / Memory / 关键词 / 结构化） |
| **角色权限** 决定能看见哪些菜单与数据 | **分路权限过滤**（`user_id`、来源标签、应用域、敏感域） |
| 搜索结果 **综合排序**（销量、相关性、广告位） | **融合打分 / Rerank / 业务加权** |
| 「猜你想搜」与 **精确 SKU 码** 两套逻辑 | **语义向量 + 关键词** 互补 |
| 机票 **联程**：前一段误机影响后一段 | **一路噪声进合并** → **挤占上下文、带偏生成** |
| 运营 **大促缓存** 热词页，换延迟 | **热制度/热 FAQ 缓存**，换多路成本 |

---

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **混合检索是「找材料」层的架构**，不替代 Prompt、Memory 存储设计或 Rerank 模型本身，而是 **规定多源如何同时上场**。
2. **必须先分路再合并**：每路 **自己的 K 与权限**；合并阶段要带 **来源元数据**（政策/个人/手册/应用域），否则 **无法做产品策略加权**。
3. **意图决定参与队列**：不是每次请求都 **A+B+C 全开**；**过度并行** 涨延迟、涨 Token、引噪声。
4. **权限是硬约束，不是软分数**：他人记忆、跨应用热词 **过滤掉**，不能指望 Rerank **「学会别选」**；隔离是结果，过滤是手段。
5. **要与 Grounding、窗口、成本联调**：无关片段进 Top5 的代价，往往比 **「少召回一路」** 更大。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像综合搜索，相关就行」 | **相关但无权、相关但无关意图** 的片段仍会害生成 | 需要 **意图门控 + 来源标签 + 否决规则** |
| 「多路一定比单路准」 | 路越多 **噪声、延迟、运维面** 越大 | 混合是 **按需组合**，要 **可观测、可降级（某路失败只走 A）** |
| 「Rerank 能摆平一切」 | Rerank **看不见权限** 时会把脏数据排前面 | **过滤在 Rerank 前**；Rerank 吃 **已消毒的候选集** |
| 「和 RAG 是一回事」 | RAG 常指 **单知识库管线**；Hybrid 是 **跨库跨索引的编排** | PRD 里写清 **「本需求是扩 RAG 还是上混合架构」** |
| 「记忆检索算 Hybrid 的一路」 | Memory 还有 **非向量注入**（档案字段直读） | 一路之内也可 **子混合**；产品图要 **落到表名与 API** |

### 动手小练习（约 15 分钟）

1. 画你负责的一个助手：**列出 3 路可能的「找材料」来源**——每路写 **谁可读、默认 K、何时触发**。  
2. 写 **1 条用户问句**（如「年假几天」）和 **1 条续办句**（如「还是上次口径」）——各填 **允许进 Top3 的来源标签白名单**，并写清 **产品入口**。  
3. 假设两路各召回 30 条，**合并后仍无关制度插队**——列 **2 个可观测指标** 证明问题在检索层而非模型层（例如：Top5 来源分布、串味工单归因）。
