## 寓言故事

### 一、问政策，答出口径

记忆灰度第三周，李女士在飞书里问小言：**「我们公司年假按自然年还是入职周年？有没有制度原文？」**

小言 **流式** 开头很稳：**「根据检索到的材料……」** 接着却念出一段：**「B 线 MES 与 PLC 以 `batch_id` 对齐，停线三条需备注导出批次……」**

李女士截图发群：**「我问年假，它跟我讲产线对账？」**

林薇调日志：同一轮请求里，**Embedding 检索（第 8 篇）** 从 **公司制度库** 捞了 5 条，又从 **`user_memory` 向量表** 捞了 5 条——**相似度分数挤在同一排行榜上**。张先生昨天写在 **李女士不该看见的测试便签** 已被隔离，但 **「B 线口径」** 被写进了 **共享的「近期高频摘要池」**（运营为降重复劳动做的 **跨用户热词索引**），**没打「仅本人」标签**，于是 **混进了公共问题的 Top3**。

门卫老关（第 18 篇）把这条标成 **P0 串味**：不是模型胡说，是 **「找材料」的入口只有一扇门**。

---

### 二、一个搜索框，两条队

站会上，工程小刘提议：**「用户哪分得清『查制度』还是『查我的记忆』？** **统一检索，分数高的赢**。」**

算法小陈摇头，在白板上画了两条队：

```
队 A：公司库房（RAG）—— 全员可读、版本受控、Grounding 可引用
队 B：用户抽屉（Memory / 个性化索引）—— 仅本人、可过期、可能含工单指针
```

**「混成一队，柜台就一个——** **李女士问 HR 政策，却排到别人的工单热词，不是幻觉，是 **排错队**。」**

大刘补：**「还有队 C 呢——** **关键词**（制度文号、年假、第 7 章）和 **结构化字段**（她的入职日、司龄）。** 只靠向量，『年假』和『年假的口语吐槽』在空间里 **挨得近**。」**

林薇问产品最该拍板的一句：**「用户一句话进来，** **系统默认查哪些队？** **结果怎么合并？** **谁有资格看见哪条？」**

---

### 三、分路召回，先过滤再上桌

第一周方案不是「关掉记忆检索」，而是 **Hybrid（混合）**：

1. **意图粗分**（规则 + 轻量分类，可走模型路由第 19 篇分诊思路）：像 **制度/福利/通用 FAQ** → **优先队 A**；像 **「接着上次对账」「还是 B 线」** → **队 A + 队 B 并行**；像 **「我入职多久」** → **队 C 结构化直取 + 队 A 补条文**。
2. **分路召回**：每路 **各自 Top-K**（例如 A 取 30、B 取 10、关键词取 10），**不在数据库层拼成一张大表**。
3. **权限硬过滤**：队 B 结果 **必须带 `user_id` 匹配**；**共享热词池下线**，改 **「仅会话内摘要」**；任何进 **对外回复** 的 Memory 片段 **不得含其他用户 ID**。
4. **合并与重排**：用 **Rerank（第 10 篇）** 或规则：**制度类问题里，来源标签为 `policy` 的 chunk **加权**；**便签类** 只在 **意图=续办工单** 时进决赛圈。

小陈强调：**「混合不是 **把两堆屎和两堆金搅一起**；是 **分厨配菜，最后一道菜再摆盘**。」**

---

### 四、合并错了，比单路更贵

灰度第二周，张先生续办对账：**「还是按 B 线，把停线三条带上。」** 这次 **队 B 命中便签指针**，**队 A 命中 MES 导出说明**——**合并后 Rerank 把「年假制度 2024 版」也塞进上下文**，因为张先生上周 **在聊天里提过「年假」**（吐槽加班），**便签向量把制度 chunk 拽进了同一邻域**。

圆桌座位（第 9 篇）被 **无关制度** 占掉一截，**停线三条的 Grounding（第 11 篇）** 反而被挤到截断线外；结构化输出 **字段齐**，**引用却错位**。

评测（第 10 篇）加了 **「混合检索」** 用例：**续办工单场景，Top5 必须含工单指针；福利场景，Top5 不得含他人或无关工单词。**

林薇在 PRD 写 **合并策略表**：

| 用户意图（产品定义） | 参与队列 | 合并规则（示例） | 禁止 |
|----------------------|----------|------------------|------|
| 公司制度/福利 | A + 关键词 C | 优先 `source=policy`；Rerank 输入带 **来源标签** | B 除非用户明确「结合我的情况」 |
| 续办个人工单 | A + B | B 的指针优先；A 补操作手册 | 跨用户 B |
| 纯闲聊 | 仅当前 session | 不查 B 档案柜 | 为「显得聪明」硬捞库房 |

**「混合检索的坑，** **常常是 **· 该并行的没并行（漏记忆）** **· 不该并行的瞎并行（串味、挤窗口）**。」**

---

### 五、接近结尾：原来柜台后面是分流器

灰度第四周，李女士再问年假：小言 **先引用制度库第 7 章 chunk**，**Grounding 链到 PDF 页码**；张先生续办对账：**便签指针 + 库房操作手册** 同屏出现，**年假条文不再插队**。

指标会上：

- **「答非所问」工单 ↓ 41%**（其中 **混检串味** 从每月 17 单 → 2 单）
- **单次检索延迟 ↑ 80ms**（多路召回 + Rerank）——林薇接受：**「用路由（第 19 篇）和 **缓存热制度 chunk** 换体验。」**
- **Token 均值 ↓ 6%**（少塞无关片段进圆桌）

复盘时，运营小夏问：**「所以用户还是一个输入框？」**

林薇笑：**「对用户是一个框；对系统是 **分流器 + 多柜台 + 安检（权限）+ 摆盘（合并）**。** 你昨天问的 **记忆和 RAG 混检**，就是这件事的学名。」**

小陈在文档里写下英文：**Hybrid Retrieval**——**多源、分路、再合一。** 张先生点头：**「难怪上次像 **在税务局排队，却听到隔壁窗口聊我家账本**。」**

## 概念正解

**Hybrid Retrieval / 混合检索**，指在一次回答用户之前，**从两种及以上不同来源或检索方式**（例如 **向量语义检索、关键词/BM25、结构化数据库、用户记忆索引、实时 API**）**分别召回候选材料**，再经过 **权限过滤、去重、融合打分或重排序**，选出最终塞进上下文（或交给模型）的片段集合。它解决的是：**真实产品里「该查什么」往往不是单一路径**——纯 RAG 会 **漏个人上下文**，纯记忆会 **答不了公司制度**，纯关键词又 **听不懂口语**；若 **简单把多路结果按一个相似度榜硬拼**，就会出现 **串权限、挤窗口、答非所问**。

回扣故事：**李女士问年假却看到 B 线口径** 对应 **多源结果未分权限、未分意图就合并**；**「统一检索分数高的赢」** 对应 **忽视来源边界与产品策略**；**分路 Top-K + 标签加权** 对应 **混合检索的标准形态**；**张先生场景塞进年假制度** 对应 **合并阶段未按意图抑制无关路**；**延迟与 Token 权衡** 对应 **混合检索的产品成本，需要路由与缓存**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 画 **检索架构图**：每路 **数据源、召回量 K、权限模型、与 Memory/RAG 的触发条件**；写 **意图→参与队列** 对照表 |
| **方案评审** | 问算法：**「合并发生在 Rerank 前还是后？跨路去重键是什么（doc_id/chunk_id）？」** 问工程：**「记忆索引与制度库是否物理分表？审计能否追溯『本轮用了哪几路』？」** |
| **指标设计** | 分路 **召回率@K、串味率、无关片段导致的 Token 浪费、P95 延迟**；按意图拆 **Grounding 失败归因** |
| **跨团队沟通** | 对法务：**「用户记忆永不与公共库 **同一排行榜无标签竞争**」**；对客服：**「答错可能是 **检索摆盘** 不是模型犯浑**」** |

## 类比锚定

**核心类比：机场安检与登机口——** **国内航班旅客、国际旅客、贵宾、货运** 都从 **同一个大门** 进航站楼（用户只打一个提问框），但 **安检通道、证件校验、登机口** 不同；**行李（检索片段）** 在 **分拣中心（合并/Rerank）** 按 **航班号（意图）** 装车，**不能把货运托盘塞进经济舱座位（上下文窗口）**。传统 PM 做 **多端/多角色入口** 时熟悉：**同一搜索 UI 背后往往是 **多索引、多过滤、再聚合结果**。

| 你熟悉的 PM 经验 | Hybrid Retrieval 要素 |
|------------------|-------------------------|
| 统一搜索框，背后 **商品库 + 店铺 + 评价** 多索引 | **多路召回**（RAG / Memory / 关键词 / 结构化） |
| **角色权限** 决定能看见哪些菜单与数据 | **分路权限过滤**（`user_id`、来源标签、敏感域） |
| 搜索结果 **综合排序**（销量、相关性、广告位） | **融合打分 / Rerank / 业务加权** |
| 「猜你想搜」与 **精确 SKU 码** 两套逻辑 | **语义向量 + 关键词** 互补 |
| 机票 **联程**：前一段误机影响后一段 | **一路噪声进合并** → **挤占上下文、带偏生成** |
| 运营 **大促缓存** 热词页，换延迟 | **热制度/热 FAQ 缓存**，换多路成本 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **混合检索是「找材料」层的架构**，不替代 Prompt、Memory 存储设计或 Rerank 模型本身，而是 **规定多源如何同时上场**。
2. **必须先分路再合并**：每路 **自己的 K 与权限**；合并阶段要带 **来源元数据**（政策/个人/手册），否则 **无法做产品策略加权**。
3. **意图决定参与队列**：不是每次请求都 **A+B+C 全开**；**过度并行** 涨延迟、涨 Token、引噪声。
4. **权限是硬约束，不是软分数**：他人记忆 **过滤掉**，不能指望 Rerank **「学会别选」**。
5. **要与 Grounding、窗口、成本联调**：无关片段进 Top5 的代价，往往比 **「少召回一路」** 更大。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像综合搜索，相关就行」 | **相关但无权、相关但无关意图** 的片段仍会害生成 | 需要 **意图门控 + 来源标签 + 否决规则** |
| 「多路一定比单路准」 | 路越多 **噪声、延迟、运维面** 越大 | 混合是 **按需组合**，要 **可观测、可降级（某路失败只走 A）** |
| 「Rerank 能摆平一切」 | Rerank **看不见权限** 时会把脏数据排前面 | **过滤在 Rerank 前**；Rerank 吃 **已消毒的候选集** |
| 「和 RAG 是一回事」 | RAG 常指 **单知识库管线**；Hybrid 是 **跨库跨索引的编排** | PRD 里写清 **「本需求是扩 RAG 还是上混合架构」** |
| 「记忆检索算 Hybrid 的一路」 | Memory 还有 **非向量注入**（档案字段直读） | 一路之内也可 **子混合**；产品图要 **落到表名与 API** |
