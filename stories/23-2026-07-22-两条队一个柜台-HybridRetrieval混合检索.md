## 寓言故事

### 一、问年假，却念出隔壁产线的账

记忆灰度第三周，李女士在飞书里点开 **内小言**——员工查制度、问福利的那个窗口。她没有对账工单，也不管产线，只打了一行字：

**「我们公司年假按自然年还是入职周年？有没有制度原文？」**

小言 **流式** 开头仍旧礼貌：**「根据检索到的材料……」** 下一句却陡然拐进另一条业务：

**「B 线 MES 与 PLC 以 `batch_id` 对齐，停线三条需备注导出批次……」**

李女士盯着屏幕。MES、PLC、`batch_id`——那是制造现场拿来对账的词，跟年假制度半毛钱关系没有。她截图丢进群：**「我问年假，它跟我讲产线对账？」**

林薇调出这一通请求的日志时，第一反应不是「模型又胡说了」。片段明明白白躺在 **检索结果** 里，还进了本次回答用的前几名。

群里的流程其实很日常：客服先建反馈单，值班产研和林薇一起看日志，把严重级别标成 **P0**——不是「回答略不准」，而是 **跨业务、疑似跨用户上下文泄漏**，用户侧叫答非所问，内部归因里先记一笔 **「串味」**。护栏负责人（第 18 篇里用「老关门卫」作比方的那套安全能力）被拉进会诊群，不是由他一个人派单定级，而是确认：**要不要先关掉热词入口、要不要按数据越权升级法务/安全值班。**

---

### 二、小夏的捷径：把「人人都问的口径」做成热词

复盘会上，运营小夏先举手。她知道那句「B 线口径」从哪来。

上周，张先生在 **修小言一侧的对账续办** 里，又和助手磨了一下午：B 线到底以哪个批次号对齐制造执行系统（MES）和产线控制器（PLC）日志，停线三条备注怎么写。谈妥之后，系统按第 22 篇的设计，给他本人留了 **记忆便签**——带用户 ID、带应用域，李女士账号本来 **读不到张先生抽屉里的那一张**。

可小夏看着工单后台发愁：同一租户里，好几个现场同事轮番来问 **几乎同一套口径**。每人谈一次，客服和助手就重复解释一次。她动了个「捷径」念头——

**别等每个人自己的便签被命中；把最近两周反复出现的结论，抽成短摘要，丢进一个「近期高频」池子，让检索更容易捞到。**

她管这池子叫 **热词库**：名字像运营活动里的热搜词，其实装的是 **从多人对话里摘出来的高频业务摘要**（例如「B 线口径」「停线三条备注规则」）。工程帮忙接得很快——把这些摘要 **向量化后，挂进记忆检索同一条入口**：用户一提问，系统除了搜公司制度库，也会在「记忆相关索引」里做向量检索（第 8 篇 Embedding：按语义远近找相似片段）。热词摘要既然进了索引，就会被当成普通可检索文本去比相似度。

捷径的代价，当时没人写进评审：

- 摘要 **跨用户汇总**，却 **没打「仅本人」**，也没锁 **「仅修小言 / 对账域」**；
- 内小言查制度时，记忆侧检索 **照样会扫到这口池子**；
- 向量检索并不懂「这是运营热词还是年假制度」，它只回答：**哪段文字和当前问题在向量空间里更近**。

李女士那句「年假……制度原文」，和池子里某些对账摘要并没有业务关系，但在一次「制度库五条 + 记忆侧五条、再按一个分数榜排序」的拼盘里，「B 线口径」仍挤进了 Top3。小言忠实地「根据检索到的材料」组织回答——于是年假窗口里，响起了隔壁产线的账。

小夏愣住：**「我以为热词是帮大家少解释两句……怎么帮到飞书问年假的人头上了？」**

小陈说得不留情：**「你把工单里的私货，做成了准公共知识，却没走公共库房的审核和权限。向量检索没有背叛你——是你把不该进公共通道的东西，塞进了它会扫到的货架。」**

---

### 三、两条队，不能挤一个柜台

工程小刘起初还想圆：**「用户哪分得清查制度还是查记忆？统一检索，分数高的赢，不就完了？」**

小陈在白板上画了两条队：

**队 A**，公司库房——制度 PDF、操作手册，租户里该看的人都能看，适合带出处引用。  
**队 B**，用户抽屉——便签、工单指针、个人偏好，默认仅本人，可过期、可清除。  
**队 C**，他后来补上：关键词和结构化字段——文号、「第 7 章」、入职日、司龄——补语义检索抓不住的钉子。

**「统一入口可以，统一排行榜不行。」** 小陈敲了敲「热词」那一栏，**「李女士问 HR，却排到修小言工单热词，不是幻觉，是排错队。热词若真是人人该知的口径，就该进队 A，走知识库上架；若仍是某人某单的结论，就该留在队 B，并锁死用户和应用。你现在这池子，两不像——所以最危险。」**

林薇没有当场甩术语。她只问了三个还没拍板的问题：用户一句话进来，**默认查哪些队？** 各队结果 **怎么合并？** **谁有资格看见哪一条**——看不见的，不能指望「分数低一点」，要 **进不了候选集**。

第一周他们上的不是「关掉记忆」，而是后来写进文档的名字：**Hybrid Retrieval（混合检索）**——分路召回，先过滤，再合并。问年假：意图落在制度，就开队 A 和关键词，**默认不开队 B**；张先生那种「还是按 B 线」才 A 与 B 并行。任何记忆侧片段必须对上当前登录用户；小夏那口 **跨用户热词池下线**，高频口径若要复用，改走制度/FAQ 库房上架。合并时带上来源标签，制度问题给政策类片段加权，再用 Rerank（第 10 篇）精排——小陈比方：**分厨配菜，最后一道菜再摆盘。**

---

### 四、人没串了，菜却又摆错一回

灰度第二周，张先生回来续办对账：**「还是按 B 线，把停线三条带上。」**

这一回，抽屉没有开错人：队 B 命中的是 **他自己的** 便签指针，队 A 也捞到了导出说明类手册。可合并之后，上下文里竟还塞进了大段 **「年假制度 2024 版」**——只因他上周吐槽加班时提过「年假」二字，向量觉得制度条文和便签「挨得近」，摆盘时又没按「续办对账」把无关制度按下去。

圆桌座位（第 9 篇）被占掉一截，停线三条的出处（第 11 篇 Grounding）反而挤到截断线外。字段填得整齐，引用却对不上本轮工单。

林薇在复盘里写得很白：**混合检索的坑是两头——该并行的没并行，会失忆；不该并行的瞎并行，会串味、会挤窗口。热词池是第一种事故的加速器；合并太松，是第二种。** 评测补了用例：福利场景 Top 结果不得出现他人记忆和对账热词；续办场景必须保住工单指针，不得因聊天提过「年假」就让制度大段插队。

---

### 五、接近结尾：柜台后面是分流器

第四周，李女士再问年假。小言引用制度库第 7 章，链接到 PDF 页码；对账字样再也没有从飞书制度窗口冒出来。张先生续办时，本人便签与操作手册同屏出现，年假条文安分地留在库房里，不再抢座位。

指标会上，答非所问工单下来一截，其中「混检串味」从每月十几单收到两单；检索慢了几十毫秒，Token 却因少塞无关片段略降。小夏问得很小声：**「所以……热词还能做吗？」**

林薇点头，又摇头：**「能做『大家该知道的高频结论』，那叫知识库运营，走队 A，要审核、要权限、要版本。不能做的是：把工单私货向量化后挂进谁都能扫到的货架，还指望相似度自己懂礼貌。」**

运营又问：**「用户不还是一个输入框？」**

**「对用户是一个框。」** 林薇说，**「对系统是分流器、多柜台、安检和摆盘。记忆和库房要一起上场时——多源、分路、再合一——就是 Hybrid Retrieval。」**

张先生听完，只甩下一句：**「上次就像在一个窗口办业务，却听见隔壁在念我家账本。」** 会议室里安静了一秒。小陈把英文写进标题栏时，没人再觉得那是空名词。

## 概念正解

**Hybrid Retrieval / 混合检索**，指在一次回答用户之前，**从两种及以上不同来源或检索方式**（例如向量语义检索、关键词、结构化库、用户记忆索引）**分别召回**，再经 **权限过滤、去重、融合打分或重排序**，选出最终进上下文的片段。它解决的是：真实产品里「该查什么」很少只有一条路；若把多路结果 **糊成一个相似度榜**，就会串权限、挤窗口、答非所问。

回扣故事：李女士问年假却念出 B 线，是 **制度库与记忆侧（含运营热词池）同榜竞争且未做应用/用户隔离**；小夏的热词库说明 **向量检索会搜到「你放进索引的任何文本」——热词不是向量的本能，是人把跨用户摘要挂进了可检索货架**；分路、过滤、再合并是混合检索的正形；张先生续办时年假插队，是 **隔离对了、合并策略仍松**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写清入口（如内小言）、问法原话、每路数据源与触发条件；高频复用走知识库还是记忆，单独成节 |
| **方案评审** | 问：合并在 Rerank 前还是后？跨用户摘要能否进记忆索引？日志能否按 trace 列出各路命中？ |
| **指标设计** | 串味率、无关片段占比、分意图 Grounding 失败、P95 检索延迟 |
| **跨团队沟通** | 对运营：**「热词复用 ≠ 挂进记忆向量库」**；对客服：**「先查检索摆盘，再查模型」** |

## 类比锚定

**核心类比：机场同一大门、多条安检与登机口。** 用户只打一个提问框；行李（片段）必须按航班（意图）分拣，不能把货运塞进经济舱座位（上下文窗口）。

| 你熟悉的 PM 经验 | Hybrid Retrieval 要素 |
|------------------|-------------------------|
| 统一搜索框背后 **商品库 + 店铺 + 评价** 多索引 | **多路召回**（RAG / Memory / 关键词等） |
| 角色权限决定菜单与数据可见范围 | **分路权限过滤**（用户、应用域、来源标签） |
| 综合排序（相关性、业务加权） | **融合打分 / Rerank** |
| 运营把爆款做成正式专题页，而不是塞进某人草稿箱还对全站开放 | **高频口径应上架公共知识库，而非跨用户热词挂记忆索引** |
| 「猜你想搜」与精确 SKU 两套逻辑 | **语义向量 + 关键词** 互补 |

## 类比边界说明

### 一、概念核心要点

1. **混合检索管的是「找材料怎么编排」**，不是替代 Prompt 或 Memory 产品设计本身。  
2. **先分路再合并**；合并必须带上来源与应用域元数据。  
3. **意图决定开哪些路**，不是每次 A+B+C 全开。  
4. **权限是硬过滤**：不匹配的片段进不了候选集；隔离是结果，过滤是手段。  
5. **运营热词 / 高频摘要**：若面向多人复用，应走 **公共知识库运营**；若仍是个人或工单结论，必须留在 **带权限的记忆**——二者都不是「往向量索引里一丢了事」。

### 二、类比在哪里会失效

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「相关就该展示」 | 相关但无权、相关但错意图仍会害生成 | 意图门控 + 来源否决 |
| 「多路一定更准」 | 路多则噪声与延迟上升 | 按需组合，可降级 |
| 「Rerank 能收拾热词」 | 脏数据进了候选集就晚了 | **过滤在 Rerank 前** |
| 「热词检索是混合检索的标配」 | 故事里的热词池是 **反例** | 混合检索要的是分路与权限，不是跨用户私货同榜 |
