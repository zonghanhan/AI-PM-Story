## 寓言故事

### 一、截图里那条「第七条」

Token 整改上线后的第五天，林薇被法务拉进紧急会议室。

屏幕上是一张用户截图。小言在对话里写：

> 「根据我司《2024 版数码产品售后细则》**第七条第二款**，您购买的降噪耳机在 **签收后 15 日内**，即使已拆封、已激活，也可申请 **无理由全额退款**，无需承担折旧费。请提供订单号，我为您发起流程。」

用户下面跟了一句：「你们自己机器人说的，现在客服告诉我没这条？我已经把耳机寄回去了。」

法务同事语气很平，但每个字都沉：「寄回地址是小言给的 **正确地址**，但 **退款依据是错的**。如果按这条『细则』赔，我们本月同类工单已经开了 37 单。」

林薇第一反应和上次一样：「是不是上下文被截断了？用户前面说过什么，后面没看到？」

她调出会话日志——**没有**。整段对话只有 4 轮，Token 占用不到窗口的 5%。系统提示、RAG 检索结果、订单查询返回，**全在桌上**。

第二反应：「是不是 RAG 检索错了，喂了过期文档？」

老周已经打开了知识库后台。检索记录显示：小言确实调了 `search_policy`，命中的三篇文档分别是《数码产品售后总则》《耳机品类换货指引》《拆封商品退款说明》——**没有一篇叫《2024 版数码产品售后细则》**。

林薇盯着那行「第七条第二款」，后背有点凉：「那它……从哪编出来的？」

---

### 二、越查越像真的

下午，林薇拉着合规、客服、研发做了「复盘三连」。

**复盘 A：文档里有没有类似表述？**

合规同事把三篇命中文章逐条对读。结论：

- 「15 日」——**总则里写的是 7 日**；
- 「拆封可退」——**耳机指引里写的是拆封仅换货、不退款**；
- 「第七条第二款」——**三篇文档都没有分「条」「款」这种结构**。

但三篇里 **各有一句话**，拼在一起，**特别像** 小言那段回复的「原料」。

**复盘 B：模型有没有调用工具捏造返回？**

研发查了 Tool 日志。`query_order` 返回正常；`search_policy` 返回了三段真实摘要，附带来源 ID。**没有任何一条 JSON 里出现「第七条」或「15 日无理由」**。

**复盘 C：是不是 Prompt 逼它「必须给明确答案」？**

林薇翻出系统提示里的一句话，是两周前改的：

> 「你是专业售后顾问，**必须给出明确、可执行的政策依据**，避免让用户觉得你在推诿。」

客服主管叹气：「当时投诉模型老说『建议您联系人工』，我们就让它 **别打太极**……」

老周在白板上画了一条线，左边写「用户要确定感」，右边写「知识库没有确定条款」。两条线在中间撞在一起，他圈了个红圈：「**空档**。」

「小言最擅长的事，」老周说，「就是把 **看起来像真的碎片**，**补成一段完整、顺口、带条款编号的话**——因为它从训练第一天起，学的就不是『查准』，是 **『接下去最像人话的那一句』**。」

---

### 三、它不是撒谎，是在「补全」

林薇还是没完全听懂：「材料都喂了，为什么不用材料？」

老周让她做了一个现场实验。他在白板上写：

```
知识库原文 A：数码类商品签收 7 日内可申请退货。
知识库原文 B：耳机拆封后不支持退款，仅支持换货。
用户问：我耳机拆封了，签收第 10 天，能退款吗？
```

「你来扮演模型，只能回答一句，而且 **不能写『建议联系人工』**，你怎么答？」

林薇犹豫：「只能说……不符合退款条件？」

「对。但你们 Prompt 里还有一条——**引用政策条款，让用户信你**。」老周把笔递给她，「你再补一句条款。」

林薇硬着头皮写：「根据售后说明，拆封商品不予退款。」

「条款第几条？」

「……库里没写第几条。」

「那你会不会 **自己编一个听起来像真的编号**？比如『第三条』——反正用户也不会立刻去翻 80 页 PDF。」

林薇愣住。老周补了一句更扎心的：「这不是小言『坏』，是它 **根本不知道自己在编造**。它生成每一个字的时候，只是在算 **『接在这个上下文后面，哪个词概率最高』**——**『第七条第二款』** 在语言模式里，比 **『我不知道』** 更像一个「专业顾问该说的话」。」

合规同事接话：「所以用户截图才会 **那么可信**。语气笃定、结构完整、还有书名号——**比真客服还像真客服**。」

---

### 四、越像真的，越危险

接下来 48 小时，林薇又收集到几类「查无此事」：

| 案例 | 小言说了什么 | 实际情况 |
|------|--------------|----------|
| 案例 1 | 「您的包裹预计 **明日下午 3 点** 送达」 | 物流工具只返回了「派送中」，**没有具体时点** |
| 案例 2 | 「为您查到 **同类案例判赔 380 元**」 | 案例库检索为空，**数字是模型填的** |
| 案例 3 | 「已为您创建工单 **WO-20240703-9912**」 | Tool 实际返回 ID 是 **WO-20240703-9910**，末两位错了 |
| 案例 4 | 「该功能在 **v3.2 版本** 将上线」 | 路线图知识库 **无此版本规划** |

林薇发现一条规律：**幻觉最爱出现在「用户最需要一句准话」的地方**——具体时点、具体金额、具体条款编号、具体工单号、具体版本号。

反过来，**模糊废话** 很少被投诉：「我们会尽快处理」「请您耐心等待」——因为 **没法证伪**。

周五站会上，老板问：「RAG 不是解决胡说的吗？Tool 不是接真数据吗？」

林薇摇头：「RAG 解决的是 **『没喂材料』**——库里有的，检索到了，它 **仍可能改头换面**。Tool 解决的是 **『能查到真值』**——查到了，它 **仍可能写错字段、多填一格**。Token 截断是 **没看见**；幻觉是 **看见了（或本该去看见），但说出来走了样**。」

老周补充：「而且 **温度（temperature）** 高一点、Prompt 逼 **『自信』**、业务逼 **『不许转人工』**——都是在给补全引擎加油门。」

---

### 五、不信，但不废

第二周，林薇没有喊「关掉小言」，而是带着大家改了一版 **「防言之凿凿」清单**：

1. **高风险字段禁止模型「自由生成」**：订单号、工单号、退款金额、政策条款号、送达时点——**必须来自 Tool 或 RAG 的结构化字段**；模型只负责 **组织语言**，不负责 **造数字**。
2. **引用必须带「可点击出处」**：回复里出现政策名，必须挂 **知识库片段 ID 或原文链接**；用户一点发现对不上，**责任链清晰**。
3. **不知道就说不知道，而且要设计得「体面」**：把「避免推诿」改成「**无依据时不承诺**；给出 **可查路径**（链接、人工入口、预计响应时间）」——**确定感的来源从「编一句」改成「给下一步」**。
4. **关键动作二次校验**：创建工单、发起退款、发送地址——**Tool 返回什么，界面展示什么**；模型文本与结构化结果 **不一致则拦截**，不让用户只看到漂亮句子。
5. **指标从「像不像人」改成「能不能证」**：加 **政策引用准确率、字段篡改率、无依据承诺率**；抽检不是看语气，是 **拿回复逐字段对库**。

上线一周后，37 单误赔工单没再新增。但「建议联系人工」占比上升了 6%。

客服主管有点不爽：「用户又说我们踢皮球。」

林薇把法务那封「感谢及时止损」的邮件转给他：「**踢皮球有时比送错球门好**。」

散会后，林薇独自看回放。用户问拆封耳机能不能退，小言新版本答：

> 「查过当前售后政策：**拆封耳机不支持退款，仅支持换货**。[查看原文片段 #policy-earphone-02] 若您的情况特殊，可点此提交人工复核，**2 小时内** 有专员回复。」

没有「第七条」，没有「15 日」，没有「全额」。

她忽然明白：这一周他们对抗的，从来不是「让模型更聪明」，而是 **承认它会_completion（补全）_，然后在产品层 **给补全划边界**——什么话可以组织，什么话 **必须来自证据**。

## 概念正解

**幻觉（Hallucination）** 指大语言模型 **生成看似合理、语气笃定，但与事实不符或无法核实的内容**——包括捏造政策条款、错误数字、假引用、编造工具结果等。它不是「故意撒谎」，而是模型按 **「最像正确回答的语言模式」** 做概率补全时的系统性偏差；**RAG、Tool Calling、大上下文窗口** 都能降低发生率，但 **不能消灭**，因为「读到了」不等于「原样说出」，「能查到」也不等于「不会写错字段」。

回扣故事：

- **「第七条第二款」** → **事实性幻觉**：把多篇真实文档碎片 **拼成不存在的条款结构**；
- **物流「明日下午 3 点」** → **过度推断幻觉**：在 **未给出的数据上** 补全具体细节；
- **工单号末两位错误** → **参数/字段幻觉**：Tool 已返回真值，模型 **改写或误记**；
- **防言之凿凿清单** → 产品侧 **接地（Grounding）、引用溯源、结构化输出校验、承认未知** 等护栏。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 划分 **高风险字段清单**（金额、日期、条款号、ID）；写明 **必须 Tool/RAG 直出、模型仅润色** 的边界；定义 **无依据时的标准降级话术** |
| **方案评审** | 问研发：「回复里的数字和 ID，是 **解析结构化结果** 还是 **解析自然语言**？不一致怎么办？引用能否 **追溯到片段 ID**？」 |
| **指标设计** | 除满意度外，加 **政策引用准确率、字段一致性率、无来源断言占比、幻觉导致的工单/赔付率**；抽检用 **字段级对库**，不只读语气 |
| **跨团队沟通** | 跟法务说：「**语气笃定 ≠ 有法律依据**」；跟业务说：「不许转人工会 **抬高幻觉代价**」；跟算法说：「降低温度只能缓解，**护栏要产品化**」 |

## 类比锚定

**核心类比：PM 审外包/供应商交来的「调研报告」——排版专业、数据齐全、结论笃定，但你要逐条核对原始出处**

传统 PM 都经历过：乙方 PPT 写得漂亮，**图表、案例、市场数字** 一应俱全；你一追问「这个份额哪来的」「这个客户名能不能公开」，对方才承认 **参考了旧报告、猜了一个数、把两个来源拼在一起了**。你不会因为 PPT 好看就直签，也不会因为发现一处造假就永远不用外包——你会建 **引用规范、数据来源字段、关键数字双人复核**。幻觉就是 AI 场景里 **「没经过核对的乙方结论」**，而且 **交稿速度是毫秒级**。

| 你熟悉的 PM 经验 | Hallucination / 护栏要素 |
|------------------|--------------------------|
| 乙方报告里的「据行业数据显示」但没有脚注 | 模型的 **无来源断言**；听起来权威，**无法溯源** |
| 把三份过期材料拼成一条「最新政策」 | **碎片拼接型幻觉**；每段似真，**组合即假** |
| 合同金额写错一位数，格式却完全正确 | **字段篡改型幻觉**；Tool 返回真值，**自然语言写错** |
| 要求关键数字必须附原始链接或附件页码 | **引用溯源（Citation）**；回复绑定 **知识库片段 ID** |
| 「暂无数据」比「编一个数填坑」安全 | **承认未知（Abstention）** 的产品设计；用 **下一步路径** 换确定感 |
| 对外发布前法务逐条核对可公开表述 | **高风险场景人工复核 / 结构化校验**；模型文本与 **JSON 字段一致** 才可发送 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **幻觉是「像真的假内容」**，根因是语言模型的 **补全机制**，不是道德问题；**自信语气与正确性无必然关系**。
2. **RAG 减少「空口胡说」**，但不保证 **忠实复述**；检索到的片段仍可能被 **改写、合并、加戏**。
3. **Tool Calling 提供真值通道**，但模型可能 **填错参数、看错字段、捏造未返回的细节**；要有 **结构化输出与文本一致性校验**。
4. **Prompt 逼「明确、自信、不许推诿」** 会 **抬高幻觉率**；产品要在 **确定感** 与 **可证伪性** 之间做显式权衡。
5. **治理靠流程与指标，不靠「换个更聪明的模型」**；高风险域必须 **默认不信自然语言里的关键事实**。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像乙方造假，所以是恶意」 | 模型 **没有欺骗意图**，也不会「心虚」；它 **同样会对真材料产生错误补全** | 用 **机制与护栏** 治理，不要用「加强道德约束」式 Prompt 代替工程校验 |
| 「核对出处就能杜绝」 | 模型可 **伪造出处格式**（假链接、假条款号），或 **张冠李戴** 把 A 文档观点安到 B 文档 | 出处要是 **系统生成的可解析 ID**，用户可点、后台可验，不是模型随手写的书名号 |
| 「没数据就说暂无，很简单」 | 业务 KPI 常惩罚「转人工率」「无法回答率」，团队会 **暗中加 Prompt 逼答** | 要在 OKR 层 **允许「不知道」**；否则产品口头上 abstain，线上 **逼出幻觉** |
| 「人工复核万能」 | 全量复核 **成本爆炸**；人也会 **被笃定语气带偏**（自动化偏见） | 应用 **风险分级**：仅对 **金额/合规/医疗** 等场景上 HITL；低风险用 **抽检 + 结构化校验** |
| 「温度调 0 就没有幻觉」 | 低温减少 **发散编造**，不消除 **事实错误、字段误读、错误拼接** | 温度是 **油门**；**接地、引用、字段绑定** 才是 **护栏** |

### 动手小练习（约 15 分钟）

1. 选一个你们 **真实知识库** 里的模糊问题（如「拆封能否退款」），用同一 Prompt 问常用 Chat 产品 **5 次**——看 **条款编号、天数、金额** 是否每次一致；记录 **「笃定但不一致」** 的次数。
2. 画一张 **「高风险字段表」**：列出你们业务里 **不能由模型自由生成的 10 个字段**，并写清 **来源（Tool / RAG / 人工）**——可直接贴进 PRD。
3. 找一条历史客诉里 **AI 说错话** 的截图，做 **「字段级拆解」**：哪句是事实、哪句是推断、哪句 **库里根本没有**——练 **抽检话术**。

### 幻觉产品 Checklist（可保存到笔记）

```markdown
## 边界定义
- [ ] 高风险字段清单（金额、日期、ID、条款号、版本号……）
- [ ] 哪些场景允许模型组织语言，哪些必须结构化直出
- [ ] 无依据时的标准降级话术（含人工入口 / 查询路径）

## 接地与溯源
- [ ] RAG 回复是否绑定可点击片段 ID / 原文链接
- [ ] Tool 返回与面向用户的自然语言是否做一致性校验
- [ ] 是否禁止模型自行构造「看起来专业」的条款编号

## 指标与抽检
- [ ] 政策引用准确率、字段篡改率、无来源断言占比
- [ ] 幻觉导致的赔付 / 投诉 / 合规事件率
- [ ] 抽检方法：字段级对库，而非只看语气

## 评审必问
- [ ] Prompt 是否在逼「必须给明确答案」从而抬高幻觉？
- [ ] 「上了 RAG」是否被误当成「不会胡说」？
- [ ] 业务 KPI 是否惩罚「不知道」，变相奖励编造？
```
