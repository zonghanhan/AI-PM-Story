## 寓言故事

### 一、用户明明说过「先不换」

Tool Calling 整改上线后的第三天，林薇又收到一条客诉。

用户原话很简单：「我上周买的耳机左耳没声，想换货。订单号 20240618-8821。」

小言先查了订单，又查了退货政策，还调了 `query_logistics` 把 **一整段物流轨迹** 拉回来——十几条节点，从「已揽收」到「派送中」逐条列出。

对话进行到第五轮，用户有点急了：「我说了 **先不换货**，你们怎么给我发了换货面单？」

林薇调出会话回放，从头看：

- **第 1 轮**，用户确实说「想换货」；
- **第 3 轮**，用户说「我先看看政策，**暂时不换**」；
- **第 6 轮**，小言却调了 `create_exchange_ticket`，还发了短信。

更诡异的是，第 6 轮小言的回复里，语气像 **完全没听过**「暂时不换」这句话：「已为您创建换货工单，请按面单寄回。」

林薇第一反应还是老问题：「是不是又选错工具了？」

老周摇头：「工具这次选对了。你看日志——**第 6 轮送进模型的上下文里，根本没有第 3 轮那句『暂时不换』**。」

---

### 二、桌上只能摞这么厚

老周在白板上画了一张小桌子，上面摞了一摞纸。

「你把小言想成一个 **只能看眼前这一摞纸** 的值班员。每来一轮对话，我们就要把东西往桌上放：

- 系统提示词（人设、规则）——**固定占几页**；
- 用户和模型的历史对话——**一页一页往上叠**；
- 每次 Tool 返回的 JSON——**往往比聊天还厚**；
- 模型这一轮要生成的新回复——**也得先留空位**。」

林薇盯着那摞纸：「所以……桌子满了，最早的纸会被抽走？」

「对。工程上叫 **截断（truncation）**。你们现在的策略是：**超了就从最老的对话开始扔**。第 3 轮那句『暂时不换』，在第 6 轮已经被 **`query_logistics` 返回的十几条物流节点** 挤出去了。」

林薇倒吸一口气：「所以不是小言『故意无视用户』，是它 **字面意义上没看到**？」

「可以这么理解。更麻烦的是——」老周在纸边写了个小字，「**不是按『第几轮』算厚度，是按『字数块』算**。」

---

### 三、不是「一个字」，是「一块一块切」

财务小陈这时敲门进来，手里拿着一张供应商账单，脸色不太好看。

「林薇，小言上个月 **Token 费用环比涨了 2.4 倍**。老板问是不是有人刷接口。」

林薇愣了：「我们日活没涨那么多啊。」

小陈把账单投影到墙上。上面没有「发了多少条消息」，只有几列她不太熟的词：

| 项目 | 上月 | 本月 |
|------|------|------|
| Input Tokens | 820 万 | 2,310 万 |
| Output Tokens | 190 万 | 540 万 |
| **合计计费 Token** | **1,010 万** | **2,850 万** |

「Input 是什么？Output 又是什么？」林薇问。

老周接话：「你可以把 **Token** 理解成模型读写时的 **最小计费块**——不是你们后台日志里的『一个汉字』，也不是 Word 里的『一个字』。英文常按词根切，中文往往 **一个汉字就可能是一块**，标点、空格、JSON 里的 `{` `}` 也都算。」

他在白板上写了几句：

```
用户说：「订单 20240618-8821 想换货」     → 大约十几到二十几块
工具返回：{"nodes":[{"time":"...","status":"..."}, ...]}  → 可能几百块
模型回复：「已为您查询，该订单支持换货……」  → 又是几十块
```

「**每一轮都是 Input + Output 分开计**。Input 是你 **塞进桌子的所有纸**——系统提示、历史、工具结果、用户新消息。Output 是模型 **新写上去的那几页**。Agent 跑一圈，查三次订单、拉一次物流，**Input 会叠三遍工具返回**；用户只觉得『问了一句』，账单里可能是 **几千块**。」

小陈补刀：「而且 **Output 通常比 Input 贵一截**。你们让小言写长回复、写逐步推理，Output 涨得最快。」

林薇忽然想起上周的需求评审：有人提议「每次工具调用都把 **完整原始 JSON** 回传给模型，方便它理解」——当时大家只讨论了「准不准」，没人算过 **厚度**。

---

### 四、装不下，就会丢；丢不起，就要裁

接下来一周，林薇拉着老周和小陈做了三次「称纸」实验。

**实验 A：长会话客服**

让一个测试账号连续聊 40 轮，中间穿插 8 次 `query_order` 和 3 次 `query_logistics`。

结果：到第 32 轮，**最早 6 轮用户确认过的关键信息** 已从上下文消失；模型开始 **重复追问** 已经答过的订单号。

**实验 B：工具返回「全量 vs 摘要」**

同一次物流查询，两种回传策略：

| 策略 | 送进模型的 Token 量（约） | 模型能否答对「包裹到哪了」 |
|------|--------------------------|---------------------------|
| 全量 JSON（17 个节点 + 原始字段） | ~1,200 | 能 |
| 摘要（最新状态 + 预计送达 + 异常标记） | ~80 | 能 |

「全量」贵了 **15 倍**，体验几乎没差——除非用户问「每一站分别几点到的」，那种 **细枝末节** 才需要全量。

**实验 C：系统提示词减肥**

他们把售后 Agent 的系统提示从 3,200 Token 压到 1,100 Token，删掉重复示例，把细则挪进 **RAG 知识库按需检索**。

任务完成率几乎不变，但 **每轮 Input 固定成本** 降了三分之一。

林薇在笔记本上写下两行字：

> **桌子有上限（上下文窗口）→ 满了就丢旧的 → 丢错了就是产品事故。**  
> **纸按块收费（Token）→ 塞什么、塞多满 → 直接决定账单和延迟。**

---

### 五、产品会上，林薇换了一种问法

周五评审，有人提：「能不能把上下文窗口开到最大？128K 够不够？」

林薇没直接答「够」，而是把白板翻过来：

**「我们先别聊 K 是多大，先聊桌上要放什么。」**

她列了四条产品原则，全场安静了几秒：

1. **关键承诺必须「钉」在桌上**：用户说过的「暂不办理」「不要退款」「已确认地址」——不能被工具日志挤掉；需要 **摘要条、状态字段或结构化记忆**，不能全靠原始聊天记录硬撑。
2. **工具返回默认走「摘要层」**：全量 JSON 落日志和数据库，给模型看的只留 **决策所需字段**；用户追问细节再二次查询。
3. **长会话要有「合页」机制**：每 N 轮或每 M Token，把已确认信息 **滚动摘要** 成一小段，替代几十轮原文。
4. **成本指标写进 PRD**：除「回答准确率」外，加 **单会话 Input/Output Token 分布、工具回传 Token 占比、P95 会话总 Token**——否则财务永远事后 surprise。

老周最后补了一句：「128K 不是免费变大桌子，是 **允许你更懒地往桌上堆纸**——堆满了照样丢，只是晚几轮丢；**懒堆 + 不裁剪工具返回**，账单会先爆。」

散会后，小陈发微信：「这个月 Token 涨幅已经刹住了。你们裁的不是能力，是 **废话厚度**。」

林薇回了一个字：「称。」

她这时才隐约反应过来：这一周他们争论的，从来不是「模型够不够聪明」，而是 **桌上那摞纸怎么摞、按块怎么收钱**——这件事，产品比算法更该先画线。

## 概念正解

**Token（令牌 / 词元）** 是大语言模型处理文本时的 **最小计量单位**：模型每次「读」和「写」，都先把内容切成 Token 再计算；供应商通常按 **Input Token（读入）** 和 **Output Token（写出）** 分别计费。所有送进模型的内容——系统提示、历史对话、RAG 检索结果、Tool 返回——都会占用 **上下文窗口（Context Window）** 这块「有限桌面」；超出上限时，系统必须 **截断、摘要或丢弃** 较早内容，模型就会 **字面意义上「失忆」**。

回扣故事：

- **桌上那摞纸** → 上下文窗口里当前可见的全部 Token；
- **`query_logistics` 全量 JSON 挤掉「暂时不换」** → 长工具返回占满窗口导致 **早期关键用户意图被截断**；
- **财务账单上的 Input / Output** → Token 计费结构；Agent 多轮调工具会 **重复堆叠 Input**；
- **摘要层、滚动摘要、关键承诺钉住** → 产品侧的 **上下文治理**，不是纯工程优化。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写清 **单轮 Input 组成**（系统提示约 X Token、历史保留 N 轮、工具返回默认摘要字段）；定义 **不可被截断的用户承诺字段** |
| **方案评审** | 问研发：「上下文满了 **丢哪一段**？工具返回是 **全量还是摘要**？128K 窗口下 **P95 会话会不会顶满**？」 |
| **指标设计** | 除准确率外，加 **单会话总 Token、工具返回 Token 占比、因截断导致的重复追问率、单用户月度 Token 成本** |
| **跨团队沟通** | 跟财务说清：**不是按「消息条数」计费**；跟算法说清：**省 Token 不是砍功能，是改「给模型看什么」**；跟业务说清：**每多塞一段全量日志，都是在花「记忆税」** |

## 类比锚定

**核心类比：PM 组织一场跨部门评审会，会议室白板只有一块，会议按「页数」向行政部结账**

传统 PM 都经历过：评审会要摆 **议程、背景材料、各方发言记录、现场结论**——但白板就那么大，早发言的会被擦掉；行政部按 **复印页数 + 会后纪要页数** 结算耗材。Token 就是 AI 场景里 **「页数」与「白板面积」的统一度量**。

| 你熟悉的 PM 经验 | Token / 上下文要素 |
|------------------|-------------------|
| 会议室白板面积有限，写满要擦最早的内容 | **上下文窗口上限**（如 32K / 128K Token） |
| 复印纸按页计费，页数越多行政成本越高 | **Input / Output Token 分开计费** |
| 会前发 80 页背景包，大家根本看不完 | **系统提示 + 全量工具 JSON 过长**，占满「桌面」却低效 |
| 会议助理做的「一页纸摘要」贴在白板角落 | **滚动摘要 / 结构化记忆**，用少量 Token 保留关键结论 |
| 「已确认需求」必须写在白板固定区，不许被擦掉 | **关键用户意图字段化**，防截断误删 |
| 只邀请必要参会人，避免人多嘴杂 | **少而准的上下文**：摘要工具返回、按需 RAG，而非全量堆叠 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Token 是计费与处理的最小单位**，不等于「一个字」；中文、英文、标点、代码的切分方式不同，**不能凭直觉估算**。
2. **Input = 模型本轮「读到的一切」**，Output = 模型本轮「新写出来的」；Agent 每多调一次工具，Input 往往 **再叠一层**。
3. **上下文窗口是硬上限**；超出后必丢内容，表现为 **遗忘、重复追问、违背早前承诺**——这是产品事故，不是「模型变笨」。
4. **窗口越大 ≠ 可以不做裁剪**；大窗口缓解的是「晚几轮才满」，**成本和延迟仍随 Token 线性上涨**。
5. **上下文治理是产品职责**：哪些必留、哪些摘要、哪些落库不送进模型——决定 **体验、成本、合规** 三角的平衡。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像白板擦了就没了」 | 工程上可用 **外部记忆库、向量检索、会话状态机** 把信息挪出窗口再 **按需捞回** | 遗忘不必然发生，但 **需要产品设计「第二存储」**；不是窗口越大越省心 |
| 「页数越多越贵，所以越短越好」 | 过短会 **丢必要约束**，导致 Tool 误调、合规缺失；省 Token 不能以 **砍掉安全规则** 为代价 | 目标是 **高信噪比**，不是无脑缩短；关键规则应 **结构化常驻** |
| 「一页纸摘要就够」 | 摘要本身会 **丢细节**，且摘要模型也会犯错；用户追问细节时要能 **二次拉全量** | 摘要 + 可回溯全量日志是 **双层设计**，不是二选一 |
| 「行政按页结账，所以能心算成本」 | Token 与字符 **无固定 1:1**；不同模型、不同语言切分不同，要 **用 Tokenizer 或平台估算器** | PRD 里应留 **Token 预算区间**，不要写「不超过 500 字」代替 Token 上限 |
| 「会议室越大越好」 | 极大窗口下，模型对 **中间段落** 的注意力可能下降（「Lost in the middle」类问题） | 产品仍要做 **信息排序**：关键约束放前/放后、用结构化字段 **钉住** |

### 动手小练习（约 15 分钟）

1. 打开 [OpenAI Tokenizer](https://platform.openai.com/tokenizer)，分别粘贴：**一句中文客诉**、**一段 200 行的模拟 `query_logistics` JSON**、**你的售后系统提示词**——记录三者 Token 数，算 **比例差**。
2. 在常用 Chat 产品里发起一个 **10 轮以上** 的对话，第 5 轮明确说一个 **独特约束**（如「不要发邮件，只发短信」），第 10 轮问「我上次说的联系方式偏好是什么」——看是否 **被遗忘**（粗测窗口/截断策略）。
3. 用表格写一版 **「单会话 Token 预算」** 草稿：系统提示 ≤？、历史 ≤？、单次工具返回 ≤？、预留 Output ≤？——可直接贴进下一份 Agent PRD。

### Token 产品 Checklist（可保存到笔记）

```markdown
## 上下文组成
- [ ] 系统提示 Token 数（目标上限）
- [ ] 历史保留策略：轮数 / Token 数 / 滚动摘要
- [ ] 不可截断字段：用户确认态、合规承诺、订单号等

## 工具与 RAG
- [ ] 工具返回默认摘要字段清单；全量落库但不进模型
- [ ] RAG 单次检索片段 Token 上限
- [ ] 用户追问细节时的「二次展开」路径

## 成本与体验
- [ ] 单会话 P50 / P95 总 Token
- [ ] Input vs Output 占比；Output 是否过长可裁剪
- [ ] 因截断导致的重复追问率 / 承诺违背率

## 评审必问
- [ ] 上下文满了丢哪一段？谁决定？
- [ ] 更大窗口是缓解问题还是掩盖「不裁剪」？
- [ ] 财务模型是否按 Token 而非 DAU 估算？
```

### 下一篇预告

Token 解决了「桌上能放多少、按块怎么收钱」——但模型有时 **明明看着材料，仍把错话当真话讲**。下一篇将拆解：**幻觉（Hallucination）**：它从哪来、和 RAG / Tool 有什么关系、产品怎么用 **流程与指标** 管住「一本正经胡说八道」。
