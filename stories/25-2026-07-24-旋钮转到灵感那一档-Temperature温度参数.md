## 寓言故事

### 一、同一剧本，十种「亲」

上岗剧本分层上线后的第一周，林薇在配置台给 **商小言** 做回归：同一商家租户、同一句用户原话、同一份退货制度检索结果，她连点五次「发送测试」。

用户句还是老样子：

**「这件衣服洗了一次就缩水，能退吗？」**

五次回复里，制度要点都对——七天无理由、洗后影响二次销售一般不退、可协商换码——但 **措辞像十个实习生各写一版**：

第一次偏短，像模板；第二次突然多了 **「亲，特别理解您的心情」**；第三次把 **「一般不退」** 说成 **「原则上不支持退货哦」**；第四次居然多了一句 **「要不给您申请一张 5 元券？」**——剧本里明明写了 **不得擅自承诺补偿**；第五次又板着脸，像念制度条文，商家运营在旁吐槽：**「这版能拿去应付工商吗？太硬了。」**

林薇截图发给小陈：**「剧本没动、检索日志也一致，怎么同一句问五遍，五个味儿？」**

小陈没先讲名词，只发来配置台截图。在「应用上岗剧本」下面，多了一行她以前没细看的控件：

**温度（Temperature）：0.9**  
旁边小字：**数值越高，用词越发散；越低，越倾向重复最常见说法。**

**「剧本管『你是谁、什么不能说』；温度管『同样合规时，敢不敢换种说法、敢不敢多走半步』。」** 小陈说，**「你现在相当于让同一个客服，每次掷骰子决定语气松紧。」**

---

### 二、阿梅把「灵感档」开到了 FAQ

问题出在周五晚。

大促前，运营阿梅要在 **商小言** 里上新一块 **「店铺营销话术生成」**：给商家一键出朋友圈文案、直播间口播、短标题。她在预发环境把温度从 **0.3 调到 0.85**，试了几条，文案 **又活又顺口**，群里一片 **「终于不像机器人」**。

阿梅很兴奋，文档里写：**「建议全应用默认开高温度，提升商家满意度。」** 提交变更单时，她勾了 **「同步到生产 · 商小言 · 全租户默认参数」**——却 **没勾选「仅营销工具接口」**，把 **整站对话的默认温度** 一起带上去了。

周六上午，工单炸得比剧本误发那次还快。

某服饰商家客服在 **退货 FAQ** 里连问三遍 **「洗后缩水能不能退」**，小言三次口径 **互相打架**：一遍说 **「按平台规则不支持」**，一遍说 **「您可以再拍细节图我们帮您争取」**，还有一遍冒出 **「亲，这次我们破例给您走特殊通道」**——商家客服差点按第三条真去承诺了。真人客服（坐席小张）在工单里备注：**「AI 说的『破例』算谁的？商家还是我们？」**

值班产研拉日志：检索片段是对的，上岗剧本也没被改；差异只在 **采样参数**——**温度 0.85** 让模型在多个「听起来都像人话」的续写里 **随机挑路径**，有的路径 **蹭到了剧本边缘的安抚话术**，有的 **踩进了未授权的补偿暗示**。

林薇建 **P1**，复盘板上写红字：

**「创意场景要活，可以；制度问答要稳，必须。把『灵感档』默认开到全站，等于让 FAQ 窗口也掷骰子。」**

当晚他们 **回滚全租户默认温度到 0.2**，并给营销工具单独挂 **「高温度模板」**——只在 **明确标记为创意生成** 的 API 路径上生效，不进退货与纠纷会话。

---

### 三、拧到零，又像念法条

回滚后的周一，另一家商家来投诉 **另一个极端**。

他们的 **3C 配件店** 用 **商小言** 做 **保修政策问答**，阿梅为了 **「绝不再飘」**，在租户配置里把温度 **手动调到 0.0**，又加了 **「请严格按检索原文组织答案，不要发挥」** 的用户侧模板。

口径是稳了——连问十次，**「保修期 12 个月，人为损坏不保」** 几乎一字不差。但商家运营反馈：**「买家觉得我们在背条例，转人工率反而涨了。」** 更麻烦的是，有个用户问的是 **「进水了算人为吗」**，检索片段里其实有 **「进液属于人为损坏」** 的判例说明，小言却 **十次里有六次** 只重复 **「人为损坏不保」** 六个字， **没把「进水」和「人为」连起来**——不是答错，是 **太不敢展开**。

小陈在白板上画了一条线：

**「温度太低，模型几乎每次都选『概率最高的下一句』——像考试时只背标准答案。制度题要稳，但不能稳到 **该解释的分支不讲**。」**

林薇让评测（第 14 篇）加 **「同问 N 次方差」** 指标：  
- **FAQ / 制度 / 对账口径**：温度 **0–0.3**，且要求 **关键结论一致率 ≥ 95%**；  
- **营销文案 / 标题灵感**：单独接口，温度 **0.7–0.9**，但必须 **带「仅供参考，请人工审核」** 水印，且 **不得接 Grounding 里的赔偿条款片段做自由发挥**（扣第 11 篇）。

阿梅问：**「那能不能前台给商家一个『活泼 / 标准 / 严谨』三档？」**

林薇摇头：**「可以做成体验语言，但 PRD 里要写清背后映射：活泼≠随便承诺；严谨≠省略必要解释。三档背后是 **温度区间 + 剧本条款 + 评测用例**，不是单个滑块万能。」**

### 四、标准流程与中小默认方案

为了不再靠「谁手滑同步全站」赌运气，林薇拉了个小会，把 **双轨** 写进平台规范。

**标准 / 规范流程（大客户、多租户、要强审计）**

| 环节 | 做什么 |
|------|--------|
| **参数分层** | 平台默认（低）→ 应用默认（商/内/修各异）→ 租户可选（在允许区间内）→ **单接口覆盖**（仅营销生成） |
| **变更门禁** | 温度默认值变更 = **二级发布**；必须写 **影响范围：哪些 API、哪些会话类型** |
| **评测** | 固定集 **同问 10 次** 看方差；抽 **越权承诺**、**口径打架**、**该解释不解释** 三类 bad case |
| **可观测** | 日志记录 **temperature、top_p（若启用）**；事故复盘能还原「当时旋钮在哪」 |

**中小常用默认方案（星尘当前灰度）**

- **买云模型、不接自研**：多数厂商默认 **0.7 左右**；星尘 **FAQ 类强制 0.2 以下**，创意工具 **单独 0.8**，商家 **前台不提供温度滑块**，只给 **话术风格三档（映射好的区间）**。  
- **可暂缓**：租户自助调参、与 **老何（模型路由）** 联动「创意走四楼、FAQ 走二楼」——先 **参数分接口**，再 **路由分模型**。  
- **升级触发**：若商家投诉 **「同一问题不同答案」** 或 **「文案太水/太硬」** 两周内超阈值，再开 **租户级微调（第 12 篇）** 评审，**不要先用高温掩盖剧本不清**。

小陈补充了一个工程事实，写在帮助文档脚注里：

**「温度 0 也不是数学上的 100% 复制粘贴——还有 top_p、种子、模型版本更新。产品承诺要写 **『高度一致』，别写『完全一致』。」**

### 五、接近结尾：剧本之后，还有一声呼吸

又一周，李女士在 **内小言** 问年假，回复依旧克制；商小言 **退货窗口** 连问五次，关键结论一致，只在礼貌用语上有轻微差别； **营销话术生成** 入口出的文案仍然活泼，但 **不带赔偿暗示**，且按钮旁有 **「请人工确认后再发布」**。

配置台上，林薇给新同学讲岗，指着 **Temperature** 那一栏：

**「用户以为小言『性格变了』。其实是我们在后台拧了一个 **采样松紧旋钮**——同一本上岗剧本，**敢不敢用不同的词把同一件事说出来**。」**

小陈把英文写在标题旁：**Temperature。** 阿梅学乖了，改参数前先问：**「这是给 **朋友圈文案** 的，还是给 **纠纷 FAQ** 的？」**

张先生在 **修小言** 群里晚到一条：**「故障码解释别给我写诗，温度拧低点；检修步骤清单可以稍微活一点，但别换顺序。」** 林薇记下来：**同一应用里也可以 **分场景默认温度**——不是全站一个数。**

## 概念正解

**Temperature / 温度参数**，是控制大语言模型在生成每一个词（Token）时 **随机性有多大** 的推理参数：数值 **越高**，越愿意选 **概率略低但更新鲜** 的续写，输出 **更多样**，也更容易 **偏离剧本边缘或编造细节**；数值 **越低**，越倾向 **最常见、最稳妥** 的下一句，输出 **更一致**，但也可能 **呆板、漏掉该展开的分支**。它 **不改变模型知识本身**，也 **不能替代** System Prompt、RAG 或护栏——只是在 **「已经合规的几条说法」之间，选哪一条、敢不敢多走一步**。

回扣故事：五次测试是 **同剧本同检索、不同采样路径 → 口吻与承诺边界漂移**；阿梅同步全站是 **创意场景参数污染 FAQ**；商家调到 0.0 是 **过度求稳导致该解释不解释**；分接口与三档映射是 **产品按场景拆参数，而非单滑块通吃**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写明 **默认温度、允许区间、哪些 API 可覆盖**；FAQ 写 **同问 N 次关键结论一致率**；创意写 **多样性 OK 但须人工审核** |
| **方案评审** | 问：高温会不会增加 **幻觉/越权承诺**？低温会不会伤 **体验与转人工率**？是否和 **top_p、固定 seed** 一起评估？ |
| **指标设计** | 口径方差率、越权话术命中率、创意采纳率、因「答复不一致」引发的工单占比 |
| **跨团队沟通** | 对运营：**「活泼档只给营销工具，不给纠纷窗口」**；对算法：**「温度变更要不要回归第 14 篇固定集」**；对客服：**「不一致先查参数版本，别先怪模型笨」** |

## 类比锚定

**核心类比：连锁客服中心的「质检抽样 vs 自由发挥」考核。** 同样背过 SOP 的坐席，有人 **每句按标准话术念**（低温），有人 **在红线内换说法拉近距离**（高温）。考核 **红线与 SOP** 仍是 System Prompt + 制度库；**温度** 管的是 **在红线内允许多大发挥空间**。

| 你熟悉的 PM 经验 | Temperature 要素 |
|------------------|------------------|
| 客服 **标准话术库** vs **个性化安抚** | **低温度：稳口径** / **高温度：活表达** |
| **「同一问题答案要一致」** 的质检 KPI | **低温度 + 同问多次回归** |
| 营销 **创意 brief** 鼓励多稿、A/B 文案 | **高温度用于灵感生成，且需人工审核** |
| 活动页 **「仅供参考，以合同为准」** 免责声明 | **高温输出必须叠加水印/护栏，不能进承诺类场景** |
| 新人 **只背条例不会举例** | **温度过低 → 该展开的分支不讲** |

## 类比边界说明

### 一、概念核心要点

1. **Temperature 管的是「选词随机性」**，不是知识多少、也不是剧本内容；与 **System Prompt、RAG、Memory** 分工不同。  
2. **场景要分档**：FAQ、制度、对账、纠纷 → **低温**；营销灵感、头脑风暴 → **高温 + 人工闸门**。  
3. **高温会放大风险**：口径打架、越权暗示、幻觉花样增多——需 **评测同问多次 + 护栏 + 剧本硬条款**。  
4. **低温不是万能稳**：可能 **体验生硬、漏解释**；不能靠把温度拧到 0 掩盖 **检索或剧本不清**。  
5. **产品对外宜映射为体验语言**（严谨/标准/活泼），对内保留 **数值区间、接口级覆盖、变更审计**。

### 二、类比在哪里会失效

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像调人性格，越高越热情」 | 高温也可能 **乱承诺、胡编**，不是单纯「更亲切」 | 是 **采样分布变宽**，质量与风险一起上升 |
| 「低温就等于质检满分」 | 0 温度仍可能 **答错事实**（检索错、幻觉仍在） | 低温保 **说法一致**，不保 **事实正确** |
| 「一个滑块全站通用」 | 不同任务应 **分接口/分路由**；还与 **top_p、模型版本** 交互 | 用 **场景默认 + 发布范围**，避免全租户误同步 |
| 「调到 0 就每次一模一样」 | 工程上仍有 **浮点、并行、版本更新** 等变量 | 承诺 **高度一致**，并靠 **评测集** 验证 |
