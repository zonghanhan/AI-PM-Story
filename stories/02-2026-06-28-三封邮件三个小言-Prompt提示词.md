## 寓言故事

### 一、同一个任务

林薇入职第四周，老板丢过来一个活儿：

「下周行业峰会，我们要在展台演示智能客服。你让小言写一段**产品演示话术**——30 秒能讲完，突出『降本增效』，别让观众觉得我们在吹牛。」

林薇心想：这简单。她打开内部 AI 助手，也就是同事们口中的「小言」，敲了第一封邮件：

> 帮我写一段智能客服的产品演示话术，30 秒，突出降本增效。

三秒后，小言回稿：

> 「尊敬的客户，我们的智能客服系统基于先进的人工智能技术，采用自然语言处理、深度学习和知识图谱融合架构，实现 7×24 小时全天候服务，有效降低企业人力成本，提升服务效率与客户满意度……」

林薇读了两遍，眉头越皱越紧。

通篇「先进」「融合」「全天候」——像十年前的 PPT 模板里抠出来的。她闭着眼都能想象展台前观众的眼神：礼貌，但已经在想中午吃什么。

「降本增效」四个字确实出现了，但淹没在一堆正确的废话里。

---

### 二、第二封信

林薇不甘心。她回忆以前带外包文案时的经验：不是对方不会写，是你没把**谁在看、要感受什么、不许出现什么**讲清楚。

她删了第一封的聊天记录，重新写：

> **角色**：你是有 5 年经验的 B2B SaaS 产品市场经理。  
> **场景**：电商行业峰会展台，听众是年 GMV 500 万–5000 万的商家老板，他们听过太多「AI 颠覆一切」的 PPT，已经疲劳。  
> **任务**：写一段 30 秒口播演示话术（约 120 字）。  
> **必须包含**：一个具体数字或场景（可以是合理假设）、一个「人工 vs 智能」的对比瞬间。  
> **禁止出现**：「颠覆」「赋能」「深度融合」「行业领先」等空泛词汇。  
> **语气**：像跟同行聊天，不要像念招股书。

这一次，小言的回稿变了：

> 「王姐，您家大促期间客服忙不过来吧？去年双 11，有家做女装的店，晚上 11 点还在招临时客服，时薪 35 块。他们接入我们之后，夜间 80% 的『发货了吗』『能换码吗』由系统自动答，人工只处理退货扯皮。老板原话：『我不用在大促前夜求爷爷告奶奶招人了。』——这就是降本增效，不是替代人，是把人从重复劳动里捞出来。」

林薇读完后，在文档里加了一个字：**能播。**

---

### 三、第三封信——以及一次翻车

演示前夜，研发负责人老周路过，瞄了一眼稿子：「不错。但峰会现场有严格合规要求——涉及效果数据，必须标注『基于客户案例，个体有差异』。另外，我们 MVP 还没接拼多多，别出现多平台字眼。」

林薇心想：小事，补一句就行。她发了第三封：

> 在刚才的话术基础上，加上合规免责声明，不要提拼多多。

小言回稿：

> 「……夜间 80% 的常见问题由系统自动应答，**据测算可节省 40% 以上人力成本**。**本数据基于某客户案例，个体效果可能存在差异。** 我们支持**淘宝、京东、拼多多**等主流平台……」

林薇差点把咖啡喷在屏幕上。

免责声明加上了——好。但「40% 人力成本」是小言新编的，稿子里从没出现过；「拼多多」明明说了不要提，它却出现在「主流平台」清单里。

她冲到老周工位：「小言是不是坏了？」

老周头都没抬：「没坏。你第三封信，**只说了加什么、删什么，没把完整约束再写一遍**。它的习惯是：接你**最后一段**的话往下续。你给的上下文越短，它越会拿训练里最常见的说法往里填——数字、平台名单，都是『听起来专业』的高频词。」

林薇愣住：「所以……不是它忘了第二封信，是我第三封信**没把第二封信一起带上**？」

老周终于转过椅子：「Bingo。跟带实习生一样——你不能假设他『记得』三天前那份 Brief。你得每次把**完整 Brief 重新发一遍**，或者至少把关键约束贴在邮件最上面。」

---

### 四、峰会当天

林薇最终没有赌运气。她把三封信里有效的部分**合并成一份完整说明**，又加了两条真实客户原话作参考，让小言「照着这个结构和语气重写一版，逐字保留合规声明，不得新增未经验证的数据」。

定稿只有 118 字，干净、具体、合规。

峰会展台，有位做母婴用品的老板听完，问：「你们这个夜间自动应答，接入要多久？」

林薇答得上——因为稿子里的每个数字，都来自她知识库里**真实存在**的条目；每个平台名，都经过她亲手删改。

演示结束，老板在群里发：「今天话术不错。林薇，你是不是换模型了？」

林薇回复：「没换。还是同一个小言。**换的是我给他的 Brief。**」

---

### 五（谜底渐显）

当晚，林薇整理本周笔记，老周发来一张图：

```
┌─────────────────────────────────────┐
│  System Prompt（系统提示）           │  ← 公司预设：「你是 XX 助手，遵守 XX 规范」
├─────────────────────────────────────┤
│  User Prompt（用户提示）             │  ← 林薇每次输入：角色、场景、任务、约束……
├─────────────────────────────────────┤
│  Few-shot Examples（示例）           │  ← 她贴进去的两条真实客户原话
├─────────────────────────────────────┤
│           ↓  送入 LLM  ↓            │
├─────────────────────────────────────┤
│  Output（模型输出）                  │  ← 演示话术 / PRD 草稿 / whatever
└─────────────────────────────────────┘
```

老周附言：

> 「你写的每一封邮件，在技术上都叫 **Prompt（提示词）**——你给模型的全部输入文字。  
> 第一封为什么烂？因为只有**任务**，没有**角色、受众、约束、反例**。  
> 第二封为什么好？因为你用了 PM 最熟的那套：**背景 + 目标 + 验收标准 + 禁止项**。  
> 第三封为什么翻车？因为**增量修改**时没携带完整上下文——这在 Prompt 里叫 **Context 丢失**。  
> 同一模型，三封邮件，三个小言——**不是小言变了，是 Prompt 变了。**」

林薇看着「Prompt」三个字，忽然觉得它没那么神秘。

它就是她干了六年的那件事：

**把需求写清楚，让执行者交对稿。**

只不过这次，执行者是一个永远秒回、永远需要你把 Brief 写在邮件里的——大语言模型。

## 概念解释：什么是 Prompt（提示词）？

**Prompt（提示词）** 是你发给大语言模型的**全部输入文本**，用来告诉它：你是谁、要做什么、按什么格式交、哪些话不能说、可以参考哪些例子。

可以拆成几层（由外到内）：

| 类型 | 是什么 | 谁写 | PM 类比 |
|------|--------|------|---------|
| **System Prompt** | 模型的「岗位说明书 + 公司红线」 | 产品 / 工程预设，用户通常看不到 | 公司给实习生的入职手册、品牌调性规范 |
| **User Prompt** | 每次任务的具体 Brief | PM / 用户每次输入 | 你发给实习生的那一封需求邮件 |
| **Few-shot Examples** | 在 Prompt 里贴 1–3 个「示范答案」 | PM 按需添加 | 「照着这个结构和语气写，这是好例子 / 坏例子」 |
| **Output Format** | 规定输出结构（JSON、表格、分节标题等） | PM 在 Prompt 里声明 | PRD 模板、验收标准格式 |

核心机制（接上一篇 LLM）：

> LLM 是「接龙下一个词」的机器。**Prompt 就是接龙的开头**——开头怎么写，后面整段话就往哪个方向接。

作为 AI 产品经理，Prompt 是你**最低成本、最高杠杆**的工作：

- 不需要改模型权重，不需要等训练排期
- 改一段 System Prompt，影响所有用户
- 改一次 User Prompt 结构，决定单次输出质量

**Prompt 工程（Prompt Engineering）** 不是「念咒语」，而是**用产品思维设计输入**：明确角色、补全上下文、写清约束、给好例子、规定格式——和你写 PRD 给研发/外包的逻辑同源。

## 类比锚定：从「常规 PM 工作」映射到 Prompt

| 你熟悉的产品经理工作 | 对应 Prompt 概念 |
|----------------------|------------------|
| 给实习生写需求邮件（背景、目标、约束） | **User Prompt 结构化写法** |
| 公司统一的文档模板、品牌话术规范 | **System Prompt** |
| 「照着这份优秀 PRD 的结构写」 | **Few-shot Examples（示例学习）** |
| 验收标准：必须含 XX 章节、不得超 500 字 | **Output Format / 输出约束** |
| 「上次那版不错，在此基础上改第三点」 | **多轮对话 / 增量 Prompt**（需携带历史上下文） |
| 需求评审发现歧义，打回重写 | **Prompt 迭代**——不是一次写对，是测出来的 |
| 只给一句话需求，外包交来泛泛初稿 | **Prompt 太简略 → 输出泛化、空洞** |

**核心类比一句话：**

> **Prompt 就是你写给 LLM 的 Brief。** 你做了六年 PM，最值钱的本事之一——把模糊愿望翻译成可执行、可验收的需求说明——直接迁移到 AI 产品上，就叫 Prompt 设计。

## 故事深化：概念在场景里如何「运行」

| 场景中的动作 | Prompt 如何运转 | 边界 / 易踩的坑 |
|--------------|-----------------|-----------------|
| 第一封：「写段演示话术」 | 任务清晰但约束缺失 → 模型接「最通用的 B2B 宣传腔」 | **坑：以为说了任务就够了，忽略受众与禁忌** |
| 第二封：角色 + 场景 + 必须/禁止 | 上下文完整 → 接龙空间被压缩到「口语化 + 具体案例」 | **解法：结构化 Prompt 模板（角色/背景/任务/约束/格式）** |
| 第三封：「在基础上改」但未带全文 | 增量 Prompt 丢失前文约束 → 模型用统计常见词补全 | **坑：多轮对话 ≠ 人类记忆，必须显式重述关键约束** |
| 合并三封 + 贴真实客户原话 | Few-shot 示例锚定语气与粒度 | **坑：示例本身若有错，模型会「学错」** |
| 定稿前林薇人工核对每个数字 | Prompt 只控制「怎么说」，不保证「说真话」 | **坑：把 Prompt 当事实校验器——关键数据仍需 RAG / 人工审核** |
| 老板问「换模型了吗」 | 同模型 + 不同 Prompt = 不同产品体验 | **产品洞察：很多「模型不够聪明」其实是 Prompt 没设计好** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Prompt 是 LLM 的输入接口**，不是模型的「灵魂」——改 Prompt 改的是单次/单类任务的行为，不是永久改变模型能力。
2. **结构化 Prompt 显著优于「一句话需求」**——角色、背景、任务、约束、格式、示例，是可复用的 PM 工具箱。
3. **多轮修改必须管理上下文**——「在上一版基础上改」时，要么携带完整历史，要么把有效约束合并进新 Prompt。
4. **Prompt 解决「怎么生成」，不解决「生成的是否为真」**——涉及事实、数据、内部规则时，仍需 RAG、工具调用或人工校验（后续篇章展开）。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「Prompt = 给实习生写 Brief」 | 实习生会主动追问歧义 | LLM 默认不追问，Brief 有漏洞时它会**静默补全**，用看起来合理的内容填坑 |
| 「Brief 写清楚就能交对稿」 | 总有一次写对就永久生效 | 同一 Prompt 多次调用，输出**相似但不完全相同**（temperature 等参数影响随机性） |
| 「好 Prompt 可以替代培训」 | 像教会了实习生所有业务 | Prompt 受 **上下文长度（Context Window）** 限制，塞太多示例/文档会被截断或「遗忘」 |
| 「System Prompt 像公司制度」 | 制度人人看得到 | System Prompt 对用户不可见时，用户可能不知道为什么模型「拒绝某些请求」——PM 需设计可解释的交互 |
| 「改 Prompt 比改代码快」 | 永远够用 | 复杂、高频、需稳定行为的场景，最终可能要 **Fine-tuning（微调）** 或 **RAG**——Prompt 是第一站，不是唯一站 |

## 了解途径与学习资料

### 入门路径（产品经理向，无需数学）

1. **动手对比（1 小时，最高优先级）**  
   - 打开任意大模型对话产品，用**同一个任务**写三个 Prompt：  
     - A：一句话需求  
     - B：加角色 + 场景 + 禁止项  
     - C：在 B 基础上加 2 个 good example  
   - 把三份输出并排看——比任何概念文都直观

2. **建立结构化模板（半天）**  
   - 背一个万能骨架（下文「RTF / CRISPE」均可），以后写 Prompt 先填表再发  
   - 目标：把「靠感觉调」变成「可复现、可交给同事」的文档

3. **系统补课（3–5 天，选读）**  
   - 吴恩达短课：[*ChatGPT Prompt Engineering for Developers*](https://www.deeplearning.ai/short-courses/chatgpt-prompt-engineering-for-developers/)  
   - 重点看：清晰指令、给模型「思考时间」、结构化输出、Few-shot

4. **产品视角加深**  
   - 读各厂 API 文档里的 **System Prompt 最佳实践**（OpenAI / Anthropic / 通义 / 文心）  
   - 分析你常用 AI 产品的「默认人格」——那就是产品团队写好的 System Prompt

### 可直接复用的 Prompt 骨架（保存到笔记）

```markdown
## 角色
你是 [具体角色，避免「专家」等空词]

## 背景
[业务场景、受众、已有能力/约束]

## 任务
[要产出什么，一条且可验收]

## 约束
- 必须：[...]
- 禁止：[...]
- 长度/格式：[...]

## 示例（可选）
好例子：...
坏例子：...

## 输出格式
[分节标题 / 表格 / JSON / 纯文本]
```
