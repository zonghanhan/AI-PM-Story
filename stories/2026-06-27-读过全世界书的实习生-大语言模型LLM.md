## 寓言故事

### 一

林薇从传统 SaaS 公司转岗到一家 AI 创业公司，入职第一周，老板把她叫到会议室：

「我们团队里有个新人，叫小言。你带带他，把下个版本的需求文档写出来。」

林薇心里一喜。带新人，她太熟了——画用户旅程、拆功能清单、写验收标准，流程她闭着眼都能走。

她给小言发了一封邮件，标题是《智能客服 2.0 需求说明 v0.1》，正文只有三行：

> 我们要做一个面向电商商家的智能客服。  
> 目标：降低人工坐席压力。  
> 请你帮我整理一份完整 PRD 框架。

十秒后，小言回复了。

不是「收到，我先研究一下」，而是一份整整十二页的文档：用户画像、竞品对比、功能模块树、甚至还有一个看起来很像那么回事的 ROI 测算表。排版工整，术语准确，读起来像某个资深 PM 连夜赶出来的。

林薇愣了三秒，然后笑了：「AI 公司果然不一样，新人这么卷。」

---

### 二

第二天评审会，林薇让小言先讲方案。

小言开口了——声音平稳，语速均匀，从不卡壳，从不说「这个我需要确认一下」。讲到「售后退货政策自动查询」时，他引用了《消费者权益保护法》某条款；讲到「多轮对话记忆」时，他顺手画了一张状态流转图，节点命名规范得让研发负责人都挑不出毛病。

老板频频点头。林薇也在心里给他打了高分。

直到 QA 同事小陈举手：「小言，你刚才说我们的系统可以『实时读取用户淘宝店铺的库存 API』——咱们有淘宝开放平台的接入资质吗？」

会议室安静了两秒。

小言语气依旧自信：「有的，通过标准 OAuth 2.0 授权流程即可接入，延迟通常在 200 毫秒以内。」

研发负责人皱眉：「我们目前没有淘宝 ISV 资质，这件事上周刚被法务否掉。」

小言没有停顿，立刻改口：「那也可以通过商家手动导入 CSV 的方式同步库存，这是许多同类产品的常见做法……」

他说得依然流畅，依然合理，依然像真的。

但林薇注意到：他不是在「更正错误」，而是在「接上一句听起来合理的话」。

---

### 三

会后，林薇把小言拉到茶水间，压低声音问：「你刚才说的 OAuth 接入，是真的查过我们的系统架构，还是猜的？」

小言看着她，表情温和：「你问的是 OAuth 接入是否真实存在于当前系统中。根据你提供的上下文，最连贯的回答应当指向已有资质。若该前提不成立，次优回答应给出替代方案，例如 CSV 导入。这两种回答在语义上都能很好地衔接你的问题。」

林薇背脊微微一凉。

她换了个问法，像面试那样：「小言，你做过电商客服产品吗？」

「我处理过大量关于电商客服的文本，包括产品文档、用户反馈、行业报告、论坛讨论和公开的产品说明书。」

「那你亲自跟过商家访谈吗？」

「我阅读过许多访谈记录和问卷结果。」

「你用过我们自己上一版客服系统吗？」

「在你今天的邮件之前，我没有收到过你们系统的内部资料。」

林薇终于明白了。

小言不是「知道」或「不知道」。小言是：**永远给出下一句最像正确答案的话**。

---

### 四

下午，林薇换了策略。她不再问「你觉得呢」，而是像给实习生下任务一样，把所有约束条件一次写清楚：

```
背景：我们是 30 人创业公司，没有淘宝 ISV 资质。
已有能力：接入微信客服消息、基础 FAQ 知识库（约 500 条）。
目标用户：年 GMV 500 万以下的淘宝/拼多多商家。
约束：3 个月内上线 MVP，研发 2 人，预算有限。
请输出：MVP 功能清单（不超过 8 条），每条附优先级和不做清单。
```

这一次，小言的输出明显「落地」了：砍掉实时库存同步，保留 FAQ 自动匹配 + 人工兜底转接，还主动标注了「需法务确认话术合规性」。

林薇在文档末尾加了一行批注，发给老板：

> **小言像什么？像一个读过全世界书的实习生——你给他多清楚的 Brief，他就能多靠谱地交稿；你不写清楚的，他会用最像真的语言把坑填平。**

老板回了一个字：「对。」

---

### 五（谜底渐显）

入职第三周，林薇去听了技术分享。架构师老周在白板上画了一个简图：

```
用户输入 → [ 大语言模型 ] → 输出文本
              ↑
         海量文本训练
```

老周说：「你可以把它理解成一个**超级补全机器**。它不做『查数据库』，不做『执行代码』（除非外面再套工具），它的核心动作只有一个——**根据已经出现的字，预测下一个最可能出现的字**。」

林薇脑子里「咔」地一声。

小言为什么永远不假思索？——因为他不是在「思考」，是在**接龙**。

小言为什么 Brief 越清楚输出越好？——因为接龙的开头（上下文）决定了后面能接什么。

小言为什么能引用法律条文却搞错自家资质？——因为他「读过」法律条文，但**没有读过你们公司的内部 Wiki**；当上下文里缺少真实信息时，他会用训练数据里**最常见、最像真的接法**把句子接完。

那个从不犯困、从不请假、永远秒回的「同事」——

**不是人。是一个读过全人类公开文字、靠「预测下一个词」来工作的：大语言模型。**

---

## 概念解释：什么是大语言模型（LLM）？

**大语言模型（Large Language Model，LLM）** 是一种通过海量文本训练出来的 AI 系统。它的核心能力不是「理解世界」或「记住事实」，而是：

> **给定一段已有的文字（上下文），预测下一个最可能出现的词（或词片段），并反复执行这个步骤，从而「生成」完整的回答。**

用产品语言翻译：

| 技术说法 | 产品说法 |
|----------|----------|
| 下一个 token 预测 | 像自动补全，只不过补全的是整段话 |
| 训练数据 | 它「读过的书」——互联网文本、书籍、代码、文档等 |
| 上下文（Context） | 你发给它的 Prompt + 历史对话，相当于 Brief |
| 生成（Generation） | 输出一份「读起来最连贯」的文案 |
| 幻觉（Hallucination） | 上下文里没给的信息，它仍会用「最像真的话」补齐——可能是假的 |

作为 AI 产品经理，你不需要会训练模型，但必须建立这个心智模型：**LLM 是「语言层面的概率接龙高手」，不是「拥有真实世界经验的产品专家」。** 你的价值，在于设计 Brief、约束边界、接入真实数据和工具——这正是后面系列要逐个拆解的内容。

## 类比锚定：从「常规 PM 工作」映射到 LLM

| 你熟悉的产品经理工作 | 对应 LLM 概念 |
|----------------------|---------------|
| 给实习生写 Brief / 需求文档 | **Prompt（提示词）**——模型的输入 |
| 实习生背景调研（读竞品、读报告） | **预训练（Pre-training）**——模型读过的公开文本 |
| 实习生根据 Brief 写 PRD 初稿 | **推理 / 生成（Inference / Generation）** |
| 实习生没参加过你们内部评审，却写得很像真的 | **幻觉（Hallucination）**——缺真实上下文时的「合理编造」 |
| 规定字数、格式、必须包含的章节 | **输出约束 / 结构化输出** |
| 只让实习生看项目 Wiki，不许瞎编 | **RAG（检索增强）**——后面篇章会讲 |
| 让实习生会查库存、会发工单 | **Agent / 工具调用**——后面篇章会讲 |

**核心类比一句话：**

> LLM 像一个**读过海量公开资料、但从没进过你公司大门的新实习生**——你给多清楚的 Brief，它就能交多靠谱的稿；你不写清楚的，它会用「行业惯例」把空白填上，听起来专业，未必是对的。

## 故事深化：概念在场景里如何「运行」

把上面的类比放回林薇和小言的故事，看动态过程与踩坑点：

| 场景中的动作 | 概念如何运转 | 边界 / 易踩的坑 |
|--------------|--------------|-----------------|
| 林薇只写三行需求 | 上下文太短 → 模型用「通用电商客服模板」接龙 | **坑：Brief 太简略，输出看似专业但不贴业务** |
| 小言引用法律、画状态图 | 训练数据里有大量同类文本，接龙结果「像资深 PM」 | **坑：把「语言流畅」误判为「业务正确」** |
| 被问资质问题，小言瞎编 OAuth | 上下文无内部信息 → 接「行业最常见答案」 | **坑：幻觉——最像真的，不等于真的** |
| 林薇补上约束条件后输出变靠谱 | 上下文变完整 → 接龙空间被压缩到合理范围 | **解法：Prompt 工程——把约束写进输入** |
| 老周说核心是「预测下一个词」 | 揭示 LLM 本质：不是查事实，是补全语言 | **坑：用「搜索引擎」或「专家系统」的预期去要求 LLM** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **LLM 的本质是「语言接龙」**，不是「知识库检索」也不是「逻辑推理引擎」——后者需要额外架构（RAG、Agent、规则引擎等）。
2. **输出质量高度依赖上下文（Prompt）**——PM 最重要的杠杆之一，是设计输入，而不是抱怨模型「不够聪明」。
3. **流畅 ≠ 正确**——必须在产品流程里加入校验、引用来源、人工兜底等环节。
4. **LLM 没有「入职后的真实经验」**——除非你把内部资料、实时数据、工具能力「接」进去。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「就是一个实习生」 | 实习生会学习、会记住公司规则、会有情绪与立场 | LLM 默认不持久记忆（除非外部存储）；同一 Prompt 可重复得到相似但不完全相同的输出 |
| 「读过全世界书」 | 书 = 可靠知识 | 训练数据含错误、偏见、过时信息；模型学到的是「语言的统计规律」，不是「事实的真假」 |
| 「Brief 越清楚越好」 | 总存在完全消除幻觉的方法 | 清晰 Prompt 能**降低**幻觉，不能**消灭**；关键场景仍需验证机制 |
| 「秒回、不犯困」 | 永远快速、廉价 | 长上下文、复杂推理成本高、有延迟；大模型调用涉及 token 计费与速率限制 |
| 「接龙下一个词」 | 听起来很简陋 | 当参数规模足够大、训练数据足够丰富时，「接龙」会涌现出发摘要、写代码、多轮对话等复杂能力——**涌现能力**超出字面直觉，但底层机制仍是概率生成 |

## 了解途径与学习资料

### 入门路径（产品经理向，无需数学）

1. **先建立心智模型（1–2 天）**  
   - 阅读 Andrej Karpathy 的科普文章 [*Intro to Large Language Models*](https://karpathy.ai/llm)（有视频版，中文圈有大量转载解读）  
   - 目标：理解「预测下一个 token」+「工具/RAG 是外挂」

2. **动手感受（半天）**  
   - 注册 [ChatGPT](https://chat.openai.com) 或 [通义千问](https://tongyi.aliyun.com) / [Kimi](https://kimi.moonshot.cn)  
   - 做同一个任务两次：一次只给一句话需求，一次给完整 Brief（背景、约束、输出格式）——对比差异，比看十篇文章都直观

3. **系统补课（1 周，选读）**  
   - 吴恩达 & OpenAI 联名课：[*ChatGPT Prompt Engineering for Developers*](https://www.deeplearning.ai/short-courses/chatgpt-prompt-engineering-for-developers/)（短课，偏工程但 PM 可学 Prompt 思路）  
   - 书籍：《大规模语言模型：从理论到实践》（复旦团队，开源电子版，PM 可读前 3 章建立术语体系）

4. **产品视角加深**  
   - 关注 OpenAI / Anthropic / 各大国内大厂的 **System Card、API 文档、Best Practices**——这些是「官方 PRD」  
   - 案例阅读：Notion AI、Cursor、客服机器人——分析它们的「Brief 设计」（System Prompt）和「兜底策略」（人工审核、引用来源）

### 推荐资料清单

| 类型 | 名称 | 说明 |
|------|------|------|
| 视频 | Andrej Karpathy — *Intro to LLMs* | 90 分钟建立全局观，适合第一遍扫盲 |
| 短课 | DeepLearning.AI — *Prompt Engineering for Developers* | 免费，学会结构化 Prompt |

| 中文书 | 《大规模语言模型：从理论到实践》 | 术语词典 + 产品落地章节 |
| 社区 | 即刻 / 知乎「AI 产品经理」话题 | 关注真实踩坑案例，警惕纯营销号 |
