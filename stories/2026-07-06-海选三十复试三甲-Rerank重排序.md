## 寓言故事

### 一、八份材料都像，答案却错了

上下文预算管理员上线后，VIP 客诉里「用户约束被挤出房间」的事故少了一大截。林薇刚松口气，运营又递来一份 **灰度周报**：

**检索命中率** 从 61% 升到 78%，但 **回答正确率** 只从 54% 涨到 58%——中间差出来的二十多个百分点，像一根刺。

她点开一条典型 **Bad Case**。

用户问：「我们买了 **企业版**，能不能接 **钉钉的单点登录（SSO）**？IT 说要看你们文档。」

小言答得流利：「企业版支持丰富的 **第三方登录** 能力，您可在 **个人中心 → 安全设置** 里绑定微信、手机号……」

林薇调出检索日志。Embedding 召回 Top8，每一段 **字面都沾边**：

| 排名 | 召回片段主题 | 为什么「像对的」 |
|------|--------------|------------------|
| 1 | 企业版 **价格与席位** | 含「企业版」 |
| 2 | **个人版** 第三方登录教程 | 含「登录」「第三方」 |
| 3 | 全站 **账号安全** 通用说明 | 含「安全」「登录」 |
| 4 | 企业版 **发票与对公** | 含「企业版」 |
| 5 | **OAuth 名词解释** | 含「单点」「授权」 |
| 6 | 企业版 **数据导出** | 含「企业版」 |
| 7 | 竞品对比：某厂 SSO | 含「SSO」「钉钉」 |
| 8 | 企业版 **试用期规则** | 含「企业版」 |

真正写「**企业版支持钉钉 SSO，配置入口在管理后台 → 集成中心**」的那段，排在 **第 23 名**。

林薇愣了两秒：「Embedding 不是按 **意思相近** 找邻居吗？用户问 SSO，第 7 段还有钉钉，怎么反而把 **个人中心绑微信** 送上桌？」

老周把日志翻到 **相似度分数** 那一列：「邻居找得快，不代表 **门牌号对**。你问的是 **企业版 + SSO 配置**，第 2 段『登录』词多，向量距离反而更近；**正确那段** 写的是『管理后台集成中心』，和用户口语里的『钉钉 SSO』 **不在同一个词袋里**。」

---

### 二、海选很快，复试才分真假

林薇第一反应是：**把 TopK 从 8 提到 30**，让正确段挤进圆桌。

老周摇头：「Context Window 那篇刚讲过——**椅子有限**。K 从 8 涨到 30，占座翻几倍，**截断会从别处开刀**；而且你把 30 段全塞给模型，它照样 **盯排名靠前的错段**——lost in the middle 可不会因为你『捞得多』就自动变聪明。」

他在白板上画了两道筛子：

```
用户问题
   ↓
【第一道：粗召回】Embedding / 关键词 / 混合检索
   → 从百万段里快速捞出候选池（比如 30～100 段）
   ↓
【第二道：精排序】Rerank（重排序模型）
   → 对「问题 + 每一段」做细读打分，重新排队
   ↓
取 Top3～5 进 Context Window，交给小言生成
```

「Embedding，」老周指着第一道筛子，「像 **海选票筛**：看简历关键词、毕业院校、大类经验，**毫秒级** 扫完整库，求 **快和全**——宁可多捞，不能漏。」

「Rerank，」他指着第二道，「像 **复试评委**：拿着用户原题，逐份问『这段 **能不能直接回答这个问题**』，**慢，但准**。」

林薇问：「复试评委是谁？」

「通常是 **专门的交叉编码模型（Cross-Encoder）**。」老周尽量不用术语，「Embedding 是 **各看各的简历** 比相似度；Rerank 是 **把问题和段落绑在一起读**，看 **配对关系**——『企业版』和『SSO』必须 **同段兑现**，不是两段各沾一个词。」

---

### 三、三十进三，不是多招人

算法同学小陈拿来 A/B 方案：

- **A 组**：Embedding Top8 直接进模型（现状）；
- **B 组**：Embedding Top50 → Rerank 取 Top3 进模型。

灰度三天，数据很有意思：

| 指标 | A 组 | B 组 |
|------|------|------|
| 检索阶段延迟（P50） | 45 ms | 48 ms |
| **Rerank 阶段延迟（P50）** | — | **+180 ms** |
| 端到端首字延迟 | 1.2 s | 1.45 s |
| **回答正确率（抽检）** | 58% | **71%** |
| 单轮 RAG 输入 Token | ~4,800 | **~1,800** |

林薇盯着最后一行：「正确率涨了，Token 反而 **少了**？」

小陈：「粗召回多捞，是为了 **给 Rerank 足够的选择面**——像海选进 50 人，不是为了 50 人全上班，是为了 **别在第 23 名就把真·SSO 文档漏掉**。复试只留 3 段 **真能答题的**，圆桌反而清爽。」

但 B 组也不是全胜。另一条 Bad Case 冒出来：

用户问：「**个人版** 能不能导出 **全部聊天记录**？」

正确段落在粗召回里排 **第 4**；Rerank 之后掉到 **第 9**——因为 rerank 模型训练语料里，「导出」常与「企业版合规归档」绑定，**把企业版大段推了上去**。

老周总结：「复试评委也会 **带偏见**。Rerank 不是玄学按钮，要 **看域、看语言、看版本**。」

---

### 四、产品要做的，是定规则不是堆模型

复盘会上，林薇没有简单地说「全量上 Rerank」，而是在 PRD 里写了 **三道闸**：

**闸 1：谁需要复试？**  
简单 FAQ、单关键词可查准的，**Embedding Top3 直达**；多条件、易混淆、高客单价场景（企业版、合规、医疗），**走 Rerank 管线**。

**闸 2：候选池多大？**  
不是越大越好。小陈测算：候选 **30～50** 段时，Rerank 延迟可控；上到 **200**，P95 延迟冲破客服 SLA。林薇把默认值定为 **粗召回 40 → 精排 3**。

**闸 3：复试不及格怎么办？**  
若 Rerank 最高分低于阈值，**不要硬答**——转人工、追问澄清、或触发 **关键词补救检索**（如强制含「SSO」「钉钉」的过滤）。

她还加了一条 **可观测性** 要求：日志里同时保留 **Embedding 排名** 与 **Rerank 排名**，方便排查「是真没召回，还是复试看走眼」。

上线两周后，那条钉钉 SSO 的工单被 replay：**粗召回第 23 名** 的集成中心文档，经 Rerank 升到 **第 1**，小言第一次答对接了配置路径。

散会上，林薇对新人说：「Embedding 解决 **库里有没有**；Rerank 解决 **哪一段该给用户看**。两道筛子，缺一道都可能 **言之凿凿地答错**。」

老周补了一句：「别指望一个模型又当海选又当复试——**快和准**，在检索里常常是 **分工，不是一个人全能**。」

## 概念正解

**Rerank（重排序）** 是在 RAG 等检索流程中，对 **粗召回** 得到的一批候选文档，用 **更精细的相关性模型**（常见为 Cross-Encoder）逐条与查询配对打分，并 **重新排序** 的技术环节。它解决的是 Embedding 等向量检索 **「语义相近 ≠ 能回答问题」** 的偏差，用 **额外延迟与算力** 换取 **Top 结果更准**；产品上要决定 **何时启用、候选池大小、最终进模型的条数、低分兜底策略**。

回扣故事：

- **「八份都像、答案却错」** → 向量相似度 **粗召回** 易被 **关键词共现** 误导；
- **「正确段落在第 23 名」** → 粗召回要 **足够宽**，但不能把宽池子 **全塞进 Context Window**；
- **「海选三十、复试三甲」** → **两阶段检索**：Recall（召回）与 Rerank（精排）分工；
- **「Rerank 把企业版大段推上去」** → 精排模型有 **域偏差**，需阈值、规则与监控；
- **「延迟 +180ms、Token 反而降」** → Rerank 是 **用算力换精度**，并可通过 **少而精的进窗** 省总成本。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写明 **粗召回通道（Embedding/关键词/混合）**、**候选数量 N**、**Rerank 后取 TopK**、**低分阈值与兜底**；标注哪些意图 **必须走 Rerank** |
| **方案评审** | 问算法：「Rerank 模型 **训练域** 是否覆盖我方知识库？Cross-Encoder **最大输入长度** 够不够用？P95 延迟 **是否击穿 SLA**？」 |
| **指标设计** | 分阶段看 **Recall@N**（正确段落在粗召回前 N 的比例）与 **MRR/NDCG@K**（精排后前排质量）；监控 **Rerank 改写排名** 的升降分布 |
| **跨团队沟通** | 跟业务说：「不是搜不到，是 **排错了**」；跟研发说：「别只调 Embedding，**两阶段要一起看日志**」 |

## 类比锚定

**核心类比：社招招聘的「简历海选 → 结构化面试打分」——先快速圈出候选人池，再逐人对岗评分，重排座次**

传统 PM 都做过或协同过 **招聘漏斗**：HR 用关键词筛出 50 份简历（快，可能漏也可能脏），用人经理 **按 JD 逐条打分**（慢，但决定谁进终面）。Rerank 就是检索场景里的 **「结构化面试」**。

| 你熟悉的 PM 经验 | Rerank 要素 |
|------------------|-------------|
| 简历系统按「Java、电商」关键词一秒筛出 200 人 | **粗召回**（Embedding/关键词）追求 **高召回、低延迟** |
| 岗位要求「Java + 电商 + 带团队」，关键词命中 ≠ 合适 | 向量 **语义相近** ≠ **能回答当前问题** |
| 用人经理花 3 天对 30 人做结构化面试，排出 Top5 进终面 | **Rerank** 对候选段 **逐条精排**，取 TopK 进模型 |
| 海选池太小，合适的人可能在第 80 名被漏掉 | 粗召回 **候选数 N 要够**（常 30～100），否则 Rerank **无力回天** |
| 终面只进 3 人，不是 30 人全进 CEO 办公室 | **Context Window 只装精排后的 3～5 段**，不是粗召回全灌 |
| 面试官有偏见，可能把「大厂光环」打高分 | Rerank 模型有 **训练偏差**，需 **阈值、人工抽检、域内微调** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Rerank 是检索链路的第二段**，输入是 **粗召回候选池**，输出是 **重排后的优先序**；不能替代 **建库、切片、Embedding 索引** 等上游工作。
2. **核心价值是提升 Top-K 精度**，尤其解决 **多义词、条件约束、版本差异**（企业版 vs 个人版）导致的 **「像而对」** 问题。
3. **粗召回与精排是权衡关系**：候选池越大，Rerank **越慢、越贵**；产品需为不同场景设 **默认 N 和 K**。
4. **Rerank 分数应参与兜底策略**：低置信度时 **拒答、追问、转人工**，比硬生成更安全。
5. **与 Context Window 协同**：精排后 **少而准** 的片段，往往比 **粗召回大块灌窗** 更省 Token、更降幻觉。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「面试一定能挑出最好的人」 | Rerank 模型对 **新领域、新表述、长尾问题** 可能打分失真 | 需 **域内评估集、阈值、规则过滤、人工回流**；不是上线即万能 |
| 「海选和面试由同一个人做就行」 | 用 **大模型对每个候选逐条打分** 当 Rerank，成本高、延迟爆炸 | 常用 **专用 Rerank 小模型** 或托管 API；与 **生成用大模型** 分工 |
| 「招 30 人面试越久越好」 | 候选 **200+** 时，Rerank **延迟线性涨**，实时客服等不起 | 产品要设 **N 的上限** 与 **分级策略**（简单问句跳过 Rerank） |
| 「排第一的就录用」 | 精排第一仍可能 **错**，且 **生成模型** 还可能无视引用 | 全链路指标要看 **最终回答正确率**；Rerank 后还有 **引用约束、校验** |
| 「只面试不改 JD」 | 若 Recall@N 长期低，说明 **切片、同义词、元数据** 有问题 | 应 **回溯粗召回与知识库治理**，不能无限堆复试 |

1. 从你们知识库挑 **5 个真实用户问句**，让研发导出 **Embedding Top10** 与 **Rerank 后 Top3** 的片段标题；标出 **排名变化最大** 的一条，写一句「为什么精排更合理/更离谱」。
2. 画一张 **检索漏斗图**：库内总段数 → 粗召回 N → Rerank K → 进模型 K'，标上你们现状的 **N、K、P95 延迟**。
3. 定义一条 **业务规则**：何种意图（如「企业版」「合规」「医疗」）**强制走 Rerank**；何种（如「营业时间」）**可跳过**——用一句话向业务方解释理由。

