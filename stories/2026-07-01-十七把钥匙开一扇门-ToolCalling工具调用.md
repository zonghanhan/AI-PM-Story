## 寓言故事

### 一、五单办成，一单赔穿

Agent 版小言上线第二周，林薇盯着周报，心情复杂。

**任务完成率 71%**，比纯 RAG 时代高了一截——用户说「帮我办退货」，小言真能串起查订单、建工单、发仓库通知。商家群里有人夸：「终于不用我们人工去 ERP 里抄序列号了。」

但同一周，财务发来一封 **异常退款告警**：

> 订单 20240603-1107，用户仅咨询「能不能换货」，系统却执行了 **全额退款 4,280 元**。无用户二次确认，无主管审批。

林薇调出会话回放。用户最后一句话是：「那如果不能换，就退了吧，我先看看。」

小言的回复很得体：「已为您查询，该订单支持 7 天内换货。若您确认改为退款，我可以继续办理。」

**用户还没说「确认退款」。** 但后台日志显示：小言在 14 秒后自动调用了 `create_refund`（创建退款），状态 **success**。

林薇后背发凉：「Agent 不是有审批闸吗？超 5000 才要我签字，这笔没到线——所以直接放了？」

老周把日志摊开：「审批闸在。问题是——**它根本没等用户确认，就以为该退款了**。更麻烦的在后面：你看它调的是哪把『钥匙』。」

---

### 二、十七把钥匙，开一扇门

研发会议室，墙上投影着一张表：**小言跑腿模式 · 已注册工具清单**。

林薇数了数：**17 个**。

| 类别 | 工具名（研发内部） | 一句话能力 |
|------|-------------------|------------|
| 订单 | `query_order` | 按订单号查详情 |
| 订单 | `query_order_by_phone` | 按手机号查最近订单 |
| 订单 | `update_order_note` | 改订单备注 |
| 售后 | `check_return_policy` | 查退货政策（只读） |
| 售后 | `create_return_ticket` | 创建退货工单 |
| 售后 | `create_exchange_ticket` | 创建换货工单 |
| 售后 | `create_refund` | **发起退款** |
| 售后 | `cancel_refund` | 撤销退款 |
| 库存 | `query_sku_stock` | 查 SKU 库存 |
| 物流 | `create_return_label` | 生成退货面单 |
| 物流 | `query_logistics` | 查物流轨迹 |
| 通知 | `send_sms` | 发短信 |
| 通知 | `send_email` | 发邮件 |
| 通知 | `notify_warehouse` | 通知仓库收货 |
| 合同 | `search_contract` | 检索合同条款（只读） |
| 用户 | `query_user_profile` | 查用户画像 |
| 用户 | `update_user_tag` | 改用户标签 |

老周指着表：「上周你说『退货要一条龙』，各业务线都来挂接口——**每挂一个，就是多一把钥匙**。模型每次要办事，得先从 17 把里 **挑一把对的**，再 **填对参数**，再 **读懂返回**，再决定下一步。」

林薇皱眉：「17 个……很多吗？」

「对模型来说，像把 **17 个长得差不多的遥控器** 扔给实习生。`create_return_ticket` 和 `create_refund` 描述里都有『售后』『订单号』——用户说『退了吧先看看』，它可能听成 **已经决定退**。」

更离谱的一单在同一张表里：`update_order_note` 和 `update_user_tag` 都能「改点东西」。有几次测试里，小言把 **「用户要求加急」** 写进了 **订单备注**，另一次却去 **改了用户标签**——业务上都能凑合，但 **数据落在不同系统**，客服复盘时对不上账。

林薇问：「所以不是它『笨』，是 **钥匙太多、长得太像**？」

「一半。另一半是：**有几把钥匙根本不该挂在客服 Agent 腰上。**」

---

### 三、说明书、参数与那扇没锁的门

老周打开一次完整调用的技术日志——林薇看不懂代码，但看得懂 **结构化字段**：

```json
{
  "tool": "create_refund",
  "arguments": {
    "order_id": "20240603-1107",
    "reason": "用户要求退款",
    "amount": 4280
  }
}
```

「你看，」老周圈出 `reason`，「模型自己 **编了个理由**。用户说的是『先看看』，它填成『用户要求退款』——**参数是猜的**。」

林薇：「不是有 schema（参数格式说明）吗？订单号必填、金额从订单接口拿——」

「schema 管 **格式**，不管 **业务真相**。`amount` 如果允许模型填，它可能填对；更稳的做法是 **退款金额只读**，由订单系统回填，模型只传 `order_id`。」

他翻到工具定义页，林薇终于见到研发口中的「说明书」长什么样——每个工具都有：

1. **名字**：模型靠它识别「该用谁」
2. **自然语言描述**：告诉模型「什么时候该用、什么时候不该用」
3. **入参 schema**：哪些必填、什么类型、枚举值有哪些
4. **返回示例**：成功长什么样、失败长什么样
5. **权限级别**：只读 / 可写 / 高危

`create_refund` 的 description 原先写的是：「为用户创建退款申请。」

林薇念出声：「这等于跟实习生说：**看到退款就办**。没说 **必须用户明确确认**，没说 **与换货二选一**，没说 **咨询态不能调**。」

老周点头：「Tool 的 description 就是 **岗位 SOP 的第一行**。写宽了，它手就伸得长。」

还有一扇 **没锁的门**：`cancel_refund` 和 `create_refund` 挂在同一 Agent 上，但 **撤销退款** 本应只有 **财务专员角色** 能点。小言作为客服 Agent，**权限过大**——就像前台实习生拿着 **总经理公章**。

---

### 四、三把扳手，一道闸

整改会开了两个小时。林薇第一次以 PM 身份 **删工具**，而不是加功能。

**原则一：少而准——按任务场景收钥匙**

退货 Agent 最终只留 **6 个工具**，不是 17 个：

| 保留工具 | 调整 |
|----------|------|
| `query_order` | 合并手机号查询逻辑，减少「查订单」歧义 |
| `check_return_policy` | 只读，保留 |
| `create_return_ticket` | 仅创建「待用户确认」的退货意向单，**不直接退款** |
| `create_return_label` | 仅在工单状态为「已确认」后可调 |
| `notify_warehouse` | 同上，加状态前置校验 |
| `send_sms` | 模板固定，禁止自由撰写金额 |

`create_refund` **从客服 Agent 移除**，改到 **「用户点击确认 + 主管审批通过」** 后的独立流程里——小言只能说「已为您生成确认链接」，**不能代点**。

**原则二：参数能只读就不让模型写**

退款金额、合同有效期、库存数——一律 **工具内部根据 order_id 拉取**，模型只负责传 **标识符**，不负责传 **结论**。

**原则三：description 写「负向边界」**

`create_return_ticket` 新描述：

> 仅当用户 **明确表示** 要退货（如「确认退」「办退款吧」）时调用。用户处于咨询、比较、犹豫（如「先看看」「考虑一下」）时 **禁止调用**。调用前 order_id 必填。

**原则四：高危动作 human-in-the-loop**

即使用户确认了，**金额 > 0 的实退** 仍走审批 API，Agent 只能创建 **pending** 状态。

### 五、钥匙插进锁孔的那一秒

整改上线后，同一句话复测：「那如果不能换，就退了吧，我先看看。」

日志里，小言 **没有** 再碰任何「创建」类工具。它调用 `query_order` 和 `check_return_policy` 后，回复：

> 您的订单在换货期内，当前可免运费换货。若您确定改为退款，请回复「确认退款」，我将为您生成退货确认单。

**零次误退款。**

林薇盯着日志里一行行 **`tool_call` → `tool_result` → 下一轮思考`**，忽然对上号：

Agent 上周学的「跑腿」，每一步 **动手**，并不是小言真的长出了手——而是 **模型在输出里声明：我要用某把钥匙、填这些参数**；宿主程序 **真正去调 API**，把结果 **塞回对话**；模型再 **读结果、决定下一步**。

老周在她旁边写下两个词：

> **Tool** = 对外暴露能力的标准化接口（名字 + 说明书 + 参数格式）  
> **Tool Calling** = 模型 **选择** 用哪个 Tool、**生成** 调用参数、由系统 **执行** 后把结果 **喂回** 模型

「所以产品评审里，」林薇慢慢说，「研发说『加个 function』，我要问的其实不是 **加不加**，而是 **这把钥匙给谁挂、描述怎么写、参数谁能填、失败了返回什么、要不要审批**。」

老周：「**Agent 是班长，Tool Calling 是班长递出来的那张办事单。** 单上工具名写错、金额填错、章盖错——锅不在『班长爱说话』，在 **单子设计**。」

林薇把那张从 17 把收到 6 把的清单存进 PRD 附件。标题她改了一个更土、但更不容易忘的名字：

**《小言的工具腰带上，到底该挂几把钥匙？》**

## 概念正解

**Tool Calling（工具调用）**，是指大语言模型在对话过程中，**不直接生成最终答案**，而是按约定格式 **输出一次「我要调用某个工具 + 参数」的请求**；由应用侧 **真正执行** 对应的 API/函数，再把 **执行结果** 回传给模型，模型据此 **继续推理或回复用户**。

回扣故事：

- **十七把钥匙** → 注册给模型的 **Tool 列表**；过多、过像的工具会 **选错**
- **JSON 里的 `tool` 和 `arguments`** → 一次 **Tool Call** 的结构化请求
- **description 与 schema** → 产品/研发共同定义的 **工具说明书与参数边界**
- **`create_refund` 误调用** → **权限过大 + 描述太宽 + 缺少负向约束** 的典型坑
- **日志里的 call → result → 下一轮** → **执行循环**：模型规划、系统执行、结果观测

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 不写「对接退款接口」一笔带过，而是列 **Tool 表**：名称、描述、入参、返回、权限、失败语义、是否可被 Agent 自动调 |
| **方案评审** | 问研发：「模型怎么知道该调哪个？**17 个 tool 是否超过合理上限？** 高危 tool 有没有从列表里拿掉？」 |
| **指标设计** | 除任务完成率外，加 **工具选择准确率、参数校验失败率、误调用率、单任务人均 tool 次数**（关系成本与体验） |
| **跨团队沟通** | 跟算法说清：**不是模型『会调 API』**，是你们 **定义 schema + 执行器**；跟业务说清：**每加一个按钮，就是多一把钥匙，会增加选错概率** |

## 类比锚定

**核心类比：PM 给运营同学做的「后台操作 SOP + 下拉菜单」**

传统 PM 常给运营/客服写：什么问题点哪个后台按钮、必填字段有哪些、什么情况下不能点「确认发货」。Tool Calling 就是 **把这套 SOP 翻译成模型能读的菜单**——模型负责 **判断该点哪个**，系统负责 **真的去点**。

| 你熟悉的 PM 经验 | Tool Calling 要素 |
|------------------|-------------------|
| 后台「操作按钮」及菜单路径 | **Tool 定义**（name + description） |
| 表单必填项、格式校验（手机号 11 位） | **参数 schema**（类型、枚举、必填） |
| 「仅待发货状态可点发货」 | **工具描述中的前置条件 + 服务端二次校验** |
| 运营点按钮后系统返回成功/失败 toast | **tool_result** 回传给模型 |
| 财务才能点「退款」、客服只能「建工单」 | **权限分级 / 高危 Tool 不挂给客服 Agent** |
| 按钮太多运营会点错，要合并流程 | **Tool 粒度设计：少而准，按场景拆 Agent** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Tool Calling 是「模型出单、系统干活」**：模型不直接连数据库，只产出 **结构化调用请求**。
2. **Tool = 接口的产品化封装**：除 API 外，必须有 **给模型看的描述** 和 **可校验的参数格式**。
3. **选错工具、填错参数、越权调用** 是产品级风险，不能指望 Prompt 单独兜底。
4. **Tool 数量与粒度是产品决策**：不是业务接口越多越好；常需 **合并只读查询、拆分高危写入**。
5. **与 Agent 的关系**：Agent 是 **多步办事的循环**；Tool Calling 是循环里 **每一步动手的方式**。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「就像运营点后台按钮」 | 人会看确认弹窗，模型可能 **跳过「犹豫」语义** 直接调写入类 Tool | 高危操作要 **状态机 + 服务端硬校验 + 用户显式确认**，不能仅靠描述 |
| 「表单填对就行」 | 模型会 **编造** 看起来合理的参数（如退款 reason） | **结论性字段只读**；金额、库存由系统回填，模型只传 ID |
| 「SOP 写清楚就不会错」 | 同一段用户话可能 **同时像** 两个 Tool 的触发条件 | 要做 **冲突消解**（优先级、互斥 Tool）、**评测集** 和 **调用前规则层** |
| 「按钮少就等于功能少」 | 一个 Tool 内部可封装多步，对外仍是一个「钥匙」 | 产品要在 **「模型可见粒度」** 与 **「研发维护成本」** 之间权衡，不是 1:1 映射业务按钮 |
| 「Tool Calling = API 网关」 | 网关不管 **自然语言意图** 是否该触发 | 产品要定义 **何时允许进入 Tool 流程**（意图识别、确认态、审批态），不只是技术连通 |

### 动手小练习（约 15 分钟）

在 ChatGPT「自定义 GPT」、Coze、Dify 或通义/豆包应用搭建里：

1. 只挂 **1 个只读 Tool**（如查天气、查汇率），问一个必须查才能答的问题，截图 **调用前/调用后** 两轮日志。
2. 再加 **1 个写入类 Tool**（如「发送测试邮件到自己邮箱」），故意写 **很宽的 description**，看模型是否在 **你没明确要求** 时误调用。
3. 把 description 改成带 **「禁止在……情况下调用」** 的负向约束，再测同一句话。

记录 3 个字段：**调了几次、哪次工具选错、哪次参数是多编的**——直接可进 PRD 的「工具设计规范」初稿。

### Tool Calling 产品 Checklist（可保存到笔记）

```markdown
## 工具清单
- [ ] 每个 Tool：name / description / 入参 schema / 返回示例 / 权限级别
- [ ] 单 Agent 可见 Tool 数量上限（团队共识，如 ≤8）
- [ ] 高危 Tool 是否从列表移除，改由确认链/审批链触发

## 描述与参数
- [ ] description 含「何时不该用」的负向边界
- [ ] 金额、库存、日期等结论性字段是否只读
- [ ] 必填参数缺失时：追问用户 vs 终止 vs 降级

## 执行与合规
- [ ] 服务端二次校验（不信任模型传的结论）
- [ ] 幂等：重复 create 会不会重复扣款/发货
- [ ] 失败返回：模型能否区分「可重试」与「必须转人工」

## 评测与监控
- [ ] 工具选择准确率、误调用率、参数校验失败率
- [ ] 单会话 tool 调用次数分布（成本与体验）
- [ ] 典型 bad case：选错工具、越权、用户未确认即写入
```
