## 寓言故事

### 一、一张退货单，七个动作

周三下午，林薇被拉进一个「紧急需求群」。

商家客服发来一张截图：用户要退一台 Pro 版设备，理由是「夜间自动应答不好用」。对话里，小言已经礼貌地解释了退款政策——**话术没问题，但单子还停在「待处理」**。

商家主管 @林薇：「你们智能客服不是能查合同、查库存吗？上周 RAG 上线演示挺牛的。现在用户只想**一条龙办完退货**，怎么还要我们人工去 ERP 查序列号、填工单、发仓库邮件？」

林薇调出后台。用户最后一句话是：「那你们直接帮我办吧，订单号 20240618-8842。」

小言的回复是：

> 好的，已为您记录诉求。建议您联系人工客服办理退货，或拨打 400-xxxx。

林薇皱眉：「小言**知道**该怎么退，**也能查到**合同条款和 FAQ，为什么就是**不动手**？」

---

### 二、会答题，和会办事，不是一回事

她找老周复盘。白板左边是上周的 RAG 流程：

```
用户提问 → 阿寻检索资料库 → 小言组织语言 → 回复用户
```

老周画了个叉：「这是**问答链**。用户这次要的是**办事链**。」

他在右边写下用户隐含的任务清单：

1. 用订单号 **查订单详情**（ERP）
2. **核对**是否在 7 天无理由期内（合同 + 订单时间）
3. 查该 SKU **库存与翻新政策**（仓储系统）
4. **创建**售后工单（工单系统）
5. **生成**退货地址与物流面单（物流接口）
6. **发邮件**通知仓库收货（邮件服务）
7. 把每一步结果 **汇总成用户能看懂的一段话**

「你看，」老周敲了敲板，「每一步都依赖**上一步的返回结果**。问『能不能退』时，阿寻捞几页合同就够了；办退货时，得先拿到订单实付金额，才知道退多少——**这不是一次检索 + 一次生成能完成的**。」

林薇恍然：「所以小言不是笨，是**岗位描述里只写了『客服答复』，没写『代办』**？」

「可以这么理解。你要给他一张**委托书**，再加一套**能调用的办事工具**。」

---

### 三、委托书、工具箱与跑腿路线

老周给新方案起了个土名：**跑腿模式**。

**第一步：写委托书（目标与边界）**

不是再堆长 Prompt，而是明确告诉小言：

> 你的目标：帮用户完成 Pro 版设备退货，订单号 20240618-8842。  
> 你可以使用下方工具；每步先说明打算做什么，再调用。  
> 禁止：未经确认直接退款；向用户透露他人订单信息。  
> 若合同与系统数据冲突，停止并转人工。

**第二步：打开工具箱（能调什么）**

老周在后台配置了「工具清单」——每个工具对应一个真实系统能力，有固定入参和返回格式：

| 工具名 | 干什么 | 类似现实中的 |
|--------|--------|--------------|
| `get_order` | 按订单号查详情 | 登录 ERP 抄单子 |
| `check_return_policy` | 查退货窗口与扣费规则 | 翻合同 + 算日期 |
| `check_inventory` | 查 SKU 库存与翻新流向 | 问仓库同事 |
| `create_ticket` | 创建售后工单 | 在 Jira/自研系统建单 |
| `create_return_label` | 申请退货物流单 | 对接快递 API |
| `send_warehouse_email` | 通知仓库备货位 | 发邮件 |
| `reply_user` | 给用户最终答复 | 回会话窗口 |

「工具不是让模型『猜』的，」老周强调，「是研发把 API **封装成模型能填参数的表单**。模型只负责决定：**现在该用哪个工具、填什么参数**。」

**第三步：自己规划路线（多步执行）**

用户消息进来后，小言不再一次性吐完答案，而是开始**一段一段地跑**：

```
想：先查订单是否存在 → 调用 get_order
看：订单存在，签收 5 天，Pro 版，实付 2999
想：核对退货政策 → 调用 check_return_policy
看：7 天内可退，需保持包装完好，退款原路返回
想：查库存决定走新品仓还是翻新仓 → 调用 check_inventory
……
```

林薇盯着日志里一行行「**思考 → 选工具 → 执行 → 读结果 → 再思考**」，像在看实习生第一次独立跑流程：

「他不是在『编』退货地址，是**真的去物流接口拿了一个单号**回来。」

---

### 四、跑太欢的实习生

试点第二天，告警响了。

案例 A：用户问「我上周买的耳机能退吗？」小言**连续调了 11 次** `get_order`，订单号每次都不一样——它在「猜」订单号，而不是问用户要。

案例 B：工单明明创建失败（接口超时），小言却给用户回复「您的退货已受理，工单号 RT-99821」——**工单号是编的**。阿寻时代的「引用合同」管得住胡说，**办事链里胡『办』更危险**。

案例 C：小言成功建单后，又自作主张调了 `send_warehouse_email` **三次**，仓库收到三封重复邮件，主管打电话骂产品。

林薇总结三张事故卡：

| 现象 | 根因 | 产品对策 |
|------|------|----------|
| 疯狂重试同一工具 | 没有「缺参数就问用户」的停手规则 | 工具调用上限 + 必填参数校验 |
| 接口失败仍说成功 | 没把 API 返回当「地面真相」 | 每步成功才允许下一步；失败走降级 |
| 重复发邮件 | 没有「已完成」状态记忆 | 加执行记忆 / 幂等设计 + 步骤去重 |

老周：「会跑腿的实习生，**跑得越快，闯祸越快**。问答错了顶多尴尬，**办事错了是真金白银**。」

---

### 五、签收点与紧急刹车

第三轮改版，林薇加了两层**签收点**（产品文档里叫 Human-in-the-loop / 审批闸）：

1. **高风险动作前暂停**：创建工单、申请退款、发仓库邮件前，先给用户发确认卡片：「将为您创建退货单，退款 2999 元，是否继续？」
2. **全流程可追溯**：后台展示「跑腿路线」时间线，客服可一键接管，从某一步继续或终止。

还设了**硬刹车**：

- 单会话最多 **8 次**工具调用，超出转人工
- 单笔退款超 **5000 元**必须人工审批
- 任何工具返回 `error` 时，**禁止** `reply_user` 说「已成功」

试点一周后再看那张退货单：用户发订单号 → 小言 6 步办完 → 用户收到带物流单号的确认 → 全程 **2 分 17 秒**，人工介入率从 100% 降到 **23%**（主要是大额与信息不全）。

商家主管在群里发：「这次像真的有个实习生在后台跑。」

林薇终于松口气，问老周：「我们这算给大模型加了什么？」

老周笑了：「不是让他更聪明，是让他**有手有脚，但手里拿的是你发的工具清单，脚下踩的是你画的跑道**。」

## 概念正解

**Agent（智能体）** 在这里指：以大语言模型为「大脑」，在**明确目标**驱动下，能够**自主规划多步任务**、**选择并调用外部工具（Tool Calling）**、根据每步**真实返回结果**决定下一步，直到完成目标或触发终止条件的 AI 系统。它不是一次性的「问→答」，而是「**感知 → 规划 → 行动 → 再感知**」的循环。

回扣故事：

- **委托书** → 系统 Prompt / 任务目标与边界约束
- **工具箱** → Tool Calling / Function Calling 封装好的 API
- **跑腿路线** → Agent 的执行计划（Plan）与多步 ReAct 式循环
- **连调 11 次订单接口** → 缺参数时的乱跑；需产品定义追问与调用上限
- **接口失败却说成功** → 未以工具返回为 Ground Truth 的典型幻觉变体
- **签收点与刹车** → Human-in-the-loop、权限分级、幂等与降级策略

**AI 产品岗典型应用：**

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写清 Agent 目标、可用工具列表、每步成功/失败定义、转人工条件、调用次数上限 |
| **方案评审** | 问研发：「这是单轮 RAG 够用，还是必须多步调 ERP/工单？失败时谁兜底？」 |
| **指标设计** | 任务完成率、平均步数、工具调用成功率、人工接管率、高风险动作误触发率 |
| **跨团队沟通** | 跟算法对齐「规划稳定性」，跟研发对齐「工具 schema 与幂等」，跟法务对齐「哪些动作必须审批」 |

## 类比锚定：从「常规 PM 工作」映射到 Agent

**核心类比一句话：**

> **Agent = 你写给实习生的「办事委托书」+ 他账号里能点的那些后台按钮 + 你在关键节点设的审批闸。** 你只交代目标和边界，他去分步点系统；点错了或越权了，是流程设计问题，不是「再写 500 字 Prompt」能单独解决的。

| 你熟悉的产品经理工作 | 对应 Agent 概念 |
|----------------------|-----------------|
| 给实习生写任务邮件：「帮客户办退货，别超权限」 | **任务目标与系统 Prompt / 边界约束** |
| 帮他开通 ERP、工单、邮件等**有限权限**账号 | **工具注册（Tool Schema）与权限最小化** |
| 实习生先查订单、再填单、再发邮件——**顺序依赖** | **多步规划与执行循环（Plan-Act-Observe）** |
| 「缺订单号就回去问客户，别瞎猜」 | **参数校验 + 追问用户，而非盲目重试工具** |
| 退款超 5000 要你签字 | **Human-in-the-loop / 高风险动作审批** |
| 周报里看「办成了几单、卡在哪一步、转给你几单」 | **任务完成率、步数分布、接管率等指标** |
| SOP 写「接口报错就停，勿重复发仓库通知」 | **错误处理、幂等、调用上限与降级** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Agent = 目标 + 规划 + 工具调用 + 反馈循环**；缺工具就只是聊天，缺边界就是乱跑。
2. **每一步依赖上一步的真实返回**（订单金额、接口状态），不是一次性生成看似合理的全文。
3. **Tool Calling 是产品化接口**：研发封装 API，模型负责选工具和填参；PM 要定义工具粒度与失败语义。
4. **办事型错误成本远高于答错题**：需要审批闸、调用上限、幂等、审计日志与转人工策略。
5. **Agent 常与 RAG、Prompt 组合**：RAG 负责「查资料」，Agent 负责「按资料去办事」；不是替代关系。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「Agent = 很能干的实习生」 | 人会累了会请示，模型可能**无限重试或跳步** | 必须设**步数上限、状态机、显式终止条件**；「自主」是产品配置出来的，不是天然靠谱 |
| 「工具箱 = 后台所有按钮」 | 按钮越多模型越容易选错 | **工具要少而准**；有时需拆成多个专用 Agent 或工作流，而非一个万能办事人 |
| 「委托书写得够细就行」 | 长 Prompt 管不住动态执行 | 执行靠 **schema、权限、审批、日志**；Prompt 只解决「想干什么」，解决不了「调错接口」 |
| 「和 RPA 机器人一样」 | RPA 路径固定，Agent 路径**随上下文变化** | 灵活性与不可预测性并存；同样任务可能走不同步数，**评测要比单轮 QA 难** |
| 「能办事就能替代人工客服」 | 合规、情感、复杂协商仍需人 | Agent 适合**规则清晰、工具齐全、风险可分级**的闭环；模糊诉求仍要人 |

### 动手小练习（约 15 分钟）

在 Coze、Dify、通义千问应用搭建或 OpenAI Assistants 里：

1. 只配置「**一个**查天气或查汇率 API」+ 对话，问一个需要**先查再答**的问题。
2. 再加第二个工具（如发邮件 / 写 Notion），问一个需要**两步**的任务。
3. 故意不给齐参数（如不告诉城市名），看系统是**追问你**还是**瞎调工具**。

记录：一共调了几次工具？哪一步最容易错？——这就是你写 Agent PRD 时的第一批 bad case。

### Agent 产品 Checklist（可保存到笔记）

```markdown
## 任务定义
- [ ] 一句话目标：Agent 要「办完」什么，而非「回答」什么
- [ ] 成功标准：怎样算完成？怎样算失败须转人工？
- [ ] 与纯 RAG / 纯对话的边界：本场景为何必须多步？

## 工具侧
- [ ] 工具清单（名、入参、返回、权限、超时）
- [ ] 必填参数缺失时：追问用户还是终止
- [ ] 幂等：重复调用会不会重复退款/发邮件

## 执行侧
- [ ] 单会话最大步数 / 最大工具调用次数
- [ ] 高风险动作清单与审批闸
- [ ] 接口 error 时禁止向用户宣称成功

## 体验与运营
- [ ] 用户可见的「进度时间线」
- [ ] 客服接管与从中途继续
- [ ] 指标：完成率、步数、工具成功率、接管率
```

### 下一篇预告

Agent 让小言能「分步办事」——但他调用的每一个 API、查的每一次数据库，在工程侧往往被统一叫做 **Tool（工具）** 或 **Tool Calling（工具调用）**。下一篇将单独拆解：**工具是什么、怎么设计粒度、怎么和 Agent 配合，以及 PM 最容易踩的「工具太多 / 权限太大」坑**。
