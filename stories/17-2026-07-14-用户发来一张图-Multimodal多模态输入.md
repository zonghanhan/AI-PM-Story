## 寓言故事

### 一、评审会上，销售递过来一张真机照片

周三下午，林薇主持 **设备售后助手 2.0** 需求评审。销售老赵把手机投屏到大屏上——不是 PPT，是一张 **用户刚发来的微信图**：

一台工业交换机，外壳鼓包，旁边贴着 **手写的故障单**，背景里还有 **半块铭牌** 和 **另一台设备的二维码**。

老赵说：「客户原话：**『小言，你看看这还能修吗？型号我看不清。』** 咱们现在的小言只会回：**『请用文字描述故障现象』**—— **单子当场就黄了**。」

客服主管阿梅补刀：「竞品上周上线了 **拍照报修**。用户 **懒得打字**，尤其现场 **一手油一手机**—— **谁先能看图，谁先拿单**。」

林薇心里清楚：文本链路他们打磨了三个月； **图** 怎么进系统、进模型、进账单， **没人画过流程图**。

她问了一个很 PM 的问题：「 **用户期望小言从图里看出什么？**」

会议室安静三秒。老赵说：「 **能不能修、要不要换、型号是啥**。」阿梅说：「 **最好还能自动建工单，别让用户再填十项**。」

林薇在白板写下：**识别对象 → 读清文字 → 判断风险 → 给可执行建议**。她宣布：「 **两周内出方案**——但先别喊『上大模型视觉』，我们拆清楚 **图到底怎么进门**。」

---

### 二、三条路：OCR、专用视觉、一把梭多模态

第二天，小陈（算法）、大刘（工程）、阿寻（知识库）挤在林薇工位旁，像 **分拣中心讨论包裹类型**。

**路线 A：传统拆解——OCR + 文本小言。**

大刘投屏一条旧流水线：

```
用户上传图片
  → OCR 抽出铭牌文字、故障单手写体
  → 文本小言 + RAG 查维修手册
  → 输出建议
```

小陈说：「 **优点**：文本链路成熟， **成本可控**，幻觉还能靠 **Grounding 引用手册** 压住。 **缺点**：图里 **鼓包、烧焦、液体渗漏** 这种 **视觉损伤**，OCR **根本看不见**——它只认字，不认 **『这台机器已经不像一台机器了』**。」

阿梅在群里秒回：「用户问的就是 **『还能修吗』**—— **光看字不够**。」

**路线 B：专用视觉模型——先分类再问答。**

小陈继续：「可以先跑一个 **图像分类/检测** 小模型：**设备类型、损伤等级、是否涉电风险**；再把 **结构化标签** 喂给文本小言。」

大刘皱眉：「等于 **维护两套模型 + 一套编排**。分类错了，后面 **全错**；而且 **手写故障单、遮挡铭牌** 这种长尾， **分类数据从哪来**？」

**路线 C：多模态大模型——图和字一起上桌。**

小陈打开一份 API 文档：「 **Vision 版** 模型支持 **图片 + 文字** 同一轮输入。用户问『还能修吗』， **图和问题一起送进去**，模型 **同时看损伤、读铭牌、组织回答**。」

林薇问：「 **和路线 A 比，产品差异在哪？**」

小陈：「 **少一道人工设计的拆解**——不用你先规定 **必须先 OCR 再 Rerank**；模型自己决定 **先看哪里**。但 **贵、慢、且会瞎编**：图里看不清的型号，它也可能 **一本正经胡说**。」

林薇想起上月 **推理成本** 复盘： **一张高清图**，折算成 Token， **常常顶得上几千字**—— **运费又涨价了**。

她在白板画了个三角形：**体验完整度 / 单次成本 / 可解释与合规**，三条边互相拽。

---

### 三、一次试点：图进来了，坑也进来了

一周后，他们拿 **五十张脱敏真机图** 做内部试点。林薇定了三条 **产品红线**：

1. **涉电、冒烟、漏液** → 必须 **建议停机 + 转人工**，不许在线拍板「继续用」；  
2. **型号、序列号** → 回答里要标 **「图中所见 / 不确定」**，禁止 **伪造铭牌信息**；  
3. **单次报修会话成本** ≤ **0.6 元**（财务给的上限）。

试点第二天，阿梅甩来三条 **翻车记录**：

**翻车 1：用户传了相册里 **去年的旧图**。**

小言热情分析鼓包位置， **工单却建在了当前站点**—— **时间线错了**。大刘：「多模态 **不会自动知道图是什么时候拍的**，除非 **读 EXIF 或让用户确认**。」

**翻车 2：图里铭牌反光，模型 **猜了一个很像的型号**。**

售后师傅到现场发现 **配件不对**—— **二次上门成本** 比 **模型费** 贵十倍。林薇立刻加规则：**低置信度必须追问「请补拍铭牌特写」**，不许 **硬答**。

**翻车 3：一张图里 **两台设备**，用户问「哪台坏了」。**

多模态模型 **盯左边那台讲半天**，用户其实指的是 **右边**。小陈：「 **没有指向性** 的多物体场景，要先 **让用户点选或画框**—— **产品交互** 问题，不是模型单独能解。」

林薇把坑归纳成一句话：**「图进门」不等于「图被正确理解」**—— **上传、压缩、裁剪、追问、置信度展示、人工兜底**，每一环都要设计。

---

### 四、方案定稿：不是「全能看图」，是「分镜头」

又过三天，林薇在评审里 **否了「所有图片都扔旗舰 Vision」** 的方案—— **账单会炸**，而且 **简单二维码扫码** 根本不值得 **Vision 价**。

她拿出 **分镜头路由表**：

| 用户意图（先判） | 输入类型 | 走哪条路 | 产品备注 |
|------------------|----------|----------|----------|
| 扫码查保修 | 清晰二维码 | 专用扫码 API → 文本小言 | **不走 Vision** |
| 报修 · 外观损伤 | 设备整机照 | 多模态（中档） + 手册 RAG | 涉电风险词 **强制转人工** |
| 报修 · 型号不清 | 铭牌特写 | OCR 优先；失败再 Vision | **低置信度追问** |
| 发票 / 合同截图 | 密集小字 | OCR + 版式解析 → 文本 | 注意 **隐私打码** |
| 语音描述故障 | 音频 | ASR 转文字 → 文本链路 | 多模态 **不止图** |

小陈补充：**「多模态」在工程上常指 **同一模型能处理多种模态输入**（图、文、音，有的还有视频）；在产品上你要问的是：** 哪一种模态 **在这步真的不可替代**？**」

交互稿里，林薇加了一个 **「拍前引导」** 浮层： **对准铭牌 / 避开反光 / 一台设备一张**——像快递 App 教你 **怎么拍包裹**。

老周（法务）只提一点：「图里常有 **人脸、手机号、门牌、邻桌文件**—— **上传前裁剪 + 服务端脱敏 + 留存周期写进隐私政策**。」

---

### 五、上线前夜：小言终于能「看」了，但 PM 要会「选」

灰度发布前，林薇给新人过了一遍 **什么叫多模态产品化**——她这时才把术语挑明：

**Multimodal / 多模态**，不是「功能列表里加一个上传按钮」，而是 **产品允许用户用多种媒介表达意图**（图、文、音、文件等），系统用 **能处理对应模态的模型或流水线** 理解并回应； **核心 PM 判断** 是：**哪种模态在这步必要、走专用链路还是统一大模型、成本与风险谁兜底**。

她强调五条 **上线后仍要盯的指标**：

1. **分意图成功率**——用户拍照后， **路由是否判对**（报修 vs 扫码 vs 误传表情包）；  
2. **视觉置信度与追问率**——多少会话需要 **补拍**；追问后 **一次解决率** 是否提升；  
3. **分场景单次成本**——Vision 档 **是否被滥用**（本可走 OCR 的也走 Vision）；  
4. **幻觉型型号错误率**—— **铭牌类事实** 必须 **可抽检、可对账**；  
5. **涉险转人工率与时延**—— **安全场景宁可误转，不可漏转**。

灰度第三天，老赵转来用户反馈：「 **终于不用打字了**。」同时财务苏姐私信：「 **带图会话成本是纯文本的 4 倍**—— **可以，但别全员默认 Vision**。」

林薇回：「 **多模态是体验杠杆，也是成本开关**—— **该用的时候用，别用空运寄明信片**。」

她给下一篇留了钩子：图能看了， **视频巡检、实时语音对讲** 也在排期—— **模态越多，路由表越长**；而 **有些图，用户根本不该上传**。

## 概念正解

**Multimodal / 多模态输入**，指 AI 系统 **同时接收并理解多种信息形态**（常见为 **文本 + 图像**，也可含 **音频、视频、文档版面** 等），在同一轮或同一任务链里 **综合这些输入** 完成识别、推理与回答。它解决的核心问题是：**用户表达并不只有字**——现场、实物、票据、口音，都承载着文本里 **没有或说不清** 的信息；产品要把 **「上传」变成「可被正确理解」**，而不是简单把文件塞进模型。

回扣故事：

- **「用户拍设备问能不能修」** → 典型 **视觉 + 语言** 联合理解场景， **纯 OCR 不够**；  
- **「OCR / 专用视觉 / Vision 大模型三条路」** → 产品常见的 **流水线拆解 vs 端到端多模态** 选型；  
- **「旧图、反光铭牌、一张图两台设备」** → **时序、置信度、指代** 等多模态特有风险；  
- **「分镜头路由表 + 拍前引导」** → **模态路由、交互引导、专用小模型与 Vision 混用**；  
- **「带图会话 4 倍成本」** → 图像常按 **Token 或像素档** 计费，要 **和推理成本优化联动**。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写清 **支持哪些模态**（仅图？图+音？）、 **每种模态的入口与限制**（大小、格式、是否裁剪）、 **低置信度追问与转人工** 规则 |
| **方案评审** | 问：「这张图 **必须 Vision 吗**？能否 **OCR/专用检测 + 文本**？ **一张图多少 Token**？ **EXIF、多物体、反光** 怎么处理？」 |
| **指标设计** | 除准确率外，加 **分模态路由命中率、补拍率、视觉事实错误率、分模态单次成本**；安全类 **漏转人工率** 单列 |
| **跨团队沟通** | 跟算法说：「 **哪些视觉问题用结构化标签接文本链路**」；跟设计说：「 **拍前引导比事后纠错便宜**」；跟法务说：「 **图里敏感信息谁脱敏、存多久**」 |

## 类比锚定

**核心类比：跨境电商的「买家秀 + 文字描述」客诉判责——传统 PM 熟 **用户同时给图和字、要先看清凭证类型再决定走视觉鉴定还是查 SKU 文本、模糊图要补材料、凭证里可能带隐私**；多模态输入是把同一套 **「先分拣媒介，再组合判断」** 搬到 AI：**不是来了一个包裹就一律开箱上鉴定师，而是先分快递、证件照、破损照，各走各的 SOP**。**

| 你熟悉的 PM 经验 | Multimodal 要素 |
|------------------|-----------------|
| 客诉 **附图**：有 **破损照、物流面单、聊天记录截图** 不同类型 | 用户 **多模态输入**：设备照、铭牌、发票、语音， **意图与模态组合各异** |
| **先识别凭证类型** 再决定走 **仅 OCR 查单号** 还是 **找鉴定看货损** | **模态路由**：扫码走 API、密集文字走 OCR、外观损伤走 Vision |
| 图 **模糊、过曝、非本次订单** → 要求 **补拍或补说明** | **低置信度追问**、 **时序确认**、 **单物体引导** |
| 凭证上的 **手机号、地址** 要 **打码存档** | 图像 **隐私合规**：裁剪、脱敏、留存周期 |
| 「 **有图有真相**」但 **鉴定师贵** → 简单件 **客服看字即可** | **成本分层**：非必要不 Vision； **分场景成本上限** |
| 判责要 **留痕**：依据 **哪张图哪个字段** | **可解释输出**：「图中所见 / 不确定」、 **引用 OCR 片段** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **多模态是「输入形态」问题，不等于「一定要最大的视觉大模型」**——很多场景 **OCR、专用检测、扫码 API + 文本模型** 更便宜、更可控；Vision 适用于 **需要整体理解画面语义** 的步骤。
2. **图像会显著推高 Token 与延迟**——产品要 **压缩、裁边、限分辨率**，并在 PRD 里写 **分模态成本**；与 **Inference Cost** 同一套经营逻辑。
3. **多模态放大幻觉与事实风险**——型号、金额、日期等 **可从图里「猜」出来**；必须 **置信度策略、追问、抽检、高风险转人工**。
4. **交互设计是多模态产品的一半**——拍前引导、点选目标物体、补拍流程，往往比 **换更大模型** 更有效。
5. **合规与隐私随模态升级**——图音视频 **含人脸、地标、第三方信息**； **上传前告知、服务端脱敏、存储期限** 要在需求阶段写清。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像鉴定师看图，一眼准」 | 模型 **会自信地错**；反光、遮挡、低分辨率下 **胡编型号** | **置信度 + 追问 + 人工抽检**；关键事实 **能 OCR 则 OCR 对账** |
| 「有图就走视觉大模型」 | **浪费钱**；二维码、纯文字截图 **不需要 Vision** | **分镜头路由**；定期审计 **Vision 调用占比是否过高** |
| 「多模态 = 用户啥都能传」 | **视频、长音频、大图** 带来 **时延、成本、审核** 爆炸； **恶意文件** 风险 | **白名单格式与大小**、 **异步处理**、 **内容安全审核** |
| 「和客诉凭证一样，图就是证据」 | AI **输出不是法律效力证据**； **旧图、修图** 无法靠模型识别 | 产品 **不承诺「以图定罪」**；涉责场景 **人工复核 + 系统记录** |
| 「听懂人话就行」 | **方言、噪音、专有名词** 下 ASR 错误会 **污染后续文本链路** | 音轨 **先 ASR 再文本** 时，要 **ASR 置信度与重录引导** |

### 动手小练习（约 15 分钟）

1. 选你业务里的 **一个「用户会发图」场景**，画 **两张流程图**：一张 **OCR+文本**，一张 **Vision 端到端**；标出 **哪一步非 Vision 不可**。
2. 写 **3 条拍前引导文案**（如对准铭牌、单台设备、避开反光）——想象用户 **一手油一手机**。
3. 打开所用模型的 **定价页**，查 **「一张 1080p 图大致等于多少 Token」**（或按张计费规则），用 **日活 × 带图比例** 粗算 **月增量成本**。

### Multimodal 产品 Checklist（可保存到笔记）

```markdown
## 需求阶段
- [ ] 列出支持的模态：图 / 音 / 视频 / 文件，及格式大小上限
- [ ] 分意图模态路由表（何时 OCR、何时专用 API、何时 Vision）
- [ ] 低置信度追问、补拍、转人工规则
- [ ] 图像隐私：裁剪、脱敏、留存周期、用户告知

## 方案评审
- [ ] 是否「非 Vision 不可」？有无更便宜链路？
- [ ] 多物体、旧图、反光、遮挡的交互与策略
- [ ] 分模态单次成本上限与监控

## 上线与运营
- [ ] 视觉事实错误抽检（型号、金额等）
- [ ] 涉险场景漏转人工率
- [ ] Vision 调用占比审计，防止滥用旗舰档
```
