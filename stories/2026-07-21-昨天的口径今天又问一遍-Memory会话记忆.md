## 寓言故事

### 一、财务认账，人不认人

结构化灰度第二周，张先生又点开小言。上回对账折腾三天，最后在确认单上签了字；财务那边 **`importReconciliation` 早已成功**。

他开门见山：**「上月差异再拉一遍，**还是按昨天说的 B 线口径**，把停线那三条备注带上。」**

小言 **流式** 回得礼貌又自信：请先说明您指的 **B 线口径** 是哪一版？MES 与 PLC 以哪个字段为准？停线记录需要从哪套系统检索？

张先生愣住，把昨天的聊天记录往上翻——**整整两屏**，全是小言带着他 **逐条对齐口径** 的过程。那些话 **还在飞书里**，却 **不在这一通对话的「脑子」里**。

他 @ 林薇：**「系统入账了，助手失忆了。」**

---

### 二、把聊天记录塞进下一轮？

站会上，工程第一反应很直觉：**「把历史消息全带上，不就等于记得？」**

大刘当场泼冷水。他调出 **Token 账单（第 6 篇）** 和 **圆桌座位图（第 9 篇上下文窗口）**：张先生这类工单，单次对账上下文 **轻松上万字**；若 **30 天内的会话都原样拼接**，还没问新问题，**座位就满了**——要么 **挤掉检索来的工单片段**，要么 **直接截断**，表现成 **「记得开头，忘了关键一句」**。

算法小陈补刀：**「全量历史 ≠ 记忆，是 **把仓库搬进圆桌**。** 圆桌（上下文窗口）装的是 **这一回合要用的材料**，不是 **人生档案柜**。」**

运营小夏举手：**「那让用户 **自己复制粘贴** 昨天结论？」**

林薇摇头：**「那是把 **产品债** 转嫁给最不该背锅的人。记忆要是能力，就得 **产品化**：记什么、给谁看、多久失效。」**

她白板上画了三层：

```
① 本轮圆桌上的话（当前 session 的多轮消息）
② 便签：从长对话里摘出来的「结论条」（摘要记忆）
③ 档案柜：跨天、跨设备还在的「用户偏好/工单上下文」（持久记忆）
```

**「缺哪一层，就会出现哪一种『像第一次见面』。」**

---

### 三、便签写歪了，比失忆更糟

第一周他们先上了 **便签层**：每轮对话结束，让模型 **自动生成三条摘要**——「用户身份」「已确认口径」「待办事项」，下次对话 **贴在 Prompt 最前面**。

张先生第二天再来，小言 **确实开口就叫出了 B 线口径**——但 **停线备注只记了「有三条」**，没记 **来自哪次 MES 导出**；检索按摘要里的模糊词 **捞错批次**，结构化 JSON **格式全对**，Grounding（第 11 篇）却 **对不上原始片段**。

更麻烦的是 **李女士** 的账号：便签里误把 **张先生会议里随口提的「先按保守估」** 写进了 **她的** 摘要——因为测试环境 **没把 user_id 锁死**，两张便签 **串了抽屉**。

门卫老关（第 18 篇护栏）连夜加规则：**便签必须带 **来源会话 ID**、**写入时间**；跨用户读取 **硬拒绝**；涉及金额口径的便签 **必须链到 Grounding 引用 ID**，不许只留自然语言总结。**

复盘会上，小陈总结：**「摘要记忆是 **压缩**。** 压缩率太高，记的是 **气氛不是事实**；压缩率太低，又贵又占圆桌。」**

林薇在 PRD 里加了一句红线：**「可入账的口径，记忆条里只存 **指针**（文档版本、检索 chunk_id），不存 **自由改写的一句概括**。」**

---

### 四、档案柜与「右键遗忘」

便签稳住后，张先生在 **手机端** 问了一半，回工位用 **桌面端** 继续——小言又问了一遍 **「您是哪条产线」**。

原来 **便签只挂在浏览器 session**，档案柜 **根本没接账号体系**；对他们公司来说，**「跨端连续」** 才是 **真·多轮**。

产品定 **档案柜最小集**：

| 类型 | 记什么 | 不记什么 | 默认保留 |
|------|--------|----------|----------|
| **偏好** | 常用产线、报表币种、是否看保守估 | 一次性的情绪吐槽 | 90 天，可用户改 |
| **工单上下文** | 未结案对账单的口径指针、确认单 hash | 已结案全文聊天 | 结案 +7 天删除 |
| **敏感** | 不自动写入 | 身份证、完整银行卡、人事处分 | 永不落库 |

法务强调 **「被遗忘权」**：设置里要有 **「清除我与小言的记忆」**——不是删聊天记录 UI，是 **删档案柜 + 便签 + 向量库里该用户的个性化索引**。

客服阿梅问：**「那 RAG（第 3 篇）算记忆吗？」**

林薇：**「RAG 是 **查公司库房**；Memory 是 **查这位用户和你之间发生过什么**。** 库房对所有人同一套规则；记忆必须 **按人隔离、可审计、可过期**。」**

---

### 五、接近结尾：原来差的是「记」的设计

灰度第四周，张先生再走一遍完整路径：手机端问到一半，桌面端接着聊；小言 **先复述已确认的 B 线口径指针**，再 **调检索补停线三条**；结构化确认单与昨天 **同源 hash 规则** 仍适用，**新算的差异** 不会覆盖 **已入账批次**。

李女士账号 **再也串不了**；她主动点了 **「清除记忆」** 后，小言 **礼貌地不再假装认识她**，但 **公司工单库里的操作手册** 照样能答。

指标会上：

- **「重复解释口径」工单 ↓ 68%**
- **单轮 Token 均值 ↓ 12%**（便签+指针，少搬全文历史）
- **记忆相关投诉**：从 **「失忆」** 变成 **「记错了」** 2 起——评测加了 **「摘要必须可追溯到引用」** 用例（第 10 篇评测体系）

林薇在复盘白板写：**「我们以前以为，多轮就是 **多发几条消息**；其实多轮是 **状态机**——**圆桌上是当下，便签是压缩结论，档案柜是跨天的用户态**。」**

张先生临走开玩笑说：**「你们终于不是 **每天换一个新实习生** 了。」**

小陈接话：**「实习生还是会忘——** **忘是我们允许他忘的**。** 该清的清，该指的指，别把 **整本聊天记录** 误当成 **产品级记忆**。」**

## 概念正解

**Memory / 会话记忆**，指在 AI 产品里 **有意图地保存、更新、检索并在后续交互中注入「与当前用户/任务相关的状态」** 的机制——常见形态包括 **当前会话的多轮消息、对话摘要、用户偏好与未结案工单上下文** 等 **持久或半持久存储**，而不是默认只靠 **单次请求里塞进的历史消息**。它解决的核心问题是：**大模型本身不会自动「记住」上次对话**；若产品需要 **连续体验、个性化、少重复劳动**，就必须在 **上下文窗口之外** 设计 **记什么、记多久、如何与 RAG/结构化/Grounding 分工**。

回扣故事：**张先生「还是按昨天口径」却遭反问** 对应 **无 session/无持久记忆时的「每次新开场」**；**全量历史塞进下一轮** 对应 **混淆「聊天记录」与「可用的记忆」+ 窗口与成本**；**便签摘要记歪、串用户** 对应 **摘要压缩丢事实、记忆隔离与合规**；**跨端断片** 对应 **记忆要绑账号与存储层，不只绑浏览器 tab**；**指针链到 Grounding 而非自由概括** 对应 **记忆条应存可验证引用，尤其对入账类状态**；**清除记忆按钮** 对应 **隐私与被遗忘权是产品能力，不是运维手工删库**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 划分 **会话内多轮 / 会话间摘要 / 用户级档案** 三层；写清 **字段表、TTL、结案删除、敏感字段黑名单、跨端同步、与 RAG 的边界** |
| **方案评审** | 问工程：**「记忆存哪（Redis/DB/向量）？是否参与每次调用的 Token 预算？」** 问算法：**「摘要谁生成、何时更新、失败回退？」** 问法务：**「清除记忆是否真删、审计留痕多久？」** |
| **指标设计** | 监控 **记忆命中率、因记忆注入增加的 Token、重复澄清率下降、记忆纠错工单、「记错」类 Grounding 失败率** |
| **跨团队沟通** | 对客服：**「小言记得的是 **授权范围内的偏好**，不是 **偷听别的渠道」**」**；对财务：**「已入账状态以 **系统 hash** 为准，记忆只帮 **复述指针**，不改写账本** |

## 类比锚定

**核心类比：CRM 里的 **客户跟进记录** + **销售个人便签** vs **每次打电话都从零自我介绍**——传统 PM 做 B 端或会员产品时熟悉：**工单系统里要有 **可继承的上下文**（上次处理结论、约定口径）；销售便签是 **压缩要点**；完整录音在 **存档** 里，不会每次通话 **从头到尾播放**。** Memory 就是给 AI 助手配的 **「客户档案 + 便签 + 当前通话记录」**，而且 **便签写错比没有便签更危险**。

| 你熟悉的 PM 经验 | Memory 要素 |
|------------------|-------------|
| 客服工单 **继承上一单结论**、关联用户 ID | **用户级持久记忆**，跨 session 绑定 `user_id` |
| 通话 **纪要**（三条 bullet）给下个值班同事 | **摘要记忆 / 对话压缩**，注入下轮 Prompt |
| 本次通话 **实时聊天记录** 仍在界面里 | **会话内多轮**，落在上下文窗口的「当前 thread」 |
| CRM **只存授权字段**，敏感信息不进备注 | **记忆白名单/黑名单**、合规与最小必要 |
| 「忘记我」营销 opt-out、账号注销删数据 | **清除记忆 / 被遗忘权**，删档案柜与个性化索引 |
| 知识库 **产品手册** 全员共用 | **RAG 公共知识** ≠ **该用户的私有记忆** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **模型无状态**：单次 API 调用 **不会自动带上** 上次对话；「记得」= **产品把状态再喂回去** 或 **从存储取出注入**。
2. **记忆要分层**：**圆桌（上下文）** 放当前回合必需材料；**长历史** 应 **摘要、指针化或外置存储**，避免与检索抢座位。
3. **记忆 ≠ RAG**：RAG 答 **「公司知道什么」**；Memory 答 **「这位用户和我们做到哪一步、偏好什么」**——隔离、权限、留存策略不同。
4. **压缩有代价**：摘要省 Token，但可能 **丢关键条件**；入账/口径类应 **链引用或可核验字段**，不只靠一句「用户说过」。
5. **遗忘是功能**：TTL、结案删除、用户一键清除 **与记忆增长同样重要**，否则 **隐私、串号、陈旧口径** 会反噬信任。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像 CRM，记得越全越好」 | **全量聊天灌回模型** 会 **爆窗口、涨成本、稀释检索** | 记忆是 **策展**，不是 **备份硬盘** |
| 「便签 = 销售随便写」 | 模型生成的摘要可能 **幻觉「用户说过」** | 摘要要 **可追溯、可纠错、关键事实用指针** |
| 「有记忆就不用用户重复」 | 记忆 **过期、串号、记错** 时 **比失忆更误导** | 高敏操作仍要 **确认单 / Grounding / 显式用户确认** |
| 「清除记忆 = 删 UI 聊天记录」 | 副本可能在 **向量库、日志、摘要表** | **删除范围** 要写进 PRD 与 **审计例外**（合规留存） |
| 「个人记忆可以替代微调」 | 微调（第 12 篇）改 **模型行为习惯**；记忆改 **实例状态** | 口头禅级能力走 **微调/LoRA**；订单上下文走 **Memory** |

### 下一篇预告
