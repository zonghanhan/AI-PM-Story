## 寓言故事

### 一、材料都在，它却说「没听过」

Embedding 灰度两周后，林薇以为「答非所问」会一路往下走。

结果客服主管又甩来一条 **VIP 客诉**。

用户是一位企业采购经理，和小言聊了 **十一轮**。对话里信息很碎，但有几句 **反复确认过**：

- 第 2 轮：「我们公司是 **对公转账**，不要走个人退款通道。」
- 第 5 轮：「这批货 **只要换货，不要退款**。」
- 第 8 轮：「发票抬头是 **某某科技有限公司**，别写简称。」

第 12 轮，用户问：「那换货面单寄到哪？**还是走对公流程对吧？**」

小言回复：「已为您申请 **原路退款至个人支付账户**，请留意到账短信。」

林薇调出当轮送进模型的 **完整输入包**——检索、工具、历史，全在日志里。她第一眼以为又是幻觉，逐段核对引用来源。

怪事来了：**RAG 捞回来的售后政策是对的**；`query_order` 返回的订单号也对；**但第 2、5、8 轮里那几句关键约束，在输入包里根本不存在**。

老周凑过来看日志：「Embedding 没骗你，段找对了。**问题是圆桌坐满了，最早那几轮被请出了房间。**」

---

### 二、不是记性差，是房间就这么大

林薇皱眉：「Token 那篇不是讲过吗？桌上纸摞太厚，早期的对话会被挤掉。」

「讲过 **计量**。」老周在白板上画了一个圆，中间摆了十二把小椅子，「今天讲 **边界**——不是『记不记得住』这种模糊说法，而是产品里必须写死的：**这一轮请求，模型能同时看见的总文本上限**。工程上叫 **Context Window（上下文窗口）**。」

他在圆桌上贴了不同颜色的便签：

| 便签颜色 | 内容 | 谁占座 |
|----------|------|--------|
| 红 | 系统提示词、安全规则、人设 | **每轮固定占座**，不走 |
| 蓝 | 用户与小言的历史对话 | 轮次越多，占座越多 |
| 绿 | RAG 检索回来的知识片段 | TopK 越大，占座越多 |
| 黄 | Tool 返回的订单、物流、库存 JSON | 往往 **一张便签顶三轮聊天** |
| 白 | 模型这一轮要生成的新回复 | **得先留空位**，不能坐满才想词 |

「十二把椅子，」老周敲了敲桌面，「不是比喻 **十二句话**，是 **十二摞 Token**——你们对外说『128K 上下文』，就是把椅子从十二把扩到一百二十八把。但 **椅子永远有限**；坐满了，新来的人只能 **挤掉旧人**。」

林薇问：「挤掉谁，谁说了算？」

「默认往往是 **先进先出**——最早进房间的先走。也有团队做 **摘要压缩**：把前六轮聊成一段话再塞回来，算 **换座不换信息**；还有 **只保留最近 N 轮** 的滑动窗口。」老周顿了顿，「**这些都是产品策略**，不是模型自己『想起来』。」

---

### 三、谁在抢椅子

为了搞清 VIP 客诉怎么发生的，林薇拉着研发做了一次 **「圆桌占座审计」**。

第 12 轮请求进来时，各模块申报的「座位需求」如下：

1. **系统提示词**（红）：约 2,400 Token——人设、合规话术、Tool 清单，**每轮必带**；
2. **RAG 检索**（绿）：TopK=8，每段平均 600 Token，合计约 4,800 Token——Embedding 改好后，研发把 K 从 5 提到 8，「提高命中率」；
3. **Tool 返回**（黄）：`query_order` + `query_logistics` + `query_inventory`，三份 JSON 合计约 6,200 Token——物流轨迹 **全量返回**，没有截断；
4. **历史对话**（蓝）：十一轮原文，约 5,100 Token；
5. **预留给输出**（白）：按产品配置留 1,024 Token。

加总 **约 19,500 Token**。而他们当时用的模型，**有效上下文窗口是 16K**——还没算格式符、工具 schema 的开销。

「所以不是模型『忘了』。」研发同学指着截断日志，「**组装 Prompt 时做了尾部截断**：系统提示词和本轮用户问题不能动，RAG 和 Tool 又不敢随便砍——**先砍的是最老的历史消息**。第 2、5、8 轮，正好在砍刀下面。」

林薇后背有点凉。她想起上周评审会上，自己还签字同意了「**提高 TopK**」和「**物流详情全量展示给用户看更专业**」——两个改动 **单独看都合理**，叠在一起却把 **用户亲口说的约束** 挤出了房间。

---

### 四、换大房间，不是免费午餐

林薇的第一反应很 PM：「换 **更长上下文** 的模型。」

老周没反对，但列了一张 **产品账**：

**账 1：钱**  
上下文越长，**输入 Token 单价往往越高**；每轮都把 RAG+Tool+全长历史塞满，账单按 **实际占座** 算，不是按「窗口上限」算。

**账 2：慢**  
同样内容，128K 窗口的模型 **首字延迟** 常比 16K 方案高——用户多等半秒，客服场景就是 **多挂一通电话**。

**账 3：中间迷失**  
算法同学补刀：有些模型 **窗口虽大**，但 **越靠中间的材料越容易被忽略**——叫 **「lost in the middle」**。你把 40 段政策全塞进去，模型反而 **盯头盯尾、漏中间那条「拆封不退」**。

**账 4：窗口大 ≠ 该全塞**  
老周：「大房间解决的是 **偶尔需要长材料** 的场景，不是鼓励你把 **垃圾也搬进来**。Context Window 是 **天花板**，不是 **推荐用量**。」

林薇在板上写了四行 **「占座优先级」**，准备写进 PRD：

1. **不可删**：系统规则、本轮用户问题、安全拦截相关；
2. **可压缩**：早期历史 → 滚动摘要；Tool 大 JSON → 只保留决策字段；
3. **可裁剪**：RAG TopK 动态下调；物流只保留最近 3 节点；
4. **可外置**：超长合同 PDF 不全文进窗，改为 **先检索再摘录** 或 **分轮追问**。

---

### 五、圆桌上的主持人

又过了一周，林薇主导上线 **「上下文预算管理员」**——同事们给这套编排逻辑起的外号。

它不做新模型，而是在 **每轮请求进模型之前** 做三件事：

1. **先算总 Token 预算**：窗口上限 − 输出预留 − 系统提示词 = **本轮可用余额**；
2. **按优先级装车**：先塞本轮 query 和关键摘要（含「对公 / 只换货 / 发票抬头」这类 **实体槽位**），再塞 RAG，最后塞 Tool 精简结果；
3. **历史分层**：最近 3 轮原文保留；更早的压成 **200 Token 摘要**，并 **单独高亮用户约束句**，避免被摘要员「润色掉」。

灰度后，同类 VIP 会话做了回放测试：**第 12 轮输入包里，第 2、5、8 轮的关键约束以摘要标签形式仍在**。

复盘会上，林薇说了一句后来写进团队 Wiki 的话：

> **Context Window 不是技术参数表里的一个数字，而是 AI 产品里的「会议室面积」。**  
> 谁进房、谁站门口、谁被请出去——**必须有人主持**，不能指望模型自己从一百页材料里猜哪句是用户真心话。

散会后，她看着监控大屏：平均输入 Token 从 14,200 降到 9,600，**回答一致性** 反而上升。老周笑：「椅子还是十二把变十六把那档，但 **不再让物流 JSON 坐 CEO 的位置**。」

林薇忽然明白：Token 教她 **怎么数纸**，Context Window 教她 **怎么排座**——**数得清，才坐得下；坐得下，才看得见。**

## 概念正解

**Context Window（上下文窗口）** 是大语言模型在 **单次请求** 中能同时接收和处理的最大文本量上限（通常以 Token 计），包括系统提示词、历史对话、检索片段、工具返回和待生成回复的预留空间。超出上限时，必须通过 **截断、摘要、滑动窗口、外置检索** 等手段取舍；窗口越大不代表应塞满，还涉及 **成本、延迟与「中间信息被忽略」** 等产品权衡。

回扣故事：

- **「圆桌十二把椅子」** → 模型的 **固定上下文容量上限**；
- **VIP 客诉「材料都在日志里，输入包却没有」** → **组装 Prompt 时的截断**，早期用户约束被挤出窗口；
- **RAG TopK 与 Tool 全量 JSON 抢座** → 多模块 **共享同一窗口预算**，需 **优先级编排**；
- **换大模型不是万能** → 长窗口有 **成本、延迟、lost in the middle** 等边界；
- **「上下文预算管理员」** → 产品层的 **Context 管理策略**（摘要、分层、动态 TopK）。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写明 **所用模型窗口上限、输出预留 Token、各模块预算占比**（系统 / 历史 / RAG / Tool）；定义 **历史超限时摘要规则** 与 **不可丢弃字段**（订单号、用户明确约束） |
| **方案评审** | 问研发：「截断发生在 **哪一层**？先砍历史还是先砍检索？多轮 Agent **工具回灌** 会不会一轮就顶满窗口？」 |
| **指标设计** | 监控 **平均/ P95 输入 Token、截断率、因截断导致的约束丢失抽检**；A/B 比较「提高 TopK」与 **回答一致性** 是否反向 |
| **跨团队沟通** | 跟业务说：「不是模型健忘，是 **会议室满员**」；跟算法说：「换 128K 模型要一起评估 **单价、时延、是否真的读完中间段**」 |

## 类比锚定

**核心类比：线下评审会只有一间会议室、一张圆桌——议程、参会人、参考资料、现场打印的材料，都要抢有限座位；会开太长，最早达成的共识可能被请出房间**

传统 PM 都经历过 **会议溢出**：2 小时议程硬塞 4 小时内容，结果 **最早定的范围** 在下半场没人提；或 **打印了 200 页合同**，与会者其实只翻 **目录里标黄的三页**。Context Window 就是这场会的 **场地面积**；你怎么 **排议程、控材料、写纪要**，决定 AI 这一轮 **到底「看见」什么。

| 你熟悉的 PM 经验 | Context Window 要素 |
|------------------|---------------------|
| 会议室最多坐 12 人，超员只能改大房或分批开 | 模型 **窗口上限固定**；扩容 = **换模型 / 多轮拆分** |
| 系统规则、合规同事、老板 **必须到场** | **系统提示词、安全策略** 每轮 **优先占座** |
| 会议越开越长，最早定的「不做退款只做换货」没人记得 | **长对话早期约束** 易被 **截断或摘要丢失** |
| 现场发 200 页 PDF，大家其实只看摘要页 | **RAG / Tool 全量灌入** 浪费窗口；应 **摘录关键字段** |
| 换更大的会议室更贵、预约更难 | **长上下文模型** → **更高 Token 成本、更长延迟** |
| 会后纪要要把「用户原话约束」单独标红 | **摘要策略** 需保留 **实体与否定句**（「不要退款」不能润色没） |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Context Window 是单次请求的硬上限**，不是「模型长期记忆」；多轮对话要靠 **每轮重新组装** 进窗口的内容来维系上下文。
2. **系统提示、历史、RAG、Tool 返回、输出预留共享同一预算**；一模块膨胀会 **挤压其他模块**，引发 **隐性截断**。
3. **截断与摘要策略是产品决策**：先进先出、滑动窗口、结构化摘要、槽位保留，直接影响 **多轮一致性** 与 **合规**。
4. **更大窗口缓解但不消灭问题**：成本、延迟、**中间段落被忽略（lost in the middle）** 仍存在；**不应默认塞满**。
5. **与 Token 概念配套**：Token 是 **计量单位**，Context Window 是 **容器容量**；评审时要 **同时看占用率与截断日志**。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「会议室大了就能把所有材料都摊开」 | 模型对 **超长输入中间段** 的利用率可能 **低于首尾** | 关键条款应 **靠前放置、重复强调、或结构化字段**；不能假设「进了窗就等于读了」 |
| 「被请出去的人就真的走了」 | 截断后内容可通过 **外部存储、数据库、会话状态机** 在下一轮 **再注入** | Agent 产品常用 **外置记忆 + 按需检索**；窗口管理是 **编排问题**，不是单次聊天室 |
| 「换大房一劳永逸」 | 128K/1M 窗口 **按实际 Token 计费**，全量灌入 **成本线性涨** | 产品应设 **预算上限与降级策略**（减 TopK、缩 Tool、触发分轮） |
| 「纪要让秘书写就行」 | **自动摘要会丢否定词、条件句、数字** | 「只换货」「不对公」类约束要 **规则保留或槽位抽取**，不能全靠摘要模型「润色」 |
| 「一场会只开一次」 | 同一用户会话在多轮中 **窗口内容每轮重新打包** | 每轮都要 **重新审计占座**；不是「第一轮进场的永远占着」 |

### 动手小练习（约 15 分钟）

1. 选你们正在用的一个模型，查 **context window 官方数字**；用 Token 估算器粗算：系统提示 + 8 段 RAG + 1 份订单 JSON + 10 轮对话，是否 **超限**。
2. 画一张 **「单轮请求占座饼图」**：标出五类内容占比，写出 **超限时你先砍哪一类** 及业务理由。
3. 找一条真实 **多轮客诉会话**，问研发要 **第 N 轮实际送进模型的输入**（可脱敏），检查 **用户早期约束是否仍在**——把发现写成一条 PRD 需求：「不可截断字段清单」。

### Context Window 产品 Checklist（可保存到笔记）

```markdown
## 规格与预算
- [ ] 模型窗口上限（是否含输出预留）
- [ ] 单轮输入 Token P95 与截断率
- [ ] 各模块预算上限：系统 / 历史 / RAG / Tool / 输出

## 截断与摘要
- [ ] 超限时的裁剪顺序（谁先谁后）
- [ ] 历史摘要策略；否定句、金额、主体名称是否规则保留
- [ ] 关键用户约束是否写入外置槽位（订单态、对公/个人）

## RAG 与 Tool
- [ ] TopK 是否随剩余预算动态调整
- [ ] Tool 返回是否字段白名单；大 JSON 是否服务端截断
- [ ] 检索片段在 Prompt 中的排序（关键段靠前）

## 评审必问
- [ ] 换更长窗口模型的成本与时延测算
- [ ] 是否存在 lost in the middle 风险场景
- [ ] 截断导致的业务事故如何监控与回放
```
