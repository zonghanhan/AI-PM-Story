## 寓言故事

### 一、像表格的一段话

流式灰度满月后，林薇以为「等太久」的老问题总算压下去了。财务苏姐却带着一张 **红头内部函** 走进会议室：**「产线对账助手可以帮业务查数，但 **入账不能靠人抄**。」**

需求写得很直白：用户在小言里问 **「上月 A 线与 B 线 MES 与 PLC 差异汇总」**，小言要从工单系统、对账日志里捞出结果，**生成一张表，自动调用财务中台的 `importReconciliation` 接口**——列名固定为 `line_id`、`mes_qty`、`plc_qty`、`diff_qty`、`currency`、`remark`，类型和枚举都要对齐。

运营小夏当场演示了一条 **看起来很成功的** 对话录屏：小言 **流式** 滚出一张漂亮的 Markdown 表，用户点赞，还 **一键复制** 到飞书文档。

苏姐把录屏暂停在表格中间：**「飞书能认，我们的 ERP **不认**。你们给的是 **给人看的排版**，我要的是 **给程序吃的字段**。」**

大刘调出实际返回体：外层是聊天 JSON，正文里塞着 **未闭合的 Markdown 管道符**——因为用户 **提前点了停止**，后半列 **`currency` 还没生成完**。财务脚本用正则扒表格， **把「CNY」扒成了表头的一部分**，接口返回 **400：currency 枚举非法**。

林薇第一次在周报里写：**「体验做对了，集成做错了。」**

---

### 二、实习生手写 Excel，财务要 XSD

站会上，工程提了三条路：

```
A. 模型照旧写 Markdown → 后端再让另一个模型「翻译成 JSON」（两次调用、两次运费）
B. 提示词里写「请只输出 JSON」→ 祈祷格式别坏（祈祷不是架构）
C. 走厂商的 Structured Output / JSON Schema 约束 → 生成过程就按格子填
```

算法小陈在白板上画了个 **方格本**：**「人说话可以含糊，** **`importReconciliation` 的契约没有『大概』**。要么字段齐、类型对，要么整单失败——没有『用户自己猜缺了哪列』。」**

售后阿梅弱弱举手：**「业务同事其实就想要 **能看的表**……」**

林薇摇头：**「这是 **双出口** 问题：同一份对账结果，**眼睛一路**（Markdown 表给用户确认），**系统一路**（JSON 给财务入账）。两路可以同源，但不能 **只修一条路**。」**

她拍板：**「四楼对账类工单，新增 **结构化输出通道**；聊天里仍可流式展示 **人类可读摘要**，但 **入账 JSON 必须走 Schema 约束 + 服务端校验**，校验不过 **不许静默入库**。」**

---

### 三、格子满了，备注写不进

灰度第一周，结构化 JSON **终于能过接口**——财务同学还没来得及笑，工单又来了。

用户问：**「差异原因写详细点，把三条产线停线记录都塞进备注。」** 小言按 Schema 吐 JSON，`remark` 字段 **被截断在 256 字符**——不是模型偷懒，是 **Schema 里 `maxLength` 就是 256**（半年前法务定的防注入长度）。

另一条：业务希望 **`diff_qty` 为空时整行仍上报**（表示「待复核」），模型却 **自己填了 0**——在人眼里「零差异」，在财务眼里 **伪造平账**。

复盘时小陈调出日志：**「结构化输出减少了 **『括号没闭合』**，但没减少 **『填错语义』**。Schema 管 **形状**，不管 **真理**。」**

林薇让产品在 PRD 里加两栏：

| 问题 | 产品规则 |
|------|----------|
| 字段被截断 | **超长走附件 ID**，`remark` 只放摘要 + `attachment_ref` |
| 可空 vs 默认值 | **`diff_qty` 允许 `null`**，禁止用 0 冒充「未知」；下游展示 **「待复核」** |

老周路过看了一眼 Schema 草案：**「你们这是在给实习生 **发填表模板**，不是让他自由写作文。」**

---

### 四、一次不过，要不要自动重试

更棘手的是 **偶发失败**。

MES 接口抖动，检索片段里 **缺了 B 线产量**；模型在 Schema 约束下 **仍吐出完整 JSON**——`mes_qty` 填了一个 **看起来合理的整数**（第 7 篇幻觉的老朋友，只是这次 **JSON 特别像真的**）。

财务批处理 **静默入账 37 笔** 后，月结对不上，升级 **P0**。

门卫老关的输出复检 **抓不到**：敏感词没有，格式也合法。**Grounding（第 11 篇）** 显示引用片段里 **根本没有 B 线数字**——模型 **用结构合规掩盖了事实缺失**。

工程提议：**「校验失败或关键字段无引用，自动 **重试一次**，换更保守的 Prompt。」**

苏姐反对：**「重试可以，但 **不能悄悄改数再提交**。第二次仍缺字段，就 **整单失败 + 人工工单**，并 **留审计日志**。」**

林薇定规：**结构化输出的失败策略三张牌**：

1. **格式失败**（缺字段、类型错）→ 服务端拒绝，可 **自动重试 1 次**（仅修形）  
2. **语义失败**（Grounding 对不上、关键列为空）→ **不自动重试填数**，转人工  
3. **部分成功**（多行表里一行不过）→ **整批不入账**，避免 **半真半假**

小陈总结：**「结构化让 **程序能读**；Grounding 和评测让 **程序敢读**。」**

---

### 五、双出口与「到底以谁为准」

灰度第三周，体验组做出 **「左右分屏」**：左边流式 Markdown **给用户点头**，右边 **折叠的 JSON 预览** 给实施同学 debug。

用户张先生吐槽：**「我又不是开发，为什么要看 JSON？」**

同时，业务李姐 **只信左边表格**， **没展开右边**，批准入账——而左边表格为了好看 **四舍五入到整数**，右边 JSON **保留两位小数**。财务系统按 JSON 入账， **和业务签字表差 0.02 元 × 800 行**。

林薇在演示厅关掉分屏，改成：

- **用户只看见「对账确认单」**（固定模板，字段与 JSON **同源渲染**）  
- **用户点「确认提交财务」** → 提交的是 **已校验 JSON 的 hash**，不是 **重新让模型现编**  
- **流式只用于解释过程**（「我为什么这么汇总」），**不用于最后一公里的数字**

大刘补充技术约束：**「Structured Output 最好走 **非流式或缓冲后一次性交付** 入账包；流式可以继续讲 **故事线**，别把 **半张 JSON** 推给财务。」**

两周后指标：

- **财务接口 400 错误率 ↓ 91%**（格式类）  
- **人工复核工单 ↑**（语义类被正确拦下）  
- **「抄错表」投诉归零**  
- **「模型填了看起来合理的假数」** 仍有 2 起——评测集加了 **「缺检索片段必须 null」** 用例

林薇在复盘白板写：**「结构化输出不是『让回答变丑』，是 **在需要对接系统的场景，签一份机器可执行的合同**。」**

苏姐收起红头函：**「人可以读散文，财务只收 **盖章的表格**——章就是 Schema + 校验 + 确认按钮。」**

## 概念正解

**Structured Output / 结构化输出**，指在调用大模型时 **用预定义的格式契约（常见为 JSON Schema）约束生成结果**，使输出 **字段、类型、层级固定**，便于程序 **直接解析、校验、入库**，而不依赖「从自然语言里抠表格」。它主要解决 **人机双受众** 问题：人需要可读解释，系统需要 **可预测的数据形状**。

回扣故事：**Markdown 表进不了 ERP** 对应 **自然语言/排版 ≠ 机器可读**；**方格本 / 填表模板** 对应 **Schema 约束生成**；**`remark` 截断与 0 冒充空** 对应 **Schema 只管形状不管业务语义**；**格式合法但数字幻觉** 对应 **结构化不替代 Grounding 与评测**；**左右分屏不一致** 对应 **双出口必须同源**；**确认按钮提交 hash** 对应 **冻结已校验结构，避免二次生成漂移**。

在 **AI 产品岗** 的典型场景：

| 场景 | 具体例子 |
|------|----------|
| **需求文档** | 写清 **哪些意图走结构化通道**、**Schema 字段表（名/类型/可空/枚举/长度）**、**失败策略（重试几次、是否转人工）**、**与用户可见 UI 的数据同源规则** |
| **方案评审** | 问工程：**「用 JSON mode 还是 response_format + schema？校验在前端还是网关？流式与结构化是否互斥？」** 问算法：**「缺证据时模型能否输出 null？」** |
| **指标设计** | 监控 **Schema 校验通过率、自动重试率、语义拦截率、入账后回滚率**；区分 **格式错误 vs 业务错误** |
| **跨团队沟通** | 对财务/ERP：**「我们保证的是 **字段形状**，金额正确性靠 **检索来源 + 人工确认**」**；对运营：**「别让用户手抄 AI 表进系统」** |

## 类比锚定

**核心类比：活动报名 **在线表单** vs **微信群接龙**——传统 PM 办活动、收调研、对接运营系统时都经历过：**接龙读起来亲切，但导入 CRM 要人工对齐列名**；**表单字段固定、必填校验、枚举下拉** 才能 **一键同步后台**。结构化输出是给模型的 **「在线表单 + 后端校验」**，不是让它继续在群里 **发一段看起来像表格的文字**。

| 你熟悉的 PM 经验 | Structured Output 要素 |
|------------------|-------------------------|
| 活动报名 **固定字段**（姓名/手机/部门）+ **必填星号** | **JSON Schema 字段定义** + **required** |
| 下拉框 **枚举**（职级只能选 P/M） | **`enum` / 类型约束**（如 `currency` 仅 CNY/USD） |
| 前端 **提交前校验**，不通过不让点「提交」 | **服务端 Schema 校验** + 失败返回明确错误码 |
| 用户看到的是 **确认页**，后台收的是 **同一份结构化 payload** | **双出口同源**：UI 渲染与 API 用同一 JSON |
| 接龙写「大概三十人」→ 导入失败 | **自然语言表格** → 程序解析不可靠，需 **生成即结构化** |
| 超长备注走 **附件上传**，正文框限字数 | **大段内容用 `attachment_ref`**，避免 **maxLength 截断语义** |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **结构化解决的是「形状」**：让下游 **能解析**；不自动保证 **数对、事对**——语义正确仍要 **检索、Grounding、人工确认、评测**。
2. **需要对接系统、批量入账、驱动 Tool 参数时**，应优先 **结构化通道**；纯 FAQ 解释可继续 **自然语言 + 流式**。
3. **Schema 是产品契约**：字段可空、默认值、枚举、长度 **都是业务决策**，不是工程随手写。
4. **失败要分类型处理**：格式错可 **有限重试**；证据不足应 **失败可见、转人工**，禁止 **自动编合理假数**。
5. **人机双受众要设计「确认点」**：用户签字的是 **与 JSON 一致的确认单**，避免 **展示层与入库层两套数**。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像表单，填对就能入账」 | 表单 **人不填就不会提交**；模型可能 **编造完整字段** | 必须 **校验 + Grounding + 关键操作二次确认** |
| 「Schema 定死就一劳永逸」 | 业务改列名、多币种、新产线 **要版本化 Schema** | 像 **API 版本管理**：`importReconciliation_v2`，兼容与迁移写进 PRD |
| 「JSON 比 Markdown 便宜」 | 约束解码、重试、 **双模型翻译** 可能 **更贵** | 算 **集成失败的人工成本** vs **多一次调用的 token** |
| 「结构化就不用流式了」 | 解释性内容仍可 **流式叙事**；**入账包** 宜 **一次性交付** | **分通道**：故事流式，契约非流式或缓冲闭合 |
| 「校验通过 = 可以静默执行」 | 高敏操作（入账、改价、删数据）**即使格式对也要权限与审计** | 结构化是 **必要非充分**；结合 **Tool Calling 权限、护栏、人工审批** |

```markdown
## 上线与运营
- [ ] Schema 校验通过率、重试率、人工兜底量

```
