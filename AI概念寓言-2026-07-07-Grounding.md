## 寓言故事

### 一、答对了，却发不出去

Rerank 上线两周，客服小队的抽检正确率从 58% 爬到 71%。林薇以为能松口气，直到法务陈姐抱着一叠打印件走进会议室。

「这条你们打算全量推？」陈姐点开一条典型回复。用户问企业版能否接钉钉单点登录；小言答得干脆：「支持。请管理员登录 **管理后台 → 集成中心 → 钉钉 SSO**，按向导配置即可。」

「内容 **没错**。」陈姐敲了敲屏幕，「但对外自动回复，我们制度要求：**每一个可执行的事实陈述，必须能追溯到官方文档的具体位置**。用户按错一步，投诉上来，我们要证明 **不是客服瞎编，是文档那么写的**。」

林薇调出后台日志。检索段确实进了模型，小言也答对了——但用户界面上 **只有一段流利中文**，没有链接、没有段落编号、没有「依据来源」。

运营同学委屈：「用户又看不懂 chunk_id……」

陈姐摇头：「看不懂编号，可以看懂 **《企业版集成指南》第 3 章第 2 节** 和一个 **可点击的锚点**。你们要的是 **口才好的实习生**，我们要的是 **带出处的工作底稿**。」

散会后，老周在白板上写了四个字：**言之有据**。

「Rerank 解决 **捞哪几段**；接下来要解决 **答的每一句，能不能指回那几段**。」

---

### 二、给每句话发一张「来源贴纸」

算法小陈拿来一版 Demo，起名 **「脚注模式」**。

流程变了：

```
用户问题
   ↓
检索 + Rerank（照旧）
   ↓
【新增】把进模型的片段标上 [1][2][3]，并带上标题、URL、段落锚点
   ↓
小言生成时：要求「关键句末标注引用编号」
   ↓
【新增】前端把 [1] 渲染成可点击脚注，点开跳到原文
```

林薇试了一条。小言回复：

> 企业版支持钉钉 SSO 单点登录。[1]  
> 配置入口在管理后台的「集成中心」。[1]  
> 个人版不支持此能力，需升级席位。[2]

底部挂着两张「来源贴纸」：

| 编号 | 文档 | 锚点 |
|------|------|------|
| [1] | 《企业版集成指南》v2.3 | §3.2 钉钉 SSO |
| [2] | 《版本功能对照表》 | 个人版 vs 企业版 |

她点 [1]，浏览器滚到文档里 **高亮的那一段**——和小言说的逐句能对上。

「这不是排版好看，」小陈说，「是 **生成时被约束在引用列表里说话**。模型拿到的指令大概是：**只能依据方括号里的材料下结论，并把句子钉到编号上**——行业里叫 **Grounding（接地）**，产品侧常叫 **引用约束** 或 **可追溯生成**。」

老周补了一句：「像实习生交报告：**正文里每个数字、每个步骤，后面必须跟脚注**。脚注不是装饰，是 **责任边界**。」

---

### 三、出处错了，比没出处更糟

灰度第三天，事故来了。

用户问：「**数据导出** 最多能导多少天的聊天记录？」

小言答：「企业版支持导出近 **180 天** 的全部记录。[1]」

用户点 [1]——《企业版数据导出说明》里写的是：**90 天**；180 天是 **即将上线** 的灰度能力，写在同页 **公告栏**，并未正式开放。

客服工单炸锅：「你们机器人 **有据可查地误导我**！」

复盘时，林薇发现三重问题：

**第一，钉子钉歪了。** 模型把「180 天」这句挂在 [1] 上，但 [1] 正文写的是 90 天——**引用编号在，对齐关系错了**。用户会比「没出处」更愤怒，因为 **看起来可核实，核实却是错的**。

**第二，一段里有两套时间。** Rerank 捞回来的片段 **同时含现行规则与预告**，小言 **拼了一句顺滑的话**，却把 **未生效能力** 当成 **已生效能力** 说出。

**第三，没有「钉不住就别说」。** 系统为了「总有脚注」，在不确定时仍硬生成 [1]，而不是 **拒答或转人工**。

小陈调出原始 Prompt，脸有点红：「我们强调了 **要标引用**，没强调 **标了就必须逐句对齐**……」

老周在事故单上写：**假出处，比幻觉更像真的。**

---

### 四、产品定规矩：什么必须引、怎么验

林薇没有关掉脚注模式，而是补了一份 **《对外答复引用规范》**，写进 PRD：

**规则 1：分档引用**  
- **操作类、承诺类、合规类**（能不能、多少钱、几步做完）：**逐句强制引用**，缺引用不出厂；  
- **寒暄、总结、过渡句**：可不标号，但 **不得含新事实**。

**规则 2：引用对齐校验**  
生成后增加一道 **轻量校验**（规则 + 小模型均可）：抽取「带编号的句子」与对应片段，做 **关键词/数字一致性** 比对；对不齐 **打回重写或降级为「请查看原文 [1]」**。

**规则 3：片段时效标签**  
知识库切片增加 **生效状态**（已上线 / 预告 / 已废弃）。进模型前过滤 **未生效** 段落，或强制小言在句首加 **「截至文档 v2.3」**。

**规则 4：钉不住就别说**  
若检索置信度低、或校验失败超过阈值：**不硬答**——展示原文链接 +「建议联系人工」，避免 **漂亮但错的脚注**。

**规则 5：可观测**  
日志记录 **每句 ↔ 引用片段 ↔ 校验结果**，方便法务抽检与客服复盘。

又灰度两周。同一条「导出多少天」的问法，小言改为：

> 根据当前正式文档，企业版可导出近 **90 天** 聊天记录。[1]  
> 页内公告提到 180 天能力正在灰度，**以您租户后台实际开关为准**。[1]

脚注仍只有 [1]，但 **数字与原文一致**；预告被 **明确降权** 表述。

陈姐在上线单上签了字：「这才像 **能进审计的工作底稿**。」

散会上，林薇对新人说：「检索解决 **有没有材料**；Rerank 解决 **哪段该上桌**；引用约束解决 **端上来的菜，是不是桌上那盘**。」

老周合上笔记本：「用户信的不是小言嘴甜，是 **点得开的出处**。」

## 概念正解

**Grounding（接地 / 引用约束）** 指：在 RAG 等流程中，**要求模型生成内容必须基于已提供的检索片段或权威资料**，并通过 **脚注、链接、段落锚点、chunk 编号** 等形式，让用户能 **逐句追溯依据**；它解决的是「答对了但 **无法证明**」「看起来专业却 **查无此据**」的信任与合规问题，核心不是多几个链接，而是 **生成与证据的对齐关系**。

回扣故事：

- **「答对了却发不出去」** → 仅有正确生成不够，对外场景需要 **可追溯性**；
- **「来源贴纸 [1][2]」** → 给检索片段编号，约束模型 **只在引用范围内陈述**；
- **「180 天挂在写 90 天的 [1] 上」** → **假引用 / 引用不对齐** 比无引用更危险；
- **「预告与正式规则混在一段」** → Grounding 依赖 **上游切片质量与时效元数据**；
- **「钉不住就别说」** → 产品需 **分档引用、对齐校验、低置信兜底**。

在 **AI 产品岗** 的典型场景：

| 场景 | 你要怎么用这个概念 |
|------|-------------------|
| **需求文档** | 写明 **哪些意图必须带引用**、引用展示形态（脚注/侧栏/高亮原文）、**对齐校验规则**、失败时 **拒答/只展示原文** 的策略 |
| **方案评审** | 问研发：「引用编号是 **模型自填** 还是 **后处理映射**？如何防 **编造 chunk**？校验失败 **是否阻断发送**？」 |
| **指标设计** | 除正确率外，增加 **引用覆盖率**（该引是否引了）、**引用准确率**（标了是否真对）、**用户点击率/核实率**、**因引用错误引发的投诉占比** |
| **跨团队沟通** | 跟法务说：「不是上 RAG 就合规，要 **可点击的证据链**」；跟算法说：「别把引用当 UI 皮肤，是 **生成约束 + 校验闭环**」 |

## 类比锚定

**核心类比：对外帮助中心 / 官网 FAQ 的「操作说明编写规范」——每个可执行步骤必须挂官方文档链接，且链接要落到具体段落，编辑发版前逐条点验**

传统 PM 多半协同过 **帮助中心、官网 FAQ、客服话术库**：编辑写「三步完成绑定」时，运营会要求 **每步附文档锚点**；发版前有人 **点开链接核对是否与正文一致**。Grounding 就是把这套 **「写一句、挂一处、点一遍」** 搬进智能客服与 Copilot。

| 你熟悉的 PM 经验 | Grounding 要素 |
|------------------|----------------|
| FAQ 里「点击设置 → 安全」必须链到产品手册对应截图位置 | 生成句末 **脚注 [n]** 对应 **检索片段 / 锚点** |
| 编辑规范：「价格、时效、权限」类句子 **禁止无来源** | **操作/承诺/合规类** 强制 **引用约束** |
| 发版前点击链接，检查是否 404、是否改版对不上 | **引用对齐校验**（数字、版本、能力名一致） |
| 文档页脚写「本文适用于 v3.1，v3.2 变更见……」 | 片段 **时效/生效状态** 元数据，防把预告当现行 |
| 找不到依据的，改成「请联系客服」而非瞎写 | **低置信拒答 / 只展示原文**，避免硬编带脚注 |
| 用户投诉时，用「当时链接截图」定责 | 日志留存 **句 ↔ 引用 ↔ 校验结果**，可审计 |

## 类比边界说明

### 一、概念核心要点（回扣定义）

1. **Grounding 约束的是「陈述 ↔ 证据」的对齐**，不是单纯在 UI 上多几个链接；无对齐校验的脚注，可能 **提高虚假可信度**。
2. **强引用适用于高风险、高承诺场景**（合规、医疗、金融、SLA）；闲聊、风格润色不宜 **句句标号**，但要防止 **无引用句里夹带新事实**。
3. **依赖上游检索与切片质量**：片段混杂、过期、预告与正式并存时，Grounding **无法自动纠错**，需 **元数据、过滤、版本策略**。
4. **产品要定义分档规则与失败策略**：什么必须引、引不了怎么办、校验失败是否 **阻断发送**——这是 **合规与体验的权衡**，不是纯算法问题。
5. **与幻觉治理互补**：RAG 减「凭空编」；Grounding 减「 **编了还能圆** 」与「 **对但无法举证** 」。

### 二、这个类比在哪里会失效？

| 类比说法 | 失效点 | 更准确的理解 |
|----------|--------|--------------|
| 「像编辑挂链接一样，挂上就合规」 | 模型可能 **引用号正确、内容却偷换数字或合并两句** | 必须 **生成约束 + 后验对齐校验 + 抽检**；链接是结果，不是万能符 |
| 「每句话都像 FAQ 逐步操作」 | 总结、对比、多段综合推理 **难以一句一引**，过度脚注 **可读性崩溃** | 按 **意图与风险分档**；综合结论可引 **多源列表** 或 **「详见原文」降级** |
| 「人工发版前点一遍链接就行」 | 生成是 **毫秒级、百万次** 的，无法靠人逐条点 | 靠 **自动化校验、抽样审计、异常告警**；人设从编辑变成 **质检流水线** |
| 「有出处就等于事实正确」 | 文档本身可能 **过时、矛盾**；引用只证明 **「模型说的是文档写的」** | 还要做 **知识库治理、版本切换、废弃标记**；Grounding 是 **忠实于资料**，不是 **忠实于真理** |
| 「客服话术库一条一条对应」 | 用户问法千变万化，生成是 **重组表述**，不是复制粘贴话术条目 | 允许 **改写**，但 **事实性成分** 必须可映射回片段；映射失败应 **拒答** |

### 动手小练习（约 15 分钟）

1. 从你们 Bot **抽 10 条已上线回复**，标出含 **价格、权限、操作步骤** 的句子；看 **有几句能点开到具体文档段落**——算一个粗糙的 **「应引覆盖率」**。
2. 让研发导出一条 **带 [1][2] 的日志**，你 **人工核对** 编号与原文是否一致；发现 1 处不对齐，写一句应走的 **兜底策略**（重写 / 只展示原文 / 转人工）。
3. 在 PRD 里写 **三档意图**：「必须逐句引用」「可选引用」「禁止编造事实」各举 1 个业务例子（如退费规则 vs 寒暄 vs 创意文案）。

### Grounding 产品 Checklist（可保存到笔记）

```markdown
## 引用策略
- [ ] 高风险意图清单：操作/价格/权限/合规 → 强制引用
- [ ] 引用展示：脚注 / 侧栏原文 / 段落高亮
- [ ] 无引用或低置信：拒答、只链原文、转人工

## 对齐与校验
- [ ] 句 ↔ 片段编号映射方式（模型标注 vs 后处理）
- [ ] 数字、版本、能力名一致性校验
- [ ] 防编造 chunk / 伪造 URL

## 知识库协同
- [ ] 切片含版本、生效状态、废弃标记
- [ ] 预告/灰度内容与正式规则分离或显式降权

## 指标与审计
- [ ] 引用覆盖率、引用准确率、用户点击核实率
- [ ] 日志：生成句、引用片段 ID、校验结果
- [ ] 因「引用错误」导致的投诉/工单单独统计
```
